変数とデータ型（基礎中の基礎）

▼

💡 データを格納する「箱」と、その種類について

なぜ変数が必要なのか？

プログラムはデータの塊です。計算結果やユーザーの入力を一時的に保存しておく「箱」が変数です。変数を使わなければ、入力データを次の処理に渡すことができません。Pythonの変数は、他言語（CやJava）のように「この箱は整数専用」といった宣言が不要で、非常に柔軟です。

PYTHON 
# ❌ 悪い例（マジックナンバー：数値が直接書かれていて意味が不明）
print(1000 * 1.1)

# ✅ 良い例（変数を使って意味を持たせる）
price = 1000
tax_rate = 1.1
total_price = price * tax_rate
print(total_price)

データ型（文字列、整数、浮動小数点、真偽値）

箱に入れるデータには種類（型）があります。文字列同士は繋げられますが、文字列と数値を直接足すとエラーになります（ここがAIコードでよく起きるバグの元です）。

PYTHON 
name = 'Alice'     # 文字列 (str)
age = 25           # 整数 (int)
height = 165.5     # 浮動小数点 (float)
is_student = True  # 真偽値 (bool)

# ❌ 型が違うとエラーになる
# print(name + str(age)) 

# ✅ 文字列に変換してから繋げるか、f-stringを使う
print(f'{name}は{age}歳です')  # 実務ではf-string（f文字列）が圧倒的に使われます

💡 プロの実務Tips

AIに「変数を定義して」と指示するよりも、「金額を表すpriceという変数と、税率を表すtax_rateを定義して」と具体的に命名を指示すると、後から人間が読みやすいコードになります。

リスト（配列）の基本と罠

リストは「順番のあるデータの列」です。複数人の名前や、時系列データを入れるのに適しています。しかし、「存在しないインデックス（番号）」にアクセスすると即座にプログラムがクラッシュします（IndexError）。

# ❌ IndexError（要素は3つしかないのに3番目=4つ目にアクセスしようとした）

辞書（dict）による名前付きデータ

リストは順番で管理しますが、辞書は「キー（名前）」で管理します。APIのレスポンス（JSON形式）は、Pythonでは自動的にこの「辞書」として扱われるため、AIアプリ開発において最も重要なデータ構造です。

PYTHON

    'id': 1,
    'name': 'Bob',
    'role': 'admin'

条件分岐（If文）

▼

💡 状況に応じてプログラムの動きを変える

なぜIf文が重要なのか

「もし〜だったら〜する」という条件分岐は、アプリの心臓部です。「ログインしていればマイページを表示、していなければログイン画面へ」といった制御はすべてIf文で行われます。

PYTHON 
age = 20

if age >= 20:
    print('お酒が買えます')
elif age >= 18:
    print('成人ですがお酒は買えません')
else:
    print('未成年です')

# 💡 Pythonではインデント（字下げ）がそのままブロック（塊）を意味します。

真偽値（True/False）の判定の省略

Pythonでは、空のリスト []、空の文字列 ''、数値の 0、None はすべて False として扱われます。これを利用してコードを短く書くのがプロのテクニックです。

PYTHON 
name = ''

# ❌ 冗長な書き方
if name == '':
    print('名前が空です')

# ✅ プロらしいスマートな書き方
if not name:
    print('名前が空です')  # nameが空ならFalseなので、notで反転して実行される

繰り返し処理（ForループとWhileループ）

▼

💡 面倒な単純作業を一瞬で終わらせる

Forループによる一括処理

リストや辞書の中身を1つずつ取り出して処理します。100万件のデータがあっても、数行のForループで処理できるのがプログラミングの最大の利点です。

PYTHON 
users = ['Alice', 'Bob', 'Charlie']

for user in users:
    print(f'Hello, {user}!')

# 辞書をループする場合
scores = {'math': 90, 'english': 85}
for subject, score in scores.items():
    print(f'{subject}の点数は{score}点です')

リスト内包表記（Pythonならではの必殺技）

ループを回して新しいリストを作る際、Pythonでは「リスト内包表記」という1行で書ける強力な構文があります。AIが生成するコードにも頻繁に登場します。

PYTHON 
numbers = [1, 2, 3, 4, 5]

# ❌ 冗長な書き方
squared = []
for n in numbers:
    squared.append(n * 2)

# ✅ リスト内包表記
squared = [n * 2 for n in numbers]
print(squared)  # [2, 4, 6, 8, 10]

関数の定義と呼び出し

▼

💡 何度も使う処理をひとまとめにして再利用する

関数の基本（def）

処理の塊に「名前」をつけて、いつでも呼び出せるようにしたものが関数です。引数（入力データ）を受け取り、戻り値（結果）を返すのが基本です。

PYTHON 
def calculate_tax(price, tax_rate=0.1):
    """税込み価格を計算する関数"""
    return int(price * (1 + tax_rate))

# 呼び出し
print(calculate_tax(1000))        # 1100（デフォルトの10%が適用）
print(calculate_tax(1000, 0.08))  # 1080（8%が適用）

なぜ関数に分けるのか？

全てをベタ書きすると、同じ処理を何度も書く羽目になり、修正時に「全ての箇所を直す」という地獄を見ます。関数に分けることで、バグが起きたときにその関数だけを直せば済むようになります。

堅牢なコードの要（try-exceptによる例外処理）

▼

💡 AIのコードによく欠けている「エラー発生時の備え」

プログラムを止めないためのtry-except

API通信のエラー、ファイルの読み込み失敗、0での割り算など、予期せぬエラーでアプリ全体がクラッシュするのを防ぎます。AIは正常系のコード（Happy Path）を書きがちなので、人間が手動でエラー処理を補強するスキルが重要です。

PYTHON 
def divide_numbers(a, b):
    try:
        result = a / b
        print(f'結果は {result} です')
    except ZeroDivisionError:
        # 特定のエラーをキャッチ
        print('エラー: 0で割ることはできません')
    except TypeError:
        # 別のエラーをキャッチ
        print('エラー: 数値を入力してください')
    except Exception as e:
        # 全ての予期せぬエラーをキャッチ（最後の砦）
        print(f'予期せぬエラーが発生しました: {e}')
    finally:
        # エラーの有無に関わらず最後に必ず実行される（ファイルのクローズなどに使う）
        print('処理を終了します')

divide_numbers(10, 0)

よくあるエラーとアンチパターン

▼

💡 初学者が陥りやすい罠と解決策

ループ中の破壊的変更の罠

リストを for ループで回している最中に、そのリスト自身の要素を `remove()` すると、Pythonが内部で管理しているインデックスが狂い、いくつかの要素がスキップされてしまいます。

PYTHON 
# ❌ 悪い例（ループ中の要素削除）
nums = [1, 2, 2, 3, 4]
for n in nums:
    if n == 2:
        nums.remove(n)  
# 結果: [1, 2, 3, 4] （2が1つ残ってしまう！）

# ✅ 良い例（新しいリストを作る）
new_nums = [n for n in nums if n != 2]
# 結果: [1, 3, 4] （正しく2が全て除去された）

ミュータブルなデフォルト引数

関数のデフォルト引数にリスト `[]` や辞書 `{}` を使うと、関数を呼び出すたびに同じリストが使い回されてしまい、データが蓄積していくという恐ろしいバグを引き起こします。

PYTHON 
# ❌ アンチパターン
def add_item(item, target_list=[]):
    target_list.append(item)
    return target_list

print(add_item('A'))  # ['A']
print(add_item('B'))  # ['A', 'B'] （空のリストを期待したのに前回のが残っている！）

# ✅ 正解パターン
def add_item_safe(item, target_list=None):
    if target_list is None:
        target_list = []
    target_list.append(item)
    return target_list

class Task(BaseModel):
    id: int
    title: str
    completed: bool = False

tasks_db = []

@app.get('/tasks', response_model=List[Task])
def get_tasks():
    return tasks_db

@app.post('/tasks', response_model=Task)
def create_task(task: Task):
    # Pydanticが型チェックを済ませているので、task.titleなどが安全に使える
    tasks_db.append(task)
    return task

上級編: デコレータの魔法

▼

💡 関数の外側から機能を「トッピング」する技術

デコレータとは何か？

FastAPIの `@app.get('/')` などで頻出する「@（アットマーク）」から始まる記述をデコレータと呼びます。これは、既存の関数の中身を一切いじらずに、実行前・実行後の処理（ログ出力、実行時間計測、権限チェックなど）をガバッと被せる（ラップする）仕組みです。

PYTHON 
import time

# 実行時間を計測する自作デコレータ
def time_logger(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)  # 元の関数を実行
        end = time.time()
        print(f'{func.__name__}の実行時間: {end - start:.4f}秒')
        return result
    return wrapper

# @をつけるだけで機能が追加される！
@time_logger
def heavy_process():
    time.sleep(1)
    return '処理完了'

print(heavy_process())
# 出力:
# heavy_processの実行時間: 1.0005秒
# 処理完了

上級編: ジェネレータとメモリ節約

▼

💡 大量のデータを少しずつ処理するエコな仕組み

yield キーワードによる一時停止

AIのチャット応答が「少しずつ流れてくる（ストリーミング）」のもジェネレータの仕組みの応用です。リストで全件を一度にメモリに載せるのではなく、`yield` を使うことで「1つ処理したら関数を一時停止し、呼ばれたら再開する」という省メモリなループを実現できます。

PYTHON 
def read_huge_file(file_path):
    # ファイル全体を読み込まず、1行ずつ返すジェネレータ
    with open(file_path, 'r') as f:
        for line in f:
            yield line.strip()  # 値を返して一時停止

# 100GBのファイルでもメモリ不足にならない
# for line in read_huge_file('huge_data.csv'):
#     print(line)

# カウントダウンの例
def countdown(num):
    print('カウント開始')
    while num > 0:
        yield num
        num -= 1

gen = countdown(3)
print(next(gen)) # カウント開始 -> 3
print(next(gen)) # 2
print(next(gen)) # 1

▼

普通の変数を変更しても、Reactの画面は更新されません。画面を書き換えたいデータ（いいね数、入力中の文字、開閉状態など）は、必ず `useState` を使ってReactに「これが更新されたら画面を描き直して」と教える必要があります。

REACT 
import { useState } from 'react';

function Counter() {
  // count: 現在の値, setCount: 値を更新する関数
  // useState(0): 初期値を0に設定
  const [count, setCount] = useState(0);

  return (

  );
}

配列やオブジェクトのState更新の罠

Reactは「前のデータと今のデータが同じか」を参照（メモリのアドレス）で比較します。配列に `push()` して中身を変えても、配列という箱自体は変わっていないため、Reactは「変化なし」と判断して画面を更新しません。

REACT 
const [items, setItems] = useState(['Apple']);

// ❌ 画面が更新されない（元の配列を直接いじっている）
const addBad = () => {
  items.push('Banana');
  setItems(items);
};

// ✅ 画面が正しく更新される（新しい配列を作ってセットしている）
const addGood = () => {
  setItems([...items, 'Banana']);
};

REACT 
import { useEffect, useState } from 'react';

// この関数が true を返せば、TSは obj を Bird 型だと確信する

    // animal は完全に Bird 型として扱われ、サジェストも効く！
    animal.fly();
  } else {
    console.log('飛べません');

Web開発の土台：HTTP・JSON・クライアント/サーバー

クライアント/サーバーとHTTPの役割

Webアプリは、ブラウザ（クライアント）がリクエストを送り、サーバーがレスポンスを返す往復で成り立ちます。HTTP はこの往復の約束事であり、メソッド（GET/POST/PATCH/DELETE）、URL、ヘッダ、ボディ、ステータスコードで意味を表現します。FastAPI はこの HTTP 契約を Python コードで宣言し、TypeScript 側はその契約を型として扱うことで、実装と仕様のズレを早期に減らせます。ここが曖昧だと後続の非同期処理・SSE・認証・キャッシュすべてが混乱するため、まず通信レイヤーの語彙を固定することが最重要です。

Python/TS 
GET /api/lessons HTTP/1.1
Host: 127.0.0.1:8765
Accept: application/json

ステータスコードとエラーハンドリング

2xx は成功、4xx はクライアント入力起因、5xx はサーバー内部起因、という大枠を運用で徹底するとデバッグ効率が上がります。例えば 422 は入力バリデーション不一致、404 は対象なし、409 は競合、500 は想定外障害です。フロントは `ok` だけ見るのではなく、ステータスとエラーボディの構造を合わせて扱うべきです。FastAPI では HTTPException を使って失敗を明示し、TypeScript 側で `ApiError` のような統一型へ落とし込むと、画面側の分岐が安定します。

Python/TS 
if (!response.ok) {
  throw new Error(`HTTP ${response.status}`)

TypeScript 概要：型で壊れ方を先に制御する

▼

💡 any回避・ユニオン・型狭窄・APIレスポンス型の設計

TypeScriptを使う理由

TypeScript は JavaScript の上位互換で、静的型検査を追加する言語です。実行時の挙動は JS ですが、ビルド時に型不一致を検出できます。特に API 駆動開発では「サーバーが返す shape」と「画面が期待する shape」の不一致が主なバグ源なので、型で先に落とす効果が大きいです。`any` を多用すると TypeScript の価値は消えるため、unknown からの型狭窄や明示的な型定義を基本とします。

Python/TS 
type LessonMeta = {
  id: string
  title: string
  order: number
}

ユニオン型と型狭窄

実務データは単一型でないことが多く、ユニオン型（A | B）で扱います。分岐時に `in`、`typeof`、ユーザー定義 type guard を使い、処理可能な型へ狭窄します。これにより「存在しないプロパティアクセス」をコンパイル時に排除できます。レスポンス成功/失敗を discriminated union で定義すると、UI分岐を安全に書けます。

Python/TS 
type Result =
  | { ok: true; data: LessonMeta[] }
  | { ok: false; message: string }

strictモード運用

tsconfig の `strict: true` を基準にし、必要な箇所だけ局所的に緩和する方針が保守上有利です。`noImplicitAny`、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes` なども段階導入すると品質が安定します。型エラーは面倒ではなく「将来の事故前払い」です。FastAPI 側変更時にフロントが即エラーになる状態は、むしろ望ましい監視機構です。

Python/TS 
{
  "compilerOptions": {
    "strict": true,
    "noUnusedLocals": true
  }
}

FastAPI 概要：ルーティング・バリデーション・依存性注入

▼

💡 最小構成から実務構成へ進むための基礎

ルート関数と自動ドキュメント

FastAPI は Python の型ヒントを活用して OpenAPI ドキュメントを自動生成します。`@app.get` などでエンドポイントを宣言し、関数シグネチャにクエリ・パス・ボディを記述するだけで、/docs で実行可能な仕様書が得られます。これはバックエンド都合ではなく、フロント連携と QA 自動化に直接効く資産です。まずは「仕様がコードから再現できる」状態を作ることが重要です。

Python/TS 
from fastapi import FastAPI

app = FastAPI()

@app.get('/api/health')
def health() -> dict:
    return {'status': 'ok'}

Pydanticモデルで入力を閉じる

リクエストボディは Pydantic モデルで受けることで、型・範囲・必須性を統一的に検証できます。`extra='forbid'` を使えば未定義フィールドを拒否でき、契約逸脱を防げます。入力検証をルート内で手書きすると重複と漏れが増えるため、モデルに寄せるのが合理的です。さらに response_model を指定すると、出力形状まで拘束でき、誤って内部情報を返す事故を減らせます。

Python/TS

    model_config = ConfigDict(extra='forbid')
    lesson_id: str
    choice_index: int = Field(..., ge=0)

DB セッション、認証情報、設定値などは `Depends` に切り出すと、ルート関数を薄く保てます。依存関数を差し替えるだけでテストが書きやすくなり、関心分離が進みます。重要なのは「依存を増やすこと」ではなく「責務を薄く保つこと」です。最初から大型DI設計に走らず、まず DB セッションと current_user 程度の抽出から始めるのが安全です。

Python/TS 
@router.get('/items')
def list_items(session: Session = Depends(get_session)):
    return session.scalars(select(Item)).all()

制御構造の特殊仕様（for...else等）

▼

💡 他言語の経験者が引っかかるPython特有の文法

for...else 構文と while...else 構文

ループ文に `else` を付けることができます。これは「ループが `break` されずに最後まで正常に完走した場合」のみ実行されます。検索処理などでフラグ変数を使わずに済む便利な機能ですが、初見では混乱しやすく試験で狙われます。

PYTHON-CERT 
for n in range(2, 10):
    for x in range(2, n):
        if n % x == 0:
            print(n, 'equals', x, '*', n//x)
            break
    else:
        # breakでループを抜けなかった場合のみここに来る
        print(n, 'is a prime number')

# 出力例:
# 2 is a prime number
# 3 is a prime number
# 4 equals 2 * 2...

pass, break, continue の違い

`break` はループを完全に抜け出し、`continue` は現在の周をスキップして次の周へ行きます。`pass` は「何もしない」ための文で、構文上文が必要な場所（空のクラスや関数など）のプレースホルダーとして使われます。

PYTHON-CERT 
for num in range(2, 10):
    if num % 2 == 0:
        print("Found an even number", num)
        continue # これ以下の処理を飛ばして次のループへ
    print("Found a number", num)

def initlog(*args):

デフォルト引数の評価タイミング（罠）

関数のデフォルト値は「関数が定義された時に一度だけ」評価されます。そのため、リストや辞書などのミュータブルなオブジェクトをデフォルト値にすると、呼び出し間で値が共有されてしまいます。

PYTHON-CERT

    return L

print(f(2)) # [1, 2] <- 空にならない！
print(f(3)) # [1, 2, 3]

# ✅ 正しい書き方
def f_safe(a, L=None):
    if L is None:
        L = []
    L.append(a)
    return L

エラーと例外処理の階層

▼

💡 例外クラスの継承ツリーとraiseの使い方

例外処理のキャッチ順序

例外クラスには階層（継承ツリー）があります。例えば、`FileNotFoundError` は `OSError` の子クラス、さらにその親は `Exception` です。`except` 節は上から順に評価されるため、子クラスから先に書かないと、親クラスがすべてをキャッチしてしまいます。

PYTHON-CERT 
class B(Exception):
    pass
class C(B):
    pass
class D(C):
    pass

for cls in [B, C, D]:
    try:
        raise cls()
    except D:
        print("D")
    except C:
        print("C")
    except B:
        print("B")

# 出力は B, C, D の順になる

💡 クラス変数とインスタンス変数の違い

クラス定義と __init__ メソッド

Pythonのクラスは実行時にオブジェクトとして作成されます。`__init__` はコンストラクタ（初期化メソッド）で、インスタンス作成時に自動で呼ばれます。メソッドの第一引数には必ず `self`（自分自身のインスタンス）を指定します。

PYTHON-CERT 
class Complex:
    def __init__(self, realpart, imagpart):
        self.r = realpart
        self.i = imagpart

x = Complex(3.0, -4.5)
print(x.r, x.i) # 3.0 -4.5

クラス変数とインスタンス変数の罠

クラス変数（`def`の外に書く）は全てのインスタンスで共有され、インスタンス変数（`self.`で書く）は個別のインスタンス専用になります。ミュータブルなオブジェクト（リストなど）をクラス変数にすると悲劇が起きます。

PYTHON-CERT 
# ❌ 危険なクラス変数
class Dog:
    tricks = []             # すべての犬で共有されるリスト
    def __init__(self, name):
        self.name = name
    def add_trick(self, trick):
        self.tricks.append(trick)

d = Dog('Fido')
e = Dog('Buddy')
d.add_trick('roll over')
print(e.tricks) # ['roll over'] <- Buddyも芸を覚えてしまう！

# ✅ 正しいインスタンス変数
class SafeDog:
    def __init__(self, name):
        self.name = name
        self.tricks = []    # 犬ごとに個別のリスト
    def add_trick(self, trick):
        self.tricks.append(trick)
