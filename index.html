01

変数とデータ型 — データの「入れ物」を理解する

▼

💡 AIが書いたコードでよく見るもの

生成AIにアプリを作らせると、name = "太郎" や count = 0 のようなコードが必ず登場します。
これは変数にデータを格納している処理です。変数は「名前のついた箱」だと思ってください。

Python 
# 文字列型（str）— テキストデータ
name = "太郎"
greeting = "こんにちは"

# 数値型（int, float）— 数字データ
age = 25
height = 170.5

# ブール型（bool）— True か False
is_student = True

# 型を確認する
print(type(name))      # <class 'str'>
print(type(age))       # <class 'int'>
print(type(is_student)) # <class 'bool'>

🔍 なぜ型が重要？

AIが生成したコードでエラーが出たとき、型の不一致が原因であることが非常に多いです。
例えば、数字の 25 と文字列の "25" は Python では別物です。

# ❌ エラーになる例
age = "25"         # これは文字列
new_age = age + 1  # TypeError! 文字列と数値は足せない

# ✅ 正しい書き方
age = int("25")     # 文字列を数値に変換
new_age = age + 1  # → 26

🎯 実践ポイント

AIが書いたコードを見たら、まず変数名と型を確認しましょう。
type() 関数を使えば、いつでも変数の型をチェックできます。

02

リストと辞書 — データをまとめて管理する

▼

💡 なぜリストと辞書を学ぶ？

AIが生成するアプリでは、ユーザーのリスト、商品カタログ、設定データなど、複数のデータをまとめて扱う場面が頻出します。
Pythonでは list（リスト）と dict（辞書）がその役割を担います。

Python 
# リスト — 順番のあるデータの集まり
fruits = ["りんご", "バナナ", "みかん"]

# インデックス（番号）でアクセス ※0から始まる！
print(fruits[0])   # → "りんご"
print(fruits[-1])  # → "みかん"（最後の要素）

# 要素の追加・削除
fruits.append("ぶどう")  # 末尾に追加
fruits.remove("バナナ")  # 指定した要素を削除

# リストの長さ
print(len(fruits))  # → 3
Python 
# 辞書 — キーと値のペア（JSONとほぼ同じ構造！）
user = {
    "name": "太郎",
    "age": 25,
    "email": "taro@example.com",
    "hobbies": ["読書", "プログラミング"]
}

# キーで値にアクセス
print(user["name"])     # → "太郎"
print(user["hobbies"])  # → ["読書", "プログラミング"]

# 値の更新と追加
user["age"] = 26           # 更新
user["city"] = "東京"      # 新しいキーを追加

# 安全にアクセス（キーが無くてもエラーにならない）
phone = user.get("phone", "未設定")  # → "未設定"

🎯 AIコードを読むコツ

APIレスポンスは辞書（dict）形式で返ってくることがほとんどです。
response["data"]["items"] のようなネストされたアクセスを見たら、「辞書の中の辞書の中のリスト」と読み解きましょう。

03

関数 — 処理をまとめて再利用する

▼

💡 関数とは？

関数は「名前をつけた処理のかたまり」です。同じ処理を何度も書かずに済むようにするためのしくみです。
AIが生成するコードでは def キーワードで始まる関数が大量に出てきます。

Python 
# 基本的な関数
def greet(name):
    """名前を受け取って挨拶を返す関数"""
    return f"こんにちは、{name}さん！"

# 関数を呼び出す
message = greet("太郎")
print(message)  # → "こんにちは、太郎さん！"

# デフォルト引数 — 引数を省略できるようにする
def create_user(name, role="一般", active=True):
    return {
        "name": name,
        "role": role,
        "active": active
    }

# いろいろな呼び方
user1 = create_user("太郎")                    # デフォルト値を使用
user2 = create_user("花子", role="管理者")       # roleだけ指定
user3 = create_user("次郎", active=False)       # activeだけ指定

🔍 f文字列（f-string）を理解する

f"..." はPython 3.6以降の便利機能で、文字列の中に変数を埋め込めます。
AIが書くコードで頻出するので、しっかり覚えましょう。

name = "太郎"
age = 25

# 古い書き方（AIが生成することもある）
msg1 = "私は" + name + "で、" + str(age) + "歳です"

# f文字列（推奨！読みやすい）
msg2 = f"私は{name}で、{age}歳です"

# 式も埋め込める
msg3 = f"来年は{age + 1}歳です"

score = 85

if score >= 90:

🎯 Pythonの特徴：インデント

Pythonではインデント（字下げ）が文法の一部です。他の言語では {} で囲みますが、Pythonではスペース4つで構造を表します。
AIが生成したコードのインデントがずれるとエラーになるので注意！

05

実践編 — API通信とファイル操作

▼

💡 実際のアプリでよくやること

AIがアプリを作るとき、外部APIからデータを取得したり、ファイルを読み書きしたりするコードを生成します。
これらのパターンを理解しておくと、AIのコードを修正・拡張する力がつきます。

Python 
import requests
import json

# APIからデータを取得する典型パターン
def fetch_weather(city):
    """天気情報を取得する"""
    url = f"https://api.example.com/weather?city={city}"

    try:
        response = requests.get(url)
        response.raise_for_status()  # エラーなら例外を投げる

        data = response.json()  # JSONを辞書に変換
        return {
            "city": data["name"],
            "temp": data["main"]["temp"],
            "desc": data["weather"][0]["description"]
        }
    except requests.RequestException as e:
        print(f"エラー: {e}")
        return None

# 使い方
weather = fetch_weather("Tokyo")
if weather:
    print(f"{weather['city']}の気温: {weather['temp']}°C")

🔍 try-except を理解する

try-except はエラーハンドリングの仕組みです。ネットワークエラーやファイルの読み取りエラーなど、
予期しないエラーでアプリが落ちないようにするために使います。
AIが生成するコードでは必ずと言っていいほど登場します。

データ構造の極意（辞書とタプルの使い分け）

▼

💡 リストだけでは対応できない実務のデータ設計

なぜリスト以外が必要なのか？

リストは「順番のあるデータの列」を扱うのには最適ですが、「名前で検索したい」「絶対に書き換えられたくない」場合には不向きです。実務では辞書（dict）やタプル（tuple）を使い分けます。

PYTHON 
user = {'id': 1, 'name': 'Alice'}
point = (10, 20)  # 一度作ったら変更できない（イミュータブル）

辞書の高度な使い方

APIのJSONレスポンスはPythonではほぼ辞書として扱われます。キーの存在確認やデフォルト値の取得は、エラーを防ぐ基本です。

PYTHON 
# ❌ キーがないとエラーになる
# name = user['nickname']

# ✅ getメソッドで安全に取得
nickname = user.get('nickname', '名無し')

# キーと値のループ
for key, value in user.items():

07

堅牢なコードの要（try-exceptによる例外処理）

▼

💡 AIのコードによく欠けている「エラー発生時の備え」

プログラムを止めないためのtry-except

API通信のエラー、ファイルの読み込み失敗、0での割り算など、予期せぬエラーでアプリ全体がクラッシュするのを防ぎます。AIは正常系のコードを書きがちなので、手動で補強するスキルが重要です。

PYTHON 
try:
    result = 10 / 0
except ZeroDivisionError as e:
    print(f'ゼロで割れません: {e}')
except Exception as e:
    print(f'予期せぬエラー: {e}')
finally:
    print('処理が終了しました')

よくあるエラーとアンチパターン

▼

💡 初学者が陥りやすい罠と解決策

インデントエラーとスコープ

Pythonはインデント（字下げ）でブロックを認識します。スペース4つが標準ですが、タブとスペースが混在するとエラーになります。また、ループの中で変数を定義すると思わぬ上書き（スコープの汚染）が起きます。

PYTHON 
# ❌ 悪い例（リストの破壊的変更）
nums = [1, 2, 3]
for n in nums:
    if n == 2:
        nums.remove(n)  # ループ中に要素を消すと挙動がおかしくなる

# ✅ 良い例（新しいリストを作る）
new_nums = [n for n in nums if n != 2]

Final Mission: FastAPIによるTodoバックエンド構築

▼

💡 APIサーバーを作って、フロントと通信してみよう

FastAPIとPydanticの連携

実際のAPIでは、「どんなデータが送られてくるか」をPydanticのSchema（型）で定義し、FastAPIのルーターと結びつけます。これにより、自動的にバリデーション（入力値チェック）が行われます。

PYTHON 
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List

app = FastAPI()

class Task(BaseModel):
    id: int
    title: str
    completed: bool = False

tasks_db = []

@app.get('/tasks', response_model=List[Task])
def get_tasks():
    return tasks_db

@app.post('/tasks', response_model=Task)
def create_task(task: Task):
    tasks_db.append(task)
    return task

CORS（Cross-Origin Resource Sharing）の設定

Reactなどのフロントエンド（例: localhost:3000）からAPI（localhost:8000）を呼び出す場合、ポート番号が違うためブラウザが通信をブロックします。これを許可するのがCORS設定です。

PYTHON 
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=['*'],  # 実際はフロントのURLを指定する
    allow_methods=['*'],
    allow_headers=['*'],
)

上級編: デコレータの魔法

▼

💡 関数の外側から機能を「トッピング」する技術

デコレータとは？

FastAPIの @app.get('/') などで頻出するデコレータは、既存の関数を変更せずに、実行前後の処理（ログ出力や権限チェック）を追加する仕組みです。

PYTHON 
def my_logger(func):
    def wrapper(*args, **kwargs):
        print(f'{func.__name__} が呼ばれました')
        result = func(*args, **kwargs)
        print(f'{func.__name__} が終了しました')
        return result
    return wrapper

@my_logger
def say_hello(name):
    print(f'Hello, {name}!')

上級編: ジェネレータとメモリ節約

▼

💡 大量のデータを少しずつ処理するエコな仕組み

yield キーワード

AIのチャット応答が「少しずつ流れてくる」のもジェネレータの仕組みです。リストで全件を一度にメモリに載せるのではなく、必要な分だけ順番に取り出します。

PYTHON 
def count_up_to(max_num):
    count = 1
    while count <= max_num:
        yield count  # ここで一時停止して値を返す
        count += 1

counter = count_up_to(3)
print(next(counter))  # 1
print(next(counter))  # 2

01

JSX — HTMLのようでHTMLではない

▼

💡 JSXとは？

JSX（JavaScript XML）は、JavaScript の中にHTMLのようなコードを書ける特別な構文です。
AIがReactアプリを生成すると、必ず return (<div>...</div>) のようなJSXが登場します。

React (JSX) 
// JSXの基本 — HTMLに似ているが、JavaScriptの式が使える
function WelcomeCard() {
  const userName = "太郎";
  const isLoggedIn = true;

  return (
    <div className="card">
      {/* 中カッコ {} でJavaScriptの式を埋め込む */}
      <h2>こんにちは、{userName}さん！</h2>

      {/* 条件付きレンダリング */}
      {isLoggedIn ? (
        <p>ログイン中です ✅</p>
      ) : (
        <p>ログインしてください</p>
      )}

      {/* 注意: class ではなく className */}
      <button className="btn-primary">クリック</button>
    </div>
  );
}

⚠️ JSXの注意点

  • class → className（classはJSの予約語）
  • for → htmlFor（forもJSの予約語）
  • 必ず1つのルート要素で囲む（<div> か <>...</>）
  • タグは必ず閉じる（<img />、<br />）

コンポーネントとProps — UIの部品化

▼

💡 コンポーネントとは？

コンポーネントはUIの再利用可能な部品です。レゴブロックのように小さな部品を組み合わせてUIを作ります。

// UserCard コンポーネント — 再利用可能なUIパーツ

      <img src={avatar} alt={name} />

// 親コンポーネントで使う — 同じ部品を異なるデータで表示

03

useState — 画面の状態を管理する

▼

💡 なぜ useState が必要？

普通の変数 (let count = 0) を変更しても、Reactは画面を更新しません。
画面と連動するデータには useState を使う必要があります。
これがReactの最も重要な概念の一つです。

React (JSX) 
import { useState } from 'react';

function Counter() {
  // [現在の値, 値を更新する関数] = useState(初期値)
  const [count, setCount] = useState(0);
  const [name, setName] = useState("");

  return (
    <div>
      <p>カウント: {count}</p>

      {/* ボタンクリックで状態を更新 → 画面も自動更新 */}
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
      <button onClick={() => setCount(0)}>
        リセット
      </button>

      {/* 入力欄と状態の連動 */}
      <input
        value={name}
        onChange={(e) => setName(e.target.value)}
        placeholder="名前を入力"
      />
      <p>入力値: {name}</p>
    </div>
  );
}

⚠️ よくある間違い

  • 直接代入しない: count = count + 1 ❌ → setCount(count + 1) ✅
  • stateは非同期で更新される: setCountの直後にcountを読むと古い値になることがある
  • オブジェクトのstateは新しいオブジェクトで更新: setUser({...user, name: "new"})
04

useEffect — 副作用を管理する

▼

💡 副作用（Side Effect）とは？

「画面の描画」以外の処理のことです。例えば：

  • APIからデータを取得する
  • タイマーをセットする
  • ブラウザのタイトルを変更する
  • localStorageにデータを保存する
React (JSX) 
import { useState, useEffect } from 'react';

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  // useEffect(実行する処理, [依存する値の配列])
  useEffect(() => {
    // userIdが変わるたびにデータを再取得
    setLoading(true);

    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      });
  }, [userId]);  // ← 依存配列：userIdが変わったら再実行

  if (loading) return <p>読み込み中...</p>;

  return (
    <div>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </div>
  );
}

🔍 依存配列のパターン

// パターン1: マウント時に1回だけ実行
useEffect(() => {
  console.log("コンポーネントが表示された");
}, []);  // 空の配列 = 最初の1回だけ

// パターン2: 特定の値が変わったら実行
useEffect(() => {
  document.title = `メッセージ(${count})`;
}, [count]);  // countが変わるたびに実行

// パターン3: クリーンアップ付き
useEffect(() => {
  const timer = setInterval(() => {
    console.log("tick");
  }, 1000);

  return () => clearInterval(timer);  // クリーンアップ
}, []);

実践編 — Todoアプリを読み解く

▼

💡 AIが作ったTodoアプリを解剖する

ここまでの知識を使って、AIが生成しそうなTodoアプリのコードを読み解きましょう。
useState、イベントハンドリング、リストのレンダリング、条件付き表示がすべて含まれています。

React (JSX) 
import { useState } from 'react';

function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState("");

  // タスクを追加する関数
  const addTodo = () => {
    if (!input.trim()) return;  // 空文字は追加しない

    setTodos([
      ...todos,  // 既存のタスクを展開（スプレッド構文）
      { id: Date.now(), text: input, done: false }
    ]);
    setInput("");  // 入力欄をクリア
  };

  // タスクの完了状態を切り替える関数
  const toggleTodo = (id) => {
    setTodos(
      todos.map(todo =>
        todo.id === id
          ? { ...todo, done: !todo.done }  // 該当タスクのdoneを反転
          : todo                            // それ以外はそのまま
      )
    );
  };

  return (
    <div>
      <h1>📝 Todoリスト</h1>

      <div>
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          placeholder="新しいタスク"
          onKeyDown={(e) => e.key === "Enter" && addTodo()}
        />
        <button onClick={addTodo}>追加</button>
      </div>

      {/* リストをmapで描画 */}
      <ul>
        {todos.map(todo => (
          <li
            key={todo.id}
            onClick={() => toggleTodo(todo.id)}
            style={{
              textDecoration: todo.done ? "line-through" : "none"
            }}
          >
            {todo.text}
          </li>
        ))}
      </ul>

      <p>残り: {todos.filter(t => !t.done).length} 件</p>
    </div>
  );
}

🎯 覚えておくべきパターン

  • ...todos — スプレッド構文。配列やオブジェクトを展開する
  • todos.map() — 配列の各要素をJSXに変換して描画
  • key={todo.id} — Reactがどの要素が変わったか判別するために必要

🔷 TypeScript — 型で安全なコードを書く

TypeScriptはJavaScriptに「型」を追加した言語です。
AIが生成するReactアプリの多くがTypeScriptで書かれています。型があることでバグを未然に防げます。

01

基本の型 — TypeScriptの第一歩

▼

💡 なぜTypeScript？

JavaScriptでは let x = "hello"; x = 42; のように、同じ変数に違う型を入れてもエラーになりません。
これが大きなアプリではバグの原因になります。TypeScriptはコードを書いている段階でミスを教えてくれるツールです。

TypeScript 
// 基本の型注釈 — 変数名: 型
let userName: string = "太郎";
let age: number = 25;
let isActive: boolean = true;

// 配列の型
let scores: number[] = [85, 90, 78];
let names: string[] = ["太郎", "花子"];

// ❌ 型エラー！ — エディタが赤線で警告してくれる
userName = 42;              // Error: number を string に代入できません
scores.push("百点");       // Error: string を number[] に追加できません

// 関数の型 — 引数と戻り値に型をつける
function greet(name: string): string {
  return `こんにちは、${name}さん！`;
}

// アロー関数の型
const add = (a: number, b: number): number => a + b;

🎯 型推論

TypeScriptは賢いので、多くの場合型を自動で推測してくれます。
let x = "hello" と書けば、自動的に string 型だと判断されます。
AIが書くコードで型注釈がない場合も、TypeScriptが内部で型を管理しています。

02

interfaceとtype — オブジェクトの形を定義する

▼

💡 なぜ形を定義する？

APIから返ってくるデータやコンポーネントのpropsなど、オブジェクトの構造を明確にすると、
タイポや必須フィールドの漏れをコードを書く段階で検出できます。

TypeScript 
// interface — オブジェクトの「設計図」
interface User {
  id: number;
  name: string;
  email: string;
  age?: number;        // ? = オプショナル（なくてもOK）
  role: "admin" | "user";  // ユニオン型（どちらか）
}

// 使い方
const user: User = {
  id: 1,
  name: "太郎",
  email: "taro@example.com",
  role: "admin"
};

// ❌ エラー: 'manager' は 'admin' | 'user' に代入できません
const badUser: User = {
  id: 2,
  name: "花子",
  email: "hanako@example.com",
  role: "manager"  // Error!
};

// type エイリアス — 型に名前をつける
type Status = "pending" | "active" | "inactive";
type UserList = User[];

🔍 interface vs type

基本的にはどちらを使っても大丈夫です。AIが書くコードでは両方出てきます。
一般的なガイドライン：

  • interface: オブジェクトの形を定義するとき（拡張しやすい）
  • type: ユニオン型やプリミティブ型の別名をつけるとき
03

ジェネリクス — 型を「引数」にする

▼

💡 ジェネリクスとは？

ジェネリクスは「使う側が型を決められる」しくみです。
AIが書くコードで <T> のような記法を見かけたら、それがジェネリクスです。
最初は難しく感じますが、「型の変数」だと思えばシンプルです。

TypeScript 
// ジェネリクスなし — 型ごとに関数を作る必要がある
function getFirstString(arr: string[]): string {
  return arr[0];
}
function getFirstNumber(arr: number[]): number {
  return arr[0];
}

// ジェネリクスあり — どんな型でも使える1つの関数！
function getFirst<T>(arr: T[]): T {
  return arr[0];
}

// 使う側で型が決まる
const name = getFirst<string>(["太郎", "花子"]);  // string型
const num = getFirst([10, 20, 30]);          // number型（型推論）

// 実践: APIレスポンスの型
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

// ユーザー一覧のレスポンス
type UsersResponse = ApiResponse<User[]>;
// 単一ユーザーのレスポンス
type UserResponse = ApiResponse<User>;

実践編 — 型ガードとユーティリティ型

▼

💡 型ガード（Type Guard）とは？

変数の型が複数の可能性がある場合に、条件分岐で型を絞り込むテクニックです。
AIが生成するコードでは、APIレスポンスの型チェックなどで頻出します。

TypeScript 
// ユニオン型と型ガード
type Result =
  | { success: true; data: User }
  | { success: false; error: string };

function handleResult(result: Result) {
  if (result.success) {
    // ✅ TypeScriptが自動でresult.dataの型を認識
    console.log(result.data.name);
  } else {
    // ✅ TypeScriptが自動でresult.errorの型を認識
    console.error(result.error);
  }
}

// よく使うユーティリティ型
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
}

// Partial — すべてオプショナルに
type UpdateUser = Partial<User>;
// → { id?: number; name?: string; ... }

// Pick — 一部だけ取り出す
type PublicUser = Pick<User, "id" | "name">;
// → { id: number; name: string }

// Omit — 一部を除外
type SafeUser = Omit<User, "password">;
// → { id: number; name: string; email: string }

🎯 AIが書いたTypeScriptを修正するコツ

  • 赤い波線が出たら、エディタのホバーで型エラーの内容を確認
  • any 型が多い場合は、具体的な型に置き換えるとバグが減る
  • as による型キャストは最終手段。まずは型ガードを試す

TYPESCRIPT 
interface Bird {
  fly: () => void;
}

// 戻り値の「obj is Bird」が型ガードの印
function isBird(obj: any): obj is Bird {
  return obj !== null && typeof obj.fly === 'function';
}

function move(animal: unknown) {
  if (isBird(animal)) {
    // ここに入ると animal は完全に Bird 型として扱われる
    animal.fly();
  } else {
    console.log('飛べません');
  }
}

01

Web開発の土台：HTTP・JSON・クライアント/サーバー

▼

💡 FastAPIとTypeScriptに入る前に押さえる通信の基礎

クライアント/サーバーとHTTPの役割

Webアプリは、ブラウザ（クライアント）がリクエストを送り、サーバーがレスポンスを返す往復で成り立ちます。HTTP はこの往復の約束事であり、メソッド（GET/POST/PATCH/DELETE）、URL、ヘッダ、ボディ、ステータスコードで意味を表現します。FastAPI はこの HTTP 契約を Python コードで宣言し、TypeScript 側はその契約を型として扱うことで、実装と仕様のズレを早期に減らせます。ここが曖昧だと後続の非同期処理・SSE・認証・キャッシュすべてが混乱するため、まず通信レイヤーの語彙を固定することが最重要です。

Python/TS 
GET /api/lessons HTTP/1.1
Host: 127.0.0.1:8765
Accept: application/json

ステータスコードとエラーハンドリング

2xx は成功、4xx はクライアント入力起因、5xx はサーバー内部起因、という大枠を運用で徹底するとデバッグ効率が上がります。例えば 422 は入力バリデーション不一致、404 は対象なし、409 は競合、500 は想定外障害です。フロントは `ok` だけ見るのではなく、ステータスとエラーボディの構造を合わせて扱うべきです。FastAPI では HTTPException を使って失敗を明示し、TypeScript 側で `ApiError` のような統一型へ落とし込むと、画面側の分岐が安定します。

Python/TS 
if (!response.ok) {
  throw new Error(`HTTP ${response.status}`)
}

JSON契約とバージョニングの意識

JSON は軽量ですが、自由度が高すぎるため契約管理が必要です。フィールド名、型、nullable、必須/任意を固定し、将来変更時は後方互換を意識します。安易な breaking change はフロントを即座に壊します。理想は OpenAPI を単一の真実源にし、FastAPI 側の Pydantic モデルと TypeScript 型生成を同期させることです。仕様を自然言語だけで管理すると、実装速度が上がるほど破綻速度も上がります。

Python/TS 
{
  "id": "lesson-1",
  "title": "HTTP Basics",
  "quiz_count": 2
}
02

TypeScript 概要：型で壊れ方を先に制御する

▼

💡 any回避・ユニオン・型狭窄・APIレスポンス型の設計

TypeScriptを使う理由

TypeScript は JavaScript の上位互換で、静的型検査を追加する言語です。実行時の挙動は JS ですが、ビルド時に型不一致を検出できます。特に API 駆動開発では「サーバーが返す shape」と「画面が期待する shape」の不一致が主なバグ源なので、型で先に落とす効果が大きいです。`any` を多用すると TypeScript の価値は消えるため、unknown からの型狭窄や明示的な型定義を基本とします。

Python/TS 
type LessonMeta = {
  id: string
  title: string
  order: number
}

ユニオン型と型狭窄

実務データは単一型でないことが多く、ユニオン型（A | B）で扱います。分岐時に `in`、`typeof`、ユーザー定義 type guard を使い、処理可能な型へ狭窄します。これにより「存在しないプロパティアクセス」をコンパイル時に排除できます。レスポンス成功/失敗を discriminated union で定義すると、UI分岐を安全に書けます。

Python/TS 
type Result =
  | { ok: true; data: LessonMeta[] }
  | { ok: false; message: string }

strictモード運用

tsconfig の `strict: true` を基準にし、必要な箇所だけ局所的に緩和する方針が保守上有利です。`noImplicitAny`、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes` なども段階導入すると品質が安定します。型エラーは面倒ではなく「将来の事故前払い」です。FastAPI 側変更時にフロントが即エラーになる状態は、むしろ望ましい監視機構です。

Python/TS 
{
  "compilerOptions": {
    "strict": true,
    "noUnusedLocals": true
  }
}
03

FastAPI 概要：ルーティング・バリデーション・依存性注入

▼

💡 最小構成から実務構成へ進むための基礎

ルート関数と自動ドキュメント

FastAPI は Python の型ヒントを活用して OpenAPI ドキュメントを自動生成します。`@app.get` などでエンドポイントを宣言し、関数シグネチャにクエリ・パス・ボディを記述するだけで、/docs で実行可能な仕様書が得られます。これはバックエンド都合ではなく、フロント連携と QA 自動化に直接効く資産です。まずは「仕様がコードから再現できる」状態を作ることが重要です。

Python/TS 
from fastapi import FastAPI

app = FastAPI()

@app.get('/api/health')
def health() -> dict:
    return {'status': 'ok'}

Pydanticモデルで入力を閉じる

リクエストボディは Pydantic モデルで受けることで、型・範囲・必須性を統一的に検証できます。`extra='forbid'` を使えば未定義フィールドを拒否でき、契約逸脱を防げます。入力検証をルート内で手書きすると重複と漏れが増えるため、モデルに寄せるのが合理的です。さらに response_model を指定すると、出力形状まで拘束でき、誤って内部情報を返す事故を減らせます。

Python/TS 
class QuizAnswerIn(BaseModel):
    model_config = ConfigDict(extra='forbid')
    lesson_id: str
    choice_index: int = Field(..., ge=0)

依存性注入（Depends）の基本

DB セッション、認証情報、設定値などは `Depends` に切り出すと、ルート関数を薄く保てます。依存関数を差し替えるだけでテストが書きやすくなり、関心分離が進みます。重要なのは「依存を増やすこと」ではなく「責務を薄く保つこと」です。最初から大型DI設計に走らず、まず DB セッションと current_user 程度の抽出から始めるのが安全です。

Python/TS 
@router.get('/items')
def list_items(session: Session = Depends(get_session)):
    return session.scalars(select(Item)).all()

💡 fetchの基本からメモリリーク回避まで

useEffectでのデータ取得の基本形

React では描画と副作用を分離するため、通信は通常 useEffect に置きます。依存配列が変わるたびに effect は再実行されるため、「どの値の変化で再取得するか」を明示する設計が必要です。lessonId のようなキーに対して再取得し、不要時は state をクリアする、という基本パターンを固定すると、画面遷移時の不整合を減らせます。

Python/TS 
useEffect(() => {
  if (!lessonId) return
  // fetch...
}, [lessonId])

AbortControllerで中断可能にする

非同期リクエストは、画面遷移や再クリックで古い通信が残ると状態競合を起こします。AbortController を毎回生成し cleanup で abort することで、アンマウント後の setState や古いレスポンス反映を抑制できます。これは体感性能よりも整合性の問題です。特に SSE や長時間推論では中断制御がないと、GPU/CPU 使用が無駄に残り続けます。

Python/TS 
const ac = new AbortController()
fetch(url, { signal: ac.signal })
return () => ac.abort()

表示状態の分割

最低限、`loading` / `error` / `data` を分けて管理すると UI の説明責務が明確になります。単一 state に詰め込むより、状態遷移を列挙しやすく、障害時メッセージも制御しやすいです。より複雑になったら reducer や Query ライブラリに移行しますが、まずは素朴な3状態管理を確実に実装する方が学習効果は高いです。

Python/TS 
type ViewState = {
  loading: boolean
  error: string | null
  data: T | null
}

💡 event loop・非ブロッキング・ストリーミング設計

event loopと非ブロッキングの考え方

asyncio は event loop が多数タスクを協調的に切り替えるモデルです。`await` で制御を返すことで他タスクが進みます。逆に time.sleep や重い同期処理を挟むとループ全体が停止し、同時接続性能が落ちます。まず「どの処理がブロッキングか」を見分ける習慣を持つことが、SSE以前の必須基礎です。

Python/TS 
async def handler():
    await asyncio.sleep(0)
    return {'ok': True}

SSEの最小構成

SSE は `text/event-stream` で逐次テキストを送る方式です。各イベントを `data: ... ` で区切る必要があります。本アプリの `/api/llm/stream` はこの形式でトークンを流しています。WebSocket ほど双方向でなくてもよい場面では、実装負荷が低く運用しやすい利点があります。

Python/TS 
yield f"data: {json.dumps({'token': tok}, ensure_ascii=False)}

"

中断とバックプレッシャ

ストリーミングは開始より終了制御が難所です。クライアント切断時に生成タスクを止めないと、GPU 推論が裏で走り続けます。さらに受信側が遅いと送信バッファが詰まり、待機が連鎖します。実務では「同時ストリーム数上限」「1ストリームあたり最大トークン」「アイドルタイムアウト」をまず導入し、観測指標（TTFB/切断率/トークン速度）で調整します。

Python/TS 
return StreamingResponse(event_iter(), media_type='text/event-stream')
