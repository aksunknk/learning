API骨格 — FastAPI + SQLAlchemyでモデルを定義する

▼

💡 モデル（DBの形）を最初に固める — APIはモデルの写像にすぎない

Python — backend/app/models.py
from sqlalchemy import ForeignKey, create_engine
from sqlalchemy.orm import (DeclarativeBase, Mapped, mapped_column,
                            relationship, sessionmaker)

DATABASE_URL = "sqlite:///./app.db"
engine = create_engine(DATABASE_URL, connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(bind=engine)

class Base(DeclarativeBase): pass

class User(Base):
    __tablename__ = "users"
    id:              Mapped[int] = mapped_column(primary_key=True)
    email:           Mapped[str] = mapped_column(unique=True, index=True)
    hashed_password: Mapped[str]                 # 平文は絶対に保存しない
    todos: Mapped[list["Todo"]] = relationship(back_populates="owner")

class Todo(Base):
    __tablename__ = "todos"
    id:       Mapped[int]  = mapped_column(primary_key=True)
    title:    Mapped[str]
    done:     Mapped[bool] = mapped_column(default=False)
    owner_id: Mapped[int]  = mapped_column(ForeignKey("users.id"), index=True)
    owner: Mapped[User] = relationship(back_populates="todos")

Base.metadata.create_all(engine)   # テーブルを作成
Python — backend/app/main.py（起動確認）
from fastapi import FastAPI

@app.get("/health")

CRUD実装 — PydanticスキーマとDependsによる責務分離

▼

💡 「DBモデル」と「APIスキーマ」を分ける — hashed_password をレスポンスに漏らさないための構造

Python — backend/app/routers/todos.py
from fastapi import APIRouter, Depends, HTTPException
from pydantic import BaseModel
from sqlalchemy.orm import Session
from ..models import SessionLocal, Todo, User
from ..auth import get_current_user

router = APIRouter(prefix="/todos", tags=["todos"])

# --- Pydantic スキーマ（APIの入出力契約） ---
class TodoCreate(BaseModel):        # 入力: クライアントが送ってよいもの
    title: str

class TodoOut(BaseModel):           # 出力: クライアントに見せてよいもの
    id: int
    title: str
    done: bool
    model_config = {"from_attributes": True}

# --- DBセッションの依存性注入 ---
def get_db():
    db = SessionLocal()
    try:
        yield db                    # ルート関数にセッションを渡す
    finally:
        db.close()                  # レスポンス後に必ず閉じる

# --- CRUD ---
@router.get("", response_model=list[TodoOut])
def list_todos(db: Session = Depends(get_db),
               user: User = Depends(get_current_user)):
    return db.query(Todo).filter(Todo.owner_id == user.id).all()

@router.post("", response_model=TodoOut, status_code=201)
def create_todo(body: TodoCreate, db: Session = Depends(get_db),
                user: User = Depends(get_current_user)):
    todo = Todo(title=body.title, owner_id=user.id)
    db.add(todo); db.commit(); db.refresh(todo)
    return todo

@router.patch("/{todo_id}", response_model=TodoOut)
def toggle_todo(todo_id: int, db: Session = Depends(get_db),
                user: User = Depends(get_current_user)):
    todo = db.get(Todo, todo_id)
    if not todo or todo.owner_id != user.id:      # 他人のTODOは404
        raise HTTPException(status_code=404)
    todo.done = not todo.done
    db.commit(); db.refresh(todo)
    return todo

🔍 他人のリソースには 403 ではなく 404 を返す

「あなたのものではない」(403) と返すと、そのIDのリソースが存在すること自体が漏れます。存在の有無を隠すため、認可エラーもまとめて404にするのがセキュリティの定石です（GitHubも他人のプライベートリポジトリには404を返します）。

💼 Tips：実務で遭遇する現場の勘所

response_model は「出力のホワイトリスト」: ORMオブジェクトをそのまま返しても、response_model に無いフィールドは自動で除外されます。「うっかり hashed_password が JSON に含まれていた」という事故を構造的に防げるのがこの設計の価値です。

認証 — パスワードハッシュとJWT

▼

💡 パスワードは「復元できない形」で保存する — bcryptハッシュは同じ入力でも毎回違う値になる（ソルト内蔵）

Python — backend/app/auth.py
import os, jwt
from datetime import datetime, timedelta, timezone
from fastapi import APIRouter, Depends, HTTPException
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm
from passlib.context import CryptContext
from pydantic import BaseModel, EmailStr
from sqlalchemy.orm import Session

SECRET_KEY = os.environ["SECRET_KEY"]     # .env から注入。コードに直書きしない
ALGORITHM = "HS256"

pwd_context = CryptContext(schemes=["bcrypt"])
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/login")
router = APIRouter(prefix="/auth", tags=["auth"])

class RegisterIn(BaseModel):
    email: EmailStr
    password: str

@router.post("/register", status_code=201)
def register(body: RegisterIn, db: Session = Depends(get_db)):
    if db.query(User).filter(User.email == body.email).first():
        raise HTTPException(409, "email already registered")
    user = User(email=body.email,
                hashed_password=pwd_context.hash(body.password))  # ハッシュ化
    db.add(user); db.commit()
    return {"id": user.id}

@router.post("/login")
def login(form: OAuth2PasswordRequestForm = Depends(),
          db: Session = Depends(get_db)):
    user = db.query(User).filter(User.email == form.username).first()
    if not user or not pwd_context.verify(form.password, user.hashed_password):
        raise HTTPException(401, "invalid credentials")   # どちらが違うかは言わない
    payload = {"sub": str(user.id),
               "exp": datetime.now(timezone.utc) + timedelta(hours=1)}
    token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
    return {"access_token": token, "token_type": "bearer"}

def get_current_user(token: str = Depends(oauth2_scheme),
                     db: Session = Depends(get_db)) -> User:
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except jwt.PyJWTError:
        raise HTTPException(401, "invalid token")
    user = db.get(User, int(payload["sub"]))
    if not user:
        raise HTTPException(401, "user not found")
    return user

🔍 JWTの仕組みと注意点

JWTは「サーバーの署名付きの身分証」です。中身（payload）はBase64エンコードされているだけで誰でも読めます。秘密情報を入れてはいけません。署名により改ざんは検知できますが、有効期限が切れるまで無効化できないのが弱点です。だから exp（有効期限）は短め（1時間程度）に設定します。

💼 Tips：実務で遭遇する現場の勘所

ログイン失敗時に「メールが存在しません」と返さない: 攻撃者に登録済みメールアドレスの一覧を推測させる材料になります（ユーザー列挙攻撃）。「メールまたはパスワードが違います」に統一するのが定石です。

セキュリティ要点 — OWASPの視点でアプリを点検する

▼

💡 セキュリティは機能ではなく「全機能に貫かれる性質」— リリース前チェックリストを持て

このアプリに関係するOWASP Top 10の項目

① インジェクション（SQLi）: SQLAlchemyのクエリビルダを使う限り安全。text() に文字列連結でユーザー入力を渡した瞬間に脆弱化する。
② 認証の不備: パスワードはbcrypt、トークンは短命JWT、ログイン失敗メッセージは曖昧に（前レッスンで対応済み）。
③ アクセス制御の不備: 「自分のTODOだけ操作できる」チェックをすべてのエンドポイントで行う。1箇所でも漏れると全ユーザーのデータが読める。
④ XSS: Reactは値の埋め込みを自動エスケープする。dangerouslySetInnerHTML を使わない限り基本的に安全。
⑤ 設定ミス: SECRET_KEYのコミット、デバッグモードの本番運用、CORSの全開放（allow_origins=["*"]）が三大事故。

Python — CORSの正しい設定
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",          # 開発時のVite
        "https://your-app.example.com",   # 本番フロント
    ],                                     # "*" は絶対NG（認証付きAPIでは特に）
    allow_credentials=True,
    allow_methods=["GET", "POST", "PATCH", "DELETE"],
    allow_headers=["Authorization", "Content-Type"],
)
チェックリスト — リリース前の10項目
□ .env が .gitignore に入っている（git log にも秘密が無い）
□ SECRET_KEY は環境変数から注入している
□ パスワードは bcrypt でハッシュ化している
□ すべてのSQL/ORMクエリでユーザー入力を連結していない
□ 全エンドポイントで「所有者チェック」をしている
□ CORS の allow_origins を列挙している（* でない）
□ JWTの有効期限が適切（数時間以内）
□ エラーレスポンスに内部情報（スタックトレース）が出ない
□ 依存パッケージの脆弱性を確認した（pip-audit / npm audit）
□ HTTPSで配信される（デプロイ先の設定確認）

💼 Tips：実務で遭遇する現場の勘所

生成AIの出力コードはセキュリティ観点で必ずレビュー: AIは「動くコード」を出しますが「安全なコード」とは限りません。特にCORS全開放・文字列連結SQL・平文パスワード保存は、AI出力に高頻度で混入します。上のチェックリストをAI出力コードのレビューにもそのまま使ってください。

フロントエンド接続 — React + TypeScriptでAPIを叩く

▼

💡 API通信は「型付きの関数」に閉じ込める — コンポーネントの中に fetch を散らばらせない

TypeScript — frontend/src/api.ts（API層）
const BASE = import.meta.env.VITE_API_URL ?? "http://localhost:8000";

export interface Todo {
  id: number;
  title: string;
  done: boolean;
}

function authHeaders(): HeadersInit {
  const token = localStorage.getItem("token");
  return token ? { Authorization: `Bearer ${token}` } : {};
}

async function request<T>(path: string, init?: RequestInit): Promise<T> {
  const res = await fetch(`${BASE}${path}`, {
    ...init,
    headers: { "Content-Type": "application/json",
               ...authHeaders(), ...init?.headers },
  });
  if (res.status === 401) {              // トークン切れ → ログインへ
    localStorage.removeItem("token");
    location.href = "/login";
  }
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json() as Promise<T>;
}

export const api = {
  listTodos: ()            => request<Todo[]>("/todos"),
  createTodo: (title: string) =>
    request<Todo>("/todos", { method: "POST",
                              body: JSON.stringify({ title }) }),
  toggleTodo: (id: number) =>
    request<Todo>(`/todos/${id}`, { method: "PATCH" }),
};
TypeScript — frontend/src/TodoList.tsx
import { useEffect, useState } from "react";
import { api, type Todo } from "./api";

export function TodoList() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [title, setTitle] = useState("");
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    api.listTodos().then(setTodos).catch(e => setError(e.message));
  }, []);

  async function add() {
    if (!title.trim()) return;
    const todo = await api.createTodo(title);
    setTodos(prev => [...prev, todo]);     // 新しい配列を作って更新
    setTitle("");
  }

  async function toggle(id: number) {
    const updated = await api.toggleTodo(id);
    setTodos(prev => prev.map(t => (t.id === id ? updated : t)));
  }

  if (error) return <p role="alert">読み込みに失敗しました: {error}</p>;

  return (
    <div>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <button onClick={add}>追加</button>
      <ul>
        {todos.map(t => (
          <li key={t.id} onClick={() => toggle(t.id)}>
            {t.done ? "✅" : "⬜"} {t.title}
          </li>
        ))}
      </ul>
    </div>
  );
}

💼 Tips：実務で遭遇する現場の勘所

💡 テストは「未来の自分への保険」— リファクタリングできるコードとできないコードの分かれ目

Python — backend/tests/test_todos.py
import pytest
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

@pytest.fixture
def auth_header():
    """テスト用ユーザーを登録してトークンを返す fixture"""
    client.post("/auth/register",
                json={"email": "test@example.com", "password": "secret123"})
    res = client.post("/auth/login",
                      data={"username": "test@example.com",
                            "password": "secret123"})
    token = res.json()["access_token"]
    return {"Authorization": f"Bearer {token}"}

def test_create_and_list_todo(auth_header):
    res = client.post("/todos", json={"title": "牛乳を買う"},
                      headers=auth_header)
    assert res.status_code == 201
    assert res.json()["title"] == "牛乳を買う"

    res = client.get("/todos", headers=auth_header)
    assert len(res.json()) == 1

def test_todos_require_auth():
    """トークン無しは401 — セキュリティの回帰を防ぐテスト"""
    assert client.get("/todos").status_code == 401

def test_cannot_touch_others_todo(auth_header):
    """他人のTODOは404 — アクセス制御の回帰を防ぐテスト"""
    res = client.patch("/todos/99999", headers=auth_header)
    assert res.status_code == 404
TypeScript — frontend/src/api.test.ts（Vitest）
import { describe, it, expect, vi, beforeEach } from "vitest";
import { api } from "./api";

beforeEach(() => {
  vi.stubGlobal("fetch", vi.fn());   // 本物の通信はしない

describe("api.listTodos", () => {

💼 Tips：実務で遭遇する現場の勘所

何をテストすべきか迷ったら「壊れたら困る順」: ①認証・認可（事故ると即インシデント）②お金・データを扱うロジック ③バグ修正した箇所（回帰防止）。UIの見た目のテストは費用対効果が低いので後回しで構いません。カバレッジ100%より「重要な20%を確実に」が実務の解です。

        condition: service_healthy      # backendが健康になってから起動

volumes:
  db-data:

# 起動:   docker compose up --build
# 停止:   docker compose down
# ログ:   docker compose logs -f backend

💼 Tips：実務で遭遇する現場の勘所

COPY の順序がビルド時間を支配する: Dockerはレイヤー単位でキャッシュします。requirements.txt を先にCOPYしておけば、コードだけ変更した再ビルドでは pip install（数分）がスキップされ数秒で終わります。この「変更頻度が低いものを上に」がDockerfile設計の第一原則です。

CI/CDとデプロイ — GitHub Actionsから無料枠公開まで

▼

💡 CI = 「壊れたコードがmainに入るのを機械に見張らせる」— 人間の注意力に頼らない

YAML — .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  backend-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r backend/requirements.txt
      - run: pytest backend/tests
        env:
          SECRET_KEY: test-only-secret     # テスト用のダミー値

  frontend-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
      - run: npm ci
        working-directory: frontend
      - run: npx vitest run
        working-directory: frontend

🔍 無料枠デプロイの選択肢（2026年時点）

フロントエンド: GitHub Pages / Cloudflare Pages（静的ビルドを配信、完全無料）
バックエンド: Render（無料枠あり・スリープあり）、Fly.io（小規模無料）。Dockerイメージをそのままデプロイできるため、レッスン09の成果物が直接使えます。
手順の骨子: ①デプロイ先にリポジトリを接続 ②環境変数（SECRET_KEY等）を管理画面で設定 ③mainへのマージで自動デプロイ。秘密情報は必ずデプロイ先の環境変数機能で注入し、リポジトリには置きません。

🎓 完走おめでとうございます — 次にやるべきこと

このアプリには伸びしろが大量にあります。①リフレッシュトークンの実装 ②SQLiteからPostgreSQLへの移行（Alembicでマイグレーション） ③E2Eテスト（Playwright） ④ログと監視（structlog + Sentry）。1つ選んで自力で拡張し、その過程をPRとして自分のリポジトリに積んでください。その履歴こそが、あなたのポートフォリオになります。

💼 Tips：実務で遭遇する現場の勘所

「デプロイして終わり」ではなく「デプロイしてから始まる」: 本番で初めて踏む問題（環境変数の設定漏れ、CORSエラー、コールドスタート）こそが最高の教材です。エラーメッセージを恐れず、ログ → 原因の仮説 → 検証のループを回してください。この診断能力がフルスタックエンジニアの本体です。
