🧭 DevTools

ブラウザの開発者ツールで、表示・通信・状態・性能を切り分けます。TaskBoard の不具合を 15 分で再現・仮説・検証する手順を身につけます。

🟢 初級編 — 五つのパネルを地図にする

Elements / Console / Network / Sources / Application の役割分担。

01

DevTools の地図 — どのパネルで何を見るか

▼

💡 ツールを開く前に『疑い』を選ぶ

見た目なら Elements、例外なら Console、通信なら Network、状態なら Application、処理の流れなら Sources。当てずっぽうに全パネルを彷徨わない。

Text 
症状 → 最初のパネル

UI が崩れた / スタイルが当たらない     → Elements
赤い例外・ログが出る                   → Console
画面は出るがデータが空 / 遅い          → Network
クリックしても論理がおかしい           → Sources（ブレーク）
ログインが消える / キャッシュ怪しい     → Application
重い・カクつく                         → Performance / Lighthouse

ショートカット（Chromium 系）:
  Windows: F12 または Ctrl+Shift+I
  要素を選択: Ctrl+Shift+C
  ネットワークを空にして再読込: 確認しつつ Disable cache

💼 Tips：実務で遭遇する現場の勘所

Disable cache は DevTools を開いている間だけ有効、が基本です。『自分だけ直った』の多くはキャッシュです。

02

Console — ログ・レベル・オブジェクト検査

▼

💡 console.log はセンサー、例外は警報

ログレベル（log/info/warn/error）を使い分け、オブジェクトは展開して形を確認します。フィルタでノイズを消す。

JavaScript 
const task = { id: "1", title: "ship", done: false };
console.log("task", task);
console.table([task]);
console.assert(task.title.length > 0, "title required");

// 例外を意図的に見る
try {
  JSON.parse("{");
} catch (e) {
  console.error("parse failed", e);
}

手元で確認

TaskBoard の追加ボタン直後に tasks 配列を log し、done の型が boolean か確認してください。文字列 "false" 混入は古典的バグです。

💼 Tips：実務で遭遇する現場の勘所

本番に debug ログを残しすぎない。必要なら構造化ログをサーバー側へ。Console は開発中の顕微鏡です。

03

Elements — DOM と CSS のライブ編集

▼

💡 見た目のバグは『当たっている宣言』を見つけるゲーム

Computed で最終値を見し、打ち消し線の付いたルールから負けた理由を読みます。

Text 
手順:
1. Ctrl+Shift+C で要素を選択
2. Styles で当たりそうなセレクタを確認
3. Computed で margin/display/color の最終値
4. ボックスモデル図で padding と content を確認
5. 一時的に CSS を編集 → 効いたらソースへ反映

よくある原因:
  - 詳細度負け（打ち消し線）
  - display:none の親
  - Flex 子の min-width:auto によるはみ出し
  - 間違った要素を編集している

💼 Tips：実務で遭遇する現場の勘所

ライブ編集で直っても、リロードで戻ります。必ずソース（または DevTools Changes）に戻してコミットしてください。

04

Network — ステータス・ヘッダ・Timing

▼

💡 フロントのバグの半分は通信の物語

status、Response、Timing（TTFB）を読めば、API 落ち・CORS・遅い DB の切り分けが速くなります。

Text 
見る順:
1. 失敗行を赤で探す（4xx/5xx/failed）
2. Headers: Request URL / Method / Status
3. Response: JSON の形と detail
4. Timing: Queueing / TTFB / Content Download

TaskBoard 例:
  GET /api/tasks  → 401  … トークン忘れ
  GET /api/tasks  → 200 だが []  … 認可は成功、データ無し or フィルタ過多
  OPTIONS … 失敗     … CORS プリフライト

Preserve log をオン:
  リダイレクトやページ遷移でログが消えない

💼 Tips：実務で遭遇する現場の勘所

『画面が白い』ときは Console より先に Network で index.html と JS バンドルが 200 かを見ると時間を節約できます。

05

Sources — ブレークポイントと Call Stack

▼

💡 推測で直すのをやめ、止めて見る

行ブレーク、条件付きブレーク、Call Stack / Scope で『なぜここに来たか』を確認します。

Text 
手順:
1. Sources で該当 JS を開く（or Ctrl+P でファイル名）
2. 行番号をクリックしてブレークポイント
3. 操作を再現 → 停止
4. Scope で変数の中身
5. Call Stack で呼び出し元を遡る

条件付き:
  右クリック → Edit breakpoint
  title === ""  のときだけ止まる

debugger; 文:
  コードに直接書いて一時停止（忘れずに削除）

💼 Tips：実務で遭遇する現場の勘所

圧縮されたバンドルは Source Map を有効に。無いときは『読みにくい本番』と割り切り、ローカル開発ビルドで追う。

06

Application — Storage / Cookie / Cache

▼

💡 状態がおかしいときは永続層を疑う

localStorage、Cookie、Cache Storage は TaskBoard 縦糸のど真ん中です。

Text 
Application パネル:
  Local Storage   → tasks JSON の形を目視
  Session Storage → タブを閉じると消える値
  Cookies         → Secure / HttpOnly / SameSite
  Cache Storage   → Service Worker 配下の古い資産

TaskBoard ドリル:
1. タスクを 2 つ追加
2. Application でキーを開く
3. 値を手で壊す（JSON を不正に）→ リロード
4. アプリが例外で死ぬならガードを足す課題になる

Clear site data:
  怪しい再現の初期化に使う（本番ユーザーには使わない）

💼 Tips：実務で遭遇する現場の勘所

Cookie の HttpOnly は JS から読めません。読めないのはバグではなくセキュリティ機能です。

07

Performance と Lighthouse — 数値の読み方

▼

💡 体感をメートル法にする

Lighthouse は監査の入口。Performance パネルは長いタスクとレイアウトを可視化します。点数より『何が主犯か』。

Text 
Lighthouse（初見で見る指標）:
  LCP  最大コンテンツの表示
  CLS  レイアウトシフト
  INP  操作への反応（旧 FID の進化）

Performance 録音の見方:
  Main スレッドの長い黄色/紫ブロック
  → 長い JS タスクを分割 or 遅延ロード

TaskBoard でやりがちな悪化:
  - 毎キー入力で localStorage へ全件 stringify
  - 巨大リストを仮想化せず全部 DOM 化

💼 Tips：実務で遭遇する現場の勘所

点数 100 を目指すより、ユーザー導線の LCP を守る方がプロductive です。測定はシークレットウィンドウで。

08

デバイスモード — レスポンシブとタッチ

▼

💡 狭い画面は別プロダクトだと思え

幅を変えるだけでなく、タッチターゲットとビューポート、ネットワーク制限を再現します。

Text 
Device Toolbar:
  - iPhone / Pixel プリセット
  - Responsive で幅をドラッグ
  - Throttling: Fast 3G で体感遅延

確認項目（TaskBoard）:
  [ ] 追加フォームが折り返しても押せる
  [ ] タップ領域が 44px 相当ある
  [ ] 横スクロールが出ない
  [ ] キーボード表示でボタンが隠れない（モバイル実機推奨）

💼 Tips：実務で遭遇する現場の勘所

エミュレーションは近似です。致命的 UI は実機 Safari/Chrome で一度は触ってください。

09

CORS と失敗リクエストの切り分け

▼

💡 赤い Network 行のメッセージをそのまま信じる

CORS はブラウザの防衛。サーバーログでは 200 なのにフロントだけ失敗、があり得ます。

Text 
切り分けフロー:
1. Network で失敗リクエストを選択
2. ステータスが (failed) / CORS error か
3. Console の CORS メッセージを読む
4. Request Headers の Origin を確認
5. サーバーの Access-Control-Allow-Origin を確認
6. Cookie 認証なら Allow-Credentials と許容 Origin の組

TaskBoard:
  開発中 localhost:5173 → API :8000 は別オリジン
  → 開発用 CORS 許可を明示（* + credentials は不可）

💼 Tips：実務で遭遇する現場の勘所

セキュリティ章の CORS レッスンとセットで読むと定着します。『とりあえず *』は Cookie 認証と両立しません。

10

メモリの入口 — リークを疑うとき

▼

💡 放置したリスナーがタブを重くする

Heap snapshot の詳細より先に、画面遷移後も増えるリスナーと巨大配列を疑います。

Text 
兆候:
  - 長時間開いたタブだけ重い
  - 画面遷移のたびに遅くなり、リロードで直る

初手:
1. Performance monitor で JS heap を眺める
2. 疑わしいコンポーネントの useEffect クリーンアップ
3. addEventListener に対する remove があるか

TaskBoard:
  setInterval で自動保存するなら clearInterval を忘れない
  グローバル配列にイベント履歴を push し続けない

💼 Tips：実務で遭遇する現場の勘所

ヒープの深い比較は上級者向け。まずは『消えるべきものが消えているか』をコードレビューで潰す方が費用対効果が高いです。

11

React DevTools — Components と Profiler

▼

💡 コンポーネント木を外科手術する

Props/State をその場で確認し、Profiler で無駄レンダーを見つけます。React 章との接続点です。

Text 
Components:
  - 木から TaskList を選択
  - props.tasks の中身を確認
  - ハイライト更新（Highlight updates）で再レンダー可視化

Profiler:
  1. Record → 操作 → Stop
  2. 長いレンダーのコンポーネントを特定
  3. 原因候補: 毎レンダー新規オブジェクト / 広い context

TaskBoard:
  入力 1 文字ごとにリスト全体が再レンダーされていないか見る

💼 Tips：実務で遭遇する現場の勘所

最適化は計測後。memo を儀式的に巻く前に、state の位置を上げすぎていないかを疑ってください。

12

15 分ランブック — TaskBoard 障害の切り分け

▼

💡 手順書があると焦らない

再現 → 仮説 → パネル選択 → 証拠 → 修正 or エスカレーション。時間を区切る。

Text 
TaskBoard 障害ランブック（15 分）

0:00-2:00 再現
  手順を 3 行で書く。シークレットウィンドウでも再現するか。

2:00-5:00 仮説を 2 つまで
  A: API が 4xx/5xx
  B: フロントの状態/ローカルストレージ破壊

5:00-10:00 証拠
  Network で該当 XHR の status/body
  Console の例外スタック
  Application の tasks 値

10:00-15:00 分岐
  API 悪い → サーバーログ / テスト追加
  フロント悪い → Sources で停止、最小修正
  不明 → 再現手順とスクショを Issue に残して休憩

終わったら:
  [ ] 回帰テスト or チェックリストを 1 行追加
  [ ] Disable cache をオフに戻した

💼 Tips：実務で遭遇する現場の勘所

切り分け力は知識量より『記録する習慣』で伸びます。仮説と証拠を残す人が、チームのデバッガになります。
