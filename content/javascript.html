🟨 JavaScript 基礎

TypeScript / React の前提となる素の JavaScript を、初級（文法）・中級（クロージャ・DOM・非同期）・上級（イベントループ・モジュール）の 3 段階で学びます。

🟢 初級編 — 文法とデータの扱いを体に染み込ませる

変数・型・制御構文・関数・配列という、あらゆる JS プログラムの土台を学びます。TypeScript や React に進む前に、素の JavaScript で「動くコード」を書ける状態を作ります。

01

JavaScript とは何か — ブラウザと Node の二面性

▼

💡 Web の「唯一の標準言語」としての立ち位置

なぜ今でも JavaScript を学ぶ必要があるのか？ ブラウザがネイティブに実行できる言語は、事実上 JavaScript だけです。TypeScript も React も最終的には JS に変換されて動きます。つまり TS / React のエラーメッセージ・実行時挙動・デバッグのすべてが「素の JS の理解」に依存します。土台を飛ばしてフレームワークに入ると、「なぜこう動くのか」が永遠にブラックボックスのままになります。

2 つの実行環境：ブラウザと Node.js

同じ言語でも API が違います。ブラウザには document や window があり、Node.js には fs や process があります。共通部分（ECMAScript 本体）と環境固有 API を区別できると、「このコードはどこで動くか」を設計段階で判断できます。

HTML 
<!-- ブラウザ：script タグで読み込む -->
<script src="app.js"></script>
<script>
  // インラインでも書けるが、実務では外部ファイルが基本
  console.log("Hello from browser");
</script>

開発の入り口：console と開発者ツール

学習中の最強の味方はブラウザの DevTools（F12）です。console.log で値を確認し、Sources タブでブレークポイントを張り、Network タブで通信を見る——この 3 つが使えるだけで学習速度が桁違いに上がります。

JavaScript 
// Node.js：ターミナルで node app.js と実行
console.log("Hello from Node");
console.log(typeof document); // → "undefined"（ブラウザ専用）

// ブラウザの Console では document が使える
// console.log(document.title);

💼 Tips：strict mode はデフォルトだと思え

モダンな JS（モジュール・バンドラ経由）は実質 strict mode です。未宣言変数への代入はエラーになり、this の挙動も変わります。「ゆるい昔の JS」を前提にした古い記事には注意してください。

02

変数とデータ型 — let / const とプリミティブ

▼

💡 「再代入できるか」で宣言を選ぶ

var は使わない。 現代の JavaScript では const を既定にし、再代入が必要なときだけ let を使います。var は関数スコープと巻き上げ（hoisting）という罠が多く、新規コードで選ぶ理由はありません。型は動的ですが、「どんな値が入っているか」を常に意識するのがバグを減らす鍵です。

const がデフォルト、let は例外

const は「再代入不可」であり、「中身が不変」ではありません。オブジェクトや配列のプロパティ変更は可能です。再代入しない変数を const にすると、意図しない上書きをコンパイル相当（実行時）で防げます。

JavaScript 
const name = "佐藤";
let score = 80;
score = 90;              // OK：let は再代入できる
// name = "田中";        // TypeError：const は再代入不可

const tags = ["js", "web"];
tags.push("node");       // OK：配列の中身の変更は許可される
// tags = [];            // TypeError：変数そのものの付け替えは不可

プリミティブ 7 種と typeof

number / string / boolean / null / undefined / bigint / symbol。注意点は typeof null === "object"（歴史的バグ）と、typeof [] === "object"（配列判定には Array.isArray）です。

JavaScript 
console.log(typeof 42);          // "number"
console.log(typeof "hi");        // "string"
console.log(typeof true);        // "boolean"
console.log(typeof undefined);   // "undefined"
console.log(typeof null);        // "object" ← 罠
console.log(typeof 10n);         // "bigint"
console.log(Array.isArray([1])); // true

💼 Tips：undefined と null を使い分ける

undefined は「未設定」、null は「意図的に空」。API 設計では「値が無い」を明示したいときだけ null を返し、未初期化のまま放置しないのが実務の作法です。

03

演算子と型変換 — == を捨て === を選ぶ

▼

💡 暗黙の型変換はバグの温床

JavaScript 最大の落とし穴は「ゆるい等価（==）」です。 0 == "" が true、null == undefined が true など、直感に反する結果が山ほどあります。実務では常に ===（厳密等価）を使い、型変換は Number() / String() / Boolean() で明示します。

厳密等価と Truthy / Falsy

条件式で「偽」とみなされる値（Falsy）は false / 0 / "" / null / undefined / NaN の 6 つだけ。それ以外（空配列 [] や空オブジェクト {} を含む）はすべて Truthy です。

JavaScript 
console.log(0 === "0");     // false（型が違う）
console.log(0 == "0");      // true  ← 使わない
console.log(null == undefined); // true ← これも罠

// Falsy の判定は Boolean() か !! で明示できる
console.log(Boolean(""));   // false
console.log(Boolean([]));   // true（空配列も Truthy！）

// 数値変換
console.log(Number("42"));  // 42
console.log(Number(""));    // 0
console.log(Number("abc")); // NaN

テンプレートリテラルと null 合体

文字列連結は + よりテンプレートリテラル（バッククォート）が読みやすいです。また「null / undefined のときだけデフォルト」は ??（Nullish coalescing）、短絡評価のデフォルトは || ですが、0 や "" を正当な値として残したいなら ?? 一択です。

JavaScript 
const user = "田中";
const msg = `こんにちは、${user}さん`;

const count = 0;
console.log(count || 10);  // 10 ← 0 が Falsy なので上書きされる
console.log(count ?? 10);  // 0  ← null/undefined だけ置換

💼 Tips：NaN は自分自身とも等しくない

NaN === NaN は false です。数値変換の成否判定には Number.isNaN(value) を使ってください。isNaN()（グローバル）は型変換してから判定するため、文字列でも true になり得ます。

04

条件分岐とループ — 制御の流れを書く

▼

💡 分岐と繰り返しはプログラムの骨格

if / else・switch・for / while・for...of は全言語共通の骨格です。JS では配列の走査に for...of を、オブジェクトのキー走査に Object.keys + ループ（または後述のメソッド）を使うのが現代的です。インデックスが必要なときだけ古典的な for (let i = 0; ...) を残します。

if / else と switch

条件が 2〜3 分岐なら if、文字列や数値の多岐分岐なら switch。switch は break 忘れで次の case に落ちる（fall-through）ので注意してください。

JavaScript 
const status = "loading";

if (status === "loading") {
  console.log("読み込み中");
} else if (status === "error") {
  console.log("失敗");
} else {
  console.log("完了");
}

switch (status) {
  case "loading":
    console.log("⏳");
    break;
  case "error":
    console.log("❌");
    break;
  default:
    console.log("✅");
}

for...of を第一選択にする

for...in はプロトタイプ連鎖上の列挙可能プロパティも拾うため、配列走査には向きません。配列・文字列・Map など Iterable は for...of が安全で読みやすいです。

JavaScript 
const fruits = ["apple", "banana", "cherry"];

for (const fruit of fruits) {
  console.log(fruit);
}

// インデックスも欲しければ entries()
for (const [i, fruit] of fruits.entries()) {
  console.log(i, fruit);
}

let n = 3;
while (n > 0) {
  console.log(n);
  n -= 1;
}

💼 Tips：早期 return でネストを減らす

深い if-else の代わりに、異常系を先に return するガード節を書くと可読性が上がります。関数の本線（ハッピーパス）が左端に揃うのが理想です。

05

関数 — 宣言・式・アロー関数

▼

💡 関数は「値」として扱える

JavaScript では関数もオブジェクト（第一級オブジェクト）です。 変数に入れ、引数に渡し、戻り値にできます。書き方は 3 系統——関数宣言、function 式、アロー関数——があり、現代のコールバックや配列メソッドではアロー関数が主流です。ただし this の扱いが違うため、次の中級編で詳しく扱います。

3 つの書き方

関数宣言は巻き上げられる（定義より前で呼べる）のに対し、関数式・アローは変数と同じルールです。実務ではモジュール内のトップレベルは宣言か const fn = () =>、短いコールバックはアロー、が定石です。

JavaScript 
// 1. 関数宣言
function add(a, b) {
  return a + b;
}

// 2. 関数式
const mul = function (a, b) {
  return a * b;
};

// 3. アロー関数（本体が式なら return と {} を省略可）
const double = (n) => n * 2;
const greet = (name) => {
  const msg = `Hello, ${name}`;
  return msg;
};

console.log(add(2, 3), mul(2, 3), double(5));

デフォルト引数・残余引数・即時実行

引数の省略にはデフォルト値、...args で可変長引数を配列として受け取れます。即時実行関数（IIFE）はモジュール以前のスコープ隔離手段でしたが、ES Modules 普及後は出番が激減しています。

JavaScript 
function createUser(name, role = "member") {
  return { name, role };
}

function sum(...nums) {
  let total = 0;
  for (const n of nums) total += n;
  return total;
}

console.log(createUser("佐藤"));     // { name: "佐藤", role: "member" }
console.log(sum(1, 2, 3, 4));        // 10

💼 Tips：副作用のある関数と純粋関数を意識する

同じ引数なら同じ結果を返し、外部状態を変えない関数を「純粋関数」と呼びます。テストしやすく、バグの追跡も容易です。DOM 操作や API 呼び出しは意図的な副作用として関数の境界に閉じ込めるのが設計のコツです。

06

配列の基本とメソッド — map / filter / reduce

▼

💡 ループを「宣言的」に書き換える

実務の JS コードの大半は配列変換です。 for で回して push する代わりに、map（変換）・filter（絞り込み）・find（1 件取得）・reduce（集約）を使うと、意図がメソッド名で読めるようになります。React のリスト描画もこの延長線上にあります。

不変っぽく扱う：元配列を壊さない

push / splice / sort は元配列を変更（破壊的）します。map / filter / toSorted（またはスプレッドしてから sort）は新しい配列を返します。状態管理（特に React）では破壊的変更がバグの原因になるため、非破壊を習慣にしてください。

JavaScript 
const scores = [55, 82, 91, 68];

const passed = scores.filter((s) => s >= 70);
const boosted = passed.map((s) => s + 10);
const total = boosted.reduce((acc, s) => acc + s, 0);

console.log(passed);   // [82, 91]
console.log(boosted);  // [92, 101]
console.log(total);    // 193
console.log(scores);   // 元配列は無傷

find / some / every / includes

「最初の 1 件」は find、「存在する？」は some または includes、「全部満たす？」は every。インデックスが欲しければ findIndex です。

JavaScript 
const users = [
  { id: 1, name: "田中", active: true },
  { id: 2, name: "佐藤", active: false },
];

const tanaka = users.find((u) => u.name === "田中");
const hasInactive = users.some((u) => !u.active);
const allActive = users.every((u) => u.active);

console.log(tanaka, hasInactive, allActive);

💼 Tips：チェーンは 2〜3 段まで

filter().map().reduce() までは読みやすいですが、それ以上は途中結果に名前を付けるか、1 回のループにまとめた方がデバッグしやすくなります。「宣言的であること」より「追跡可能であること」を優先してください。

🔶 中級編 — スコープ・DOM・非同期で「動く UI」を作る

クロージャ、this、DOM イベント、Promise / async-await など、ブラウザで実際に動くコードに必要な中核概念を学びます。ここを越えると React の「なぜそう書くか」が理解できるようになります。

07

オブジェクトと分割代入 — データの設計図

▼

💡 キーと値の集まりが JS のデータ中心

オブジェクトは JS で最も使うデータ構造です。 API レスポンス、設定、コンポーネントの props——ほぼすべてがオブジェクトです。分割代入（destructuring）とスプレッド構文を使いこなすと、読み取り・コピー・マージが簡潔になり、React / TypeScript への接続もスムーズになります。

作成・アクセス・省略記法

プロパティ名と変数名が同じなら省略記法が使えます。存在しないキーへのアクセスは undefined を返し、例外にはなりません（ここで Optional Chaining ?. が活躍します）。

JavaScript 
const name = "佐藤";
const user = {
  name,                    // name: name の省略
  age: 28,
  greet() {                // メソッド省略記法
    return `Hi, ${this.name}`;
  },
};

console.log(user.name);
console.log(user["age"]);
console.log(user.address?.city); // undefined（エラーにならない）

分割代入とスプレッド

分割代入で必要なプロパティだけ取り出し、スプレッドで浅いコピーやマージができます。ネストしたオブジェクトの「深いコピー」には structuredClone や専用ライブラリが必要です。

JavaScript 
const product = { id: 1, title: "本", price: 1200, stock: 3 };
const { title, price, ...rest } = product;
console.log(title, price, rest); // "本" 1200 { id: 1, stock: 3 }

const updated = { ...product, price: 980 }; // 非破壊更新
console.log(updated.price, product.price);  // 980 1200

💼 Tips：Object.keys / values / entries

オブジェクトを配列として扱いたいときはこの 3 兄弟です。特に Object.entries(obj).map(([k, v]) => ...) はフォームや設定オブジェクトの変換で頻出です。

08

スコープとクロージャ — 関数が「覚えている」環境

▼

💡 JS の関数は定義時の変数環境を閉じ込める

クロージャは「関数＋その関数が作られたときの外側スコープ」の組み合わせです。 コールバック、イベントハンドラ、モジュールパターン、React のフック——現代 JS のほぼすべてがクロージャの上に成り立っています。「なぜこの変数が見えるのか／見えないのか」を説明できれば、中級者の門をくぐったと言えます。

ブロックスコープ（let / const）

{} で囲まれた範囲がブロックです。let / const はこのブロックの外からは見えません。var は関数スコープなのでブロックを無視します（だから使わない）。

JavaScript 
function demo() {
  if (true) {
    const secret = 42;
    var leaked = 99;
  }
  // console.log(secret); // ReferenceError
  console.log(leaked);    // 99（var の罠）
}
demo();

クロージャの実例：プライベート変数とファクトリ

外側の関数が終わっても、内側の関数が外側の変数を参照し続けます。これが「状態を閉じ込めた関数」を作る仕組みです。

JavaScript 
function createCounter(start = 0) {
  let count = start; // 外からは触れない
  return {
    inc() { count += 1; return count; },
    value() { return count; },
  };
}

const c = createCounter(10);
console.log(c.inc());   // 11
console.log(c.inc());   // 12
console.log(c.value()); // 12
// console.log(count);  // 参照不可

💼 Tips：ループ内のクロージャに注意

昔は var + ループで「全部最後の値になる」バグが頻発しました。let は反復ごとに別バインディングを作るため、現代ではこの罠はほぼ消えています。それでも非同期コールバック内でループ変数を使うときは、意図した値が閉じ込まれているか一度確認してください。

09

this と実行コンテキスト — 呼び出し方で決まる

▼

💡 this は「定義場所」ではなく「呼び出し方」で決まる

他言語の this（自己参照）感覚で JS の this を理解すると必ず詰みます。 JS の this は関数がどう呼ばれたかで決まります。メソッド呼び出し、通常呼び出し、new、call/apply/bind、アロー関数——この 5 パターンを整理すれば、React クラスや DOM イベントでの謎挙動が消えます。

呼び出しパターンで this が変わる

オブジェクトのメソッドとして呼べばそのオブジェクトが this。単独で呼べば strict mode では undefined。アロー関数は自分の this を持たず、外側のスコープの this をそのまま使います（語彙的 this）。

JavaScript 
"use strict";

const user = {
  name: "佐藤",
  greet() {
    console.log(this.name);
  },
};

user.greet();              // "佐藤"（メソッド呼び出し）
const fn = user.greet;
// fn();                   // TypeError：this が undefined

// アローは外側の this を引き継ぐ（ここではモジュール/グローバル）
const obj = {
  name: "田中",
  greet: () => console.log(obj.name), // 明示参照の方が安全なことも多い
};

bind で固定する

コールバックにメソッドを渡すとき、this が剥がれる問題は bind かアローで解決します。クラスコンポーネント時代の定番でしたが、フック中心の現代でもイベントやタイマーで同じ発想が必要です。

JavaScript 
const button = {
  label: "保存",
  onClick() {
    console.log(`${this.label} が押されました`);
  },
};

// コールバックとして渡す想定
setTimeout(button.onClick.bind(button), 0);

// アローで包む同等のパターン
setTimeout(() => button.onClick(), 0);

💼 Tips：迷ったらアロー＋明示的な引数

this パズルに時間を使うより、必要な値を引数やクロージャで渡す設計の方が堅牢です。モダンコードでは「this に依存するオブジェクト指向」より「データを引数で渡す関数型寄り」が主流です。

10

DOM 操作とイベント — ブラウザを動かす

▼

💡 HTML を JS から読み書きする

DOM（Document Object Model）は、HTML を木構造のオブジェクトとして扱うブラウザ API です。 React は仮想 DOM でこの操作を抽象化しますが、「素の DOM で何が起きているか」を知らないと、イベントの伝播・フォーム制御・アクセシビリティで必ず詰みます。最低限のクエリ・作成・イベント登録を手で書いてください。

要素の取得と書き換え

querySelector / querySelectorAll が現代の標準です。innerHTML は XSS のリスクがあるため、テキストなら textContent、構造が必要なら createElement を優先します。

JavaScript 
const title = document.querySelector("#title");
title.textContent = "更新後のタイトル";

const list = document.querySelector("#todo-list");
const li = document.createElement("li");
li.textContent = "新しいタスク";
list.appendChild(li);

// 危険：ユーザー入力をそのまま入れない
// el.innerHTML = userInput;  // XSS

イベントとイベント委譲

addEventListener で登録し、不要になったら removeEventListener で外します。リストの各項目に個別リスナーを付けるより、親に 1 つ付けて event.target で判別する「イベント委譲」が効率的です。

JavaScript 
const form = document.querySelector("#signup");
form.addEventListener("submit", (event) => {
  event.preventDefault(); // ページ遷移を止める
  const data = new FormData(form);
  console.log(Object.fromEntries(data.entries()));
});

// イベント委譲：動的に増えるボタンにも対応
document.querySelector("#list").addEventListener("click", (e) => {
  if (e.target.matches("[data-delete]")) {
    e.target.closest("li").remove();
  }
});

💼 Tips：DOMContentLoaded を待つ

スクリプトが <head> にあると、HTML 解析前に実行されて要素が取れません。defer 属性を付けるか、DOMContentLoaded イベント内で初期化してください。モジュール（type="module"）はデフォルトで defer 相当です。

11

Promise と async/await — 非同期を直線的に書く

▼

💡 コールバック地獄からの脱出

ネットワーク通信やタイマーは「今すぐ終わらない」処理です。 昔はコールバックの入れ子（コールバック地獄）が常態でしたが、Promise と async/await により、非同期コードを同期コードに近い見た目で書けるようになりました。fetch・ファイル読み込み・React のデータ取得——すべてこの上に載っています。

Promise の 3 状態と then / catch

Promise は pending → fulfilled または rejected へ遷移します。then で成功、catch で失敗を扱います。チェーンすると前の戻り値が次に渡ります。

JavaScript 
function wait(ms) {
  return new Promise((resolve) => {
    setTimeout(() => resolve(`${ms}ms 経過`), ms);
  });
}

wait(300)
  .then((msg) => {
    console.log(msg);
    return wait(200);
  })
  .then((msg) => console.log(msg))
  .catch((err) => console.error(err));

async/await：then チェーンを同期風に

async 関数は常に Promise を返します。await はその解決を待ち、値を取り出します。エラーは try/catch で受けます。実務ではほぼこちらの書き方が標準です。

JavaScript 
async function loadUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}`);
    }
    const user = await res.json();
    return user;
  } catch (err) {
    console.error("取得失敗", err);
    throw err;
  }
}

// 並列実行は Promise.all
async function loadDashboard() {
  const [user, posts] = await Promise.all([
    fetch("/api/me").then((r) => r.json()),
    fetch("/api/posts").then((r) => r.json()),
  ]);
  return { user, posts };
}

💼 Tips：for ループ内の await は直列になる

配列を for...of で await すると 1 件ずつ待ちます。独立したリクエストなら Promise.all(items.map(...)) で並列化してください。逆にレート制限がある API では意図的に直列にします。

12

よくあるエラーとアンチパターン — 現場で踏む穴

▼

💡 「動いた」と「正しい」は別物

初学者が量産しがちなバグパターンを先に潰します。 変異の共有、非同期の取りこぼし、==、グローバル汚染、innerHTML へのユーザー入力——これらを認識しているだけで、レビュー指摘の大半を自己チェックできます。

参照の共有と浅いコピー

オブジェクトや配列は参照渡しです。const b = a はコピーではなく別名です。意図せず両方変わるバグの温床なので、コピーが必要ならスプレッドや structuredClone を使います。

JavaScript 
const original = { tags: ["a"] };
const alias = original;          // 同じオブジェクト
alias.tags.push("b");
console.log(original.tags);      // ["a", "b"] ← 巻き込み事故

const copy = { ...original, tags: [...original.tags] };
copy.tags.push("c");
console.log(original.tags);      // ["a", "b"] ← 無傷

async 関数のエラーを握りつぶさない

async 関数内の例外は rejected Promise になります。呼び出し側で await + try/catch、または .catch() を付けないと、未処理の Rejection として実行環境に怒られます。

JavaScript 
async function risky() {
  throw new Error("失敗");
}

// ❌ 呼び捨て：エラーが消える/警告になる
// risky();

// ✅ 必ず待つ or catch する
risky().catch((err) => console.error(err.message));

💼 Tips：デバッグは「値を疑う」から

console.log にオブジェクトを渡すと後から中身が変わって見えることがあります。確定したスナップショットが欲しければ console.log(JSON.stringify(obj)) や DevTools のブレークポイントを使ってください。

🔴 上級編 — 実行モデルとモジュールで「なぜ動くか」を説明できる

イベントループ、プロトタイプ、ES Modules、モダンなツールチェーンまで。フレームワークの裏側を言葉で説明できるレベルを目指します。

13

イベントループ — 非同期が「後回し」になる理由

▼

💡 単一スレッドでも止まらない仕組み

JavaScript は基本的に単一スレッドです。 それでも UI が固まらずに非同期処理ができるのは、呼び出しスタック・Web API・タスクキュー・（マイクロタスクキュー）をイベントループが回しているからです。setTimeout(fn, 0) が「すぐ」ではなく「今の同期処理の後」に走る理由がここで解けます。

同期 → マイクロタスク → マクロタスクの順

Promise の then コールバックはマイクロタスク、setTimeout はマクロタスク（タスク）です。同じ「非同期」でも、Promise の方が先に消化されます。

JavaScript 
console.log("1: sync");

setTimeout(() => console.log("2: timeout"), 0);

Promise.resolve().then(() => console.log("3: promise"));

console.log("4: sync end");

// 出力順:
// 1: sync
// 4: sync end
// 3: promise
// 2: timeout

重い同期処理は UI を殺す

イベントループは「今のスタックが空になるまで」次のタスクを取れません。数千万回の計算をメインスレッドで回すと画面が固まります。重い処理は Web Worker へ逃がすか、チャンクに分割してください（本サイトの JS 実行も Worker を使っています）。

JavaScript 
// 悪い例：メインスレッドを長時間占有
function block(ms) {
  const end = Date.now() + ms;
  while (Date.now() < end) {}
}
// block(3000); // 画面が 3 秒固まる

// 良い方向性：小さな単位に分けて間を空ける
async function chunkedWork(items, fn) {
  for (const item of items) {
    fn(item);
    await new Promise((r) => setTimeout(r, 0)); // 描画や入力の隙間を作る
  }
}

💼 Tips：async/await もイベントループの上

await の続きはマイクロタスクとして再開されます。「await した瞬間に別スレッドに移る」わけではありません。ブロッキングな同期コードは await しても軽くなりません。

14

プロトタイプとクラス — 継承の仕組み

▼

💡 クラス構文の裏にあるプロトタイプ連鎖

ES2015 の class は糖衣構文です。 実体はプロトタイプベースの継承です。new、extends、super を使う現代コードでも、「インスタンス → プロトタイプ → Object.prototype」という探索のイメージを持っていると、メソッド解決や instanceof の挙動が腹落ちします。

class 構文での基本

コンストラクタで初期化し、メソッドはプロトタイプに定義されます（インスタンスごとに関数を複製しない）。プライベートフィールドは # 接頭辞です。

JavaScript 
class Animal {
  #energy = 100;

  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a sound`;
  }

  get energy() {
    return this.#energy;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name}: ワン!`;
  }
}

const d = new Dog("ポチ");
console.log(d.speak());
console.log(d instanceof Dog);    // true
console.log(d instanceof Animal); // true

プロトタイプ連鎖を覗く

Object.getPrototypeOf で一段上へ、obj.__proto__（非推奨だが DevTools では目にする）でも連鎖が見えます。「プロパティが見つからない → プロトタイプを見る」がメソッド呼び出しの正体です。

JavaScript 
const d = new Dog("ポチ");
console.log(Object.getPrototypeOf(d) === Dog.prototype); // true
console.log(Dog.prototype.speak); // 親ではなく Dog の speak

// オブジェクトの「辞書」用途とクラス用途は分ける
const dict = Object.create(null); // プロトタイプ汚染を避ける空辞書
dict.safe = true;

💼 Tips：継承よりコンポジション

深いクラス階層は変更に弱いです。共通処理は関数や小さなオブジェクトに切り出し、必要なら委譲する方が JS らしい設計です。React も「継承よりコンポジション」を公式に推奨しています。

ES Modules とモダン JS — import / バンドラの地図

▼

💡 ファイルを分割し、ツールチェーンにつなぐ

1 ファイルに全部書く時代は終わりました。 ES Modules（import / export）が言語標準のモジュールシステムです。ブラウザの type="module"、Node の "type": "module"、Vite / webpack などのバンドラ——いずれもこの構文が中心です。TypeScript や React プロジェクトのエントリポイントが何をしているか、ここで地図が読めます。

named export と default export

1 ファイルから複数を名前付きで出すのが基本、そのファイルの「主役」が 1 つなら default も使えます。チームでは named に統一する方がリネーム・自動補完に強い、という流派が有力です。

JavaScript 
// math.js
export function add(a, b) {
  return a + b;
}
export const PI = 3.14159;

// main.js
import { add, PI } from "./math.js";
console.log(add(2, PI));

// default の例
// export default function App() { ... }
// import App from "./App.js";

ブラウザとビルドツール

素のブラウザでもモジュールは動きますが、JSX・TypeScript・大量の npm パッケージを扱う実務では Vite などのバンドラが前提です。開発時はネイティブ ESM に近い体験、本番では結合・圧縮されたファイルを配信します。

HTML 
<!-- ブラウザで ESM を直接読む -->
<script type="module" src="./main.js"></script>

<!-- package.json（Node） -->
<!-- { "type": "module" } -->

💼 Tips：次の一歩は TypeScript と React

この章まで理解できていれば、TypeScript は「型注釈が付いた JavaScript」、React は「状態に応じて DOM を宣言的に描画するライブラリ」として読めます。ロードマップ上は HTML/CSS → JavaScript → TypeScript / React の順が最短です。
