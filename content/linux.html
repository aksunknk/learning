🐧 Linux / CLI

Docker・デプロイ・障害対応の前提となるシェル操作を、初級（操作）・中級（パイプ / プロセス / SSH）・上級（スクリプト / systemd）の 3 段階で学びます。

🟢 初級編 — シェルで「どこに何があるか」を操る

ターミナルの基本操作、パス、ファイル操作、権限、検索まで。Docker やサーバー作業の土台となる「手元の Linux を怖がらない」状態を作ります。

01

なぜ Linux / CLI なのか — サーバーは GUI がない

▼

💡 本番環境の共通言語はシェルである

クラウド上のサーバー、Docker コンテナ、CI の実行環境はほぼ Linux です。 しかも GUI は無く、操作はすべてコマンドライン（CLI）です。GUI クリックで覚えた操作は本番に持ち込めません。フルスタック志望者にとってシェルは「趣味の上級技」ではなく、デプロイ・ログ確認・障害対応の最低条件です。Windows でも WSL2 を入れれば同じコマンドが使えます。

シェルとは何か

シェル（bash / zsh など）は、あなたが打った文字を OS への命令に翻訳するプログラムです。ファイル操作もプロセス起動も、最終的にはシェル経由で行われます。プロンプト（$）が表示されている行にコマンドを打ち、Enter で実行します。

Bash 
# 今のユーザー名とホスト名を確認
whoami
hostname

# OS のざっくり情報（ディストリビューション名など）
uname -a
cat /etc/os-release 2>/dev/null || true

# ホームディレクトリへ移動して確認
cd ~
pwd

学習環境の選び方

macOS は標準で Unix 系シェルが使えます。Windows は WSL2（Ubuntu） を入れるのが最短です。Git Bash だけでも初歩は学べますが、権限やパッケージ管理までやるなら WSL か Linux VM / クラウドインスタンスが本番に近いです。

💼 Tips：怖いコマンドの前に man / --help

man ls または ls --help で使い方が出ます。全部読む必要はなく、「よく使うオプション」を拾う習慣が大事です。破壊的な操作（rm -rf など）は、まずパスを echo や ls で確認してから実行してください。

02

ディレクトリとパス — pwd / ls / cd

▼

💡 迷子にならないための「現在地」感覚

CLI で最も多い初歩ミスは「今どのディレクトリにいるか分からない」ことです。 pwd（Print Working Directory）で現在地を出し、ls で中身を見、cd で移動する——この 3 つが全操作の起点です。絶対パス（/home/...）と相対パス（./src）の違いを体で覚えると、Docker の WORKDIR や設定ファイルのパス指定が一気に読みやすくなります。

現在地と一覧

ls -la は詳細表示（隠しファイル含む）の定番です。. はカレント、.. は親ディレクトリを表します。

Bash 
pwd                    # 今いる場所の絶対パス
ls                     # 名前だけ
ls -la                 # 権限・所有者・日時つき + 隠しファイル
ls -lh /var/log        # 人間が読めるサイズ表示

cd /tmp                # 絶対パスで移動
cd ..                  # 一つ上へ
cd ~                   # ホームへ
cd -                   # 直前のディレクトリへ戻る

絶対パスと相対パス

絶対パスはルート / から始まる住所、相対パスは「今いる場所から見た道順」です。スクリプトや Docker では絶対パスの方が誤解が少なく、対話作業では相対パスが速い、という使い分けが一般的です。

Bash 
# 例: ホームが /home/dev のとき
cd /home/dev/projects/app   # 絶対
cd ~/projects/app           # ~ はホームの省略形（絶対相当）
cd projects/app             # ホームにいる前提の相対

# パス結合の確認
realpath ./projects/app     # 絶対パスに正規化（使える環境なら）

💼 Tips：Tab 補完を使え

パスやコマンド名は Tab キーで補完できます。曖昧なら二度押しで候補一覧が出ます。手打ちのタイプミスによる rm 事故を減らす、最もコストの低い防衛策です。

03

ファイル操作 — mkdir / cp / mv / rm

▼

💡 作る・移す・消すを安全に行う

ファイル操作コマンドは「元に戻せない」ものが多いです。 特に rm にゴミ箱はありません。まずは mkdir / touch / cp / mv で作業し、rm は対象を ls で確認してから、可能なら -i（確認）付きで実行する癖を付けてください。

作成とコピー・移動

mkdir -p は途中のディレクトリもまとめて作ります。cp -r はディレクトリごとコピー、mv は移動兼リネームです。

Bash 
mkdir -p ~/work/demo/src
touch ~/work/demo/README.md
echo "hello" > ~/work/demo/README.md

cp ~/work/demo/README.md ~/work/demo/README.bak
cp -r ~/work/demo ~/work/demo-copy

mv ~/work/demo/src ~/work/demo/app
mv ~/work/demo/README.md ~/work/demo/readme.md   # リネーム

削除は段階的に

ファイル削除は rm、空ディレクトリは rmdir、中身ごと消すのは rm -r。本番では rm -rf / のような命令は論外で、変数展開ミス（rm -rf $DIR/ で DIR が空）も実在する事故です。

Bash 
# まず確認
ls -la ~/work/demo-copy

# 対話確認付きで削除（推奨）
rm -i ~/work/demo-copy/readme.md

# ディレクトリごと（実行前にパスを再確認）
rm -ri ~/work/demo-copy

# 危険パターン（やってはいけない例）
# rm -rf $EMPTY_VAR/   # EMPTY_VAR が空だと / 配下を消そうとする

💼 Tips：消す前に tar か cp で退避

自信が無い削除の前に cp -r target target.bak や tar czf backup.tgz target で退避すると精神衛生が保たれます。クラウドのスナップショットと同じ発想をローカルでも使ってください。

04

ファイルの中身を見る — cat / less / head / tail

▼

💡 ログ調査の第一歩は「読む」こと

サーバー障害対応の大半はログを読む作業です。 短いファイルは cat、長いファイルは less、先頭だけ head、末尾（最新ログ）は tail。tail -f でリアルタイム追従できると、起動中のアプリや Docker コンテナの様子が手に取るように分かります。

基本の読み方

less では Space で次ページ、/ で検索、q で終了です。cat で巨大ファイルを流すとターミナルが埋まるので、ログには less / tail を優先してください。

Bash 
cat ~/work/demo/readme.md
less /etc/hosts          # q で抜ける
head -n 20 /etc/passwd   # 先頭20行
tail -n 50 /var/log/syslog 2>/dev/null || tail -n 20 ~/.bashrc

# 追従（Ctrl+C で停止）
# tail -f /var/log/nginx/access.log

行番号と差分の入口

nl や cat -n で行番号付き表示、設定ファイルの比較には後で diff を使います。まずは「末尾を見る」「キーワード周辺を見る」ができれば十分です。

Bash 
cat -n ~/.bashrc | head
wc -l ~/.bashrc          # 行数・単語数・バイト数
diff -u file_a file_b    # 統一フォーマットの差分（あれば）

💼 Tips：sudo が必要なログがある

/var/log 配下は権限で読めないことがあります。sudo less /var/log/... や sudo journalctl（後述）を使います。パスワードを聞かれたら、今のユーザーのログインパスワードです（root パスワードとは限りません）。

05

権限の基礎 — ls -l / chmod / sudo

▼

💡 誰が読めて、誰が実行できるか

Linux のセキュリティモデルの中心は「ユーザー・グループ・その他」への読み書き実行権限です。 Docker イメージ内で「Permission denied」が出る、デプロイ先でスクリプトが動かない——原因の大半はここに帰着します。ls -l の左端 10 文字が読めるようになれば、権限トラブルの切り分けが格段に速くなります。

ls -l の読み方

-rwxr-xr-- のような表記は、ファイル種別 1 文字＋所有者 rwx＋グループ rwx＋その他 rwx です。r=4, w=2, x=1 の合計が数値モード（例: 755）になります。

Bash 
ls -l ~/work/demo/readme.md
# 例: -rw-r--r-- 1 dev dev 6 Apr 1 12:00 readme.md
#     │└所有者┘└グループ┘└その他┘
#     └ 通常ファイル

# よく使う数値モード
# 644: 所有者 rw、他は r（設定・文書）
# 755: 所有者 rwx、他は rx（ディレクトリや実行スクリプト）
# 600: 所有者だけ rw（秘密鍵・.env）

chmod と sudo

chmod で権限変更、chown で所有者変更（多くは sudo 必須）。sudo は「一時的に管理者権限で実行」です。常用ユーザーで作業し、必要なときだけ sudo するのが原則です。

Bash 
chmod 644 ~/work/demo/readme.md
chmod 755 ~/work/demo/run.sh
chmod u+x ~/work/demo/run.sh     # 所有者に実行権を追加（記号モード）

# 秘密情報は他者に読ませない
chmod 600 ~/.ssh/id_ed25519 2>/dev/null || true

# 管理者権限が必要な例（環境による）
# sudo chown -R $USER:$USER /var/www/app

💼 Tips：ディレクトリの x は「進入権」

ディレクトリに実行権（x）が無いと、中に cd できません。ファイルの x は「実行できる」、ディレクトリの x は「通れる」——意味が違う点に注意してください。

06

検索する — find / grep / which

▼

💡 「どこにある？」を数秒で終わらせる

巨大なリポジトリやサーバー上でファイルを目視で探すのは非現実的です。 名前なら find、中身の文字列なら grep、コマンドの実体パスなら which / command -v。この 3 つが使えれば、日常の調査速度が桁変わりします。

find と grep

find はディレクトリ木を歩き、grep -R はファイル内容を再帰検索します。巨大な node_modules は除外するのが実務の作法です。

Bash 
# 名前で探す
find ~ -name "*.log" 2>/dev/null | head
find . -type f -name "Dockerfile"

# 中身で探す（-n 行番号, -I バイナリ除外）
grep -Rn "DATABASE_URL" . --exclude-dir=node_modules --exclude-dir=.git

# コマンドの場所
which node
command -v docker
type python3

パイプでつなぐ予習

検索結果をさらに絞り込むとき、find ... | head のようにパイプ（|）を使います。次の中級編で本格的に扱いますが、出力が多すぎるときの応急処置として今すぐ使えます。

Bash 
ls -la /usr/bin | grep python | head
dmesg 2>/dev/null | tail -n 5 || true

💼 Tips：ripgrep (rg) を知っておく

開発マシンでは rg（ripgrep）が grep -R より速くて使いやすいことが多いです。サーバーに入っていないこともあるので、標準の grep / find は必ず残してください。

🔶 中級編 — パイプ・環境変数・プロセス・SSH で実務の作業をする

コマンドを組み合わせ、環境を渡し、プロセスを制御し、リモートサーバーへ入る——Docker やデプロイで毎日使う中核スキルです。

07

パイプとリダイレクト — コマンドを組み立てる

▼

💡 小さな命令を繋いで大きな仕事にする

Unix 哲学の中心は「一つのことをうまくやるプログラムをパイプで繋ぐ」ことです。 | で標準出力を次の標準入力へ、> / >> でファイルへ保存、2> でエラー出力を分けます。ログ解析もデプロイスクリプトも、この組み合わせで書かれています。

パイプとフィルタ

sort / uniq / wc / head はパイプ先の定番です。長いチェーンでも、途中で一度ファイルに落とすとデバッグしやすくなります。

Bash 
# 行数を数える
ls -1 | wc -l

# 重複を除いて数える
cut -d: -f1 /etc/passwd | sort | uniq | wc -l

# エラー出力だけ捨てる
find /etc -name "*.conf" 2>/dev/null | head

リダイレクトの使い分け

> は上書き、>> は追記。2>&1 は標準エラーを標準出力に合流（ログをまとめて保存するとき頻出）です。

Bash 
echo "first" > /tmp/demo.txt      # 上書き作成
echo "second" >> /tmp/demo.txt    # 追記
cat /tmp/demo.txt

# 成功も失敗も同じログへ
# ./deploy.sh > /tmp/deploy.log 2>&1

# 入力をファイルから
sort < /tmp/demo.txt

💼 Tips：誤って > で消さない

重要なファイルへの > は中身を即座に消します。上書き前に cp file file.bak、または set -o noclobber で上書き防止する習慣が有効です。

08

環境変数と PATH — プログラムが見る「設定」

▼

💡 同じコマンドでも環境で挙動が変わる理由

アプリの接続先 DB や秘密鍵は、コードに直書きせず環境変数で渡すのが現代の標準です。 Docker の -e、docker-compose の environment、クラウドのシークレットマネージャ——すべて同じ発想です。あわせて PATH を理解すると、「コマンドが見つからない」エラーを自分で解決できます。

表示・設定・エクスポート

シェル変数と環境変数は違います。export した変数が子プロセス（起動するプログラム）に引き継がれます。

Bash 
echo $HOME
echo $PATH
env | head                  # 環境変数一覧（冒頭）

# 現在のシェルだけ
MSG="hello"
echo $MSG

# 子プロセスにも渡す
export DATABASE_URL="postgres://localhost:5432/app"
export NODE_ENV=development
printenv DATABASE_URL

PATH と .env の作法

PATH はコマンド検索パスのリスト（: 区切り）です。プロジェクトの秘密情報は .env に書き、Git にコミットしない（.gitignore）のが鉄則です。

Bash 
# PATH の先頭に個人用 bin を足す例（~/.bashrc などに書く）
export PATH="$HOME/.local/bin:$PATH"

# .env を読んで export（簡易・本番では専用ツール推奨）
# set -a; source ./.env; set +a

# 秘密が履歴に残らないよう注意
# export API_KEY=... を素のシェルに直打ちすると履歴に残ることがある

💼 Tips：確認は echo / printenv、引き回しは export

「コンテナ内で変数が入っているか」は printenv NAME で確認します。Dockerfile の ENV と実行時の -e のどちらで入ったかも、この章の知識で切り分けできます。

09

プロセス管理 — ps / top / kill

▼

💡 動いているプログラムを見つけて止める

ポートが既に使われている、CPU が張り付いている、ゾンビのように残った Node プロセス——開発でも本番でも必ず遭遇します。ps で一覧、top/htop で監視、kill で終了。Docker では docker ps がこれに相当しますが、ホスト側のプロセス理解は依然として必須です。

探す・見る

ps aux は全部のプロセスを出す定番です。pgrep / pidof で名前から PID を特定できます。

Bash 
ps aux | head
ps aux | grep -i node | grep -v grep

# 自分のプロセスだけ
ps -u $USER

# インタラクティブ監視（q で終了）
# top
# htop   # 入っていればこちらが読みやすい

止める：SIGTERM から

まず kill PID（SIGTERM：優雅な終了要求）、ダメなら kill -9 PID（SIGKILL：強制）。最初から -9 はデータ破損のリスクがあるため最終手段です。

Bash 
# 例: PID を変数に入れる
PID=$(pgrep -n python3)   # 条件に注意して使う
echo "target=$PID"
# kill "$PID"             # 確認してから外す
# kill -9 "$PID"          # 最終手段

# ポートを塞いでいるプロセスを調べる（後述の ss と併用）
# ss -ltnp | grep 8000

💼 Tips：バックグラウンドとジョブ

コマンド末尾の & で背景実行、jobs で一覧、fg で前面に戻せます。長時間タスクは tmux / screen や systemd に載せる方が本番的です。

10

ネットワーク確認 — curl / ss / ping

▼

💡 「繋がらない」を切り分ける

API が落ちているのか、DNS か、ポート未解放か、ファイアウォールか——切り分けの起点は CLI のネットワークコマンドです。curl で HTTP、ss（または lsof）で待ち受けポート、ping で疎通の大まかな確認。Docker 公開ポートのデバッグでも同じ道具を使います。

curl で HTTP を叩く

ブラウザの代わりにリクエストを送り、ステータスやヘッダを見ます。API 開発の必携ツールです。

Bash 
# ヘッダ付きで見る（-i） / 失敗を判別しやすく（-f） / 詳細（-v）
curl -i https://example.com
curl -sS -o /dev/null -w "%{http_code}\n" https://example.com

# JSON API
curl -sS https://httpbin.org/get | head
curl -sS -X POST https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"name":"dev"}'

ポートと疎通

ss -ltnp で LISTEN 中の TCP ポートが見えます（環境により sudo が必要）。コンテナを立てたのに繋がらないときは、まずホストで待ち受けているかを確認します。

Bash 
ping -c 3 1.1.1.1
ss -ltnp 2>/dev/null | head
# sudo ss -ltnp | grep 3000

# DNS の確認（どちらか使える方）
getent hosts example.com
# dig example.com +short

💼 Tips：localhost と 0.0.0.0 の違い

アプリが 127.0.0.1 だけに bind していると、コンテナ外や他マシンから届きません。Docker では 0.0.0.0:ポート で待ち受ける設定が多いです。「ローカルでは動くのに外から見えない」の典型原因です。

11

SSH — リモートサーバーへ入る

▼

💡 デプロイ先の操作は SSH が入口

VPS やクラウドインスタンスに入る標準手段が SSH です。 鍵認証を使い、パスワードログインを閉じるのが現代の最低ラインです。Git の SSH 鍵とも概念は同じで、「秘密鍵は渡さない・権限は 600・エージェントに載せる」を守れば事故りにくいです。

鍵の作成と接続

鍵ペアを作り、公開鍵をサーバーの ~/.ssh/authorized_keys に置きます。接続は ssh user@host。初回はホスト指紋の確認が出ます。

Bash 
# 鍵を作る（パスフレーズ付き推奨）
ssh-keygen -t ed25519 -C "you@example.com" -f ~/.ssh/id_ed25519_deploy

# 公開鍵をサーバーへ（ssh-copy-id が使える場合）
# ssh-copy-id -i ~/.ssh/id_ed25519_deploy.pub user@203.0.113.10

# 接続
# ssh -i ~/.ssh/id_ed25519_deploy user@203.0.113.10

# 設定ファイルで省略（~/.ssh/config）
# Host myvps
#   HostName 203.0.113.10
#   User user
#   IdentityFile ~/.ssh/id_ed25519_deploy
# 接続: ssh myvps

トンネルとファイル転送

一時的にポートを持ち帰るなら SSH トンネル、scp / rsync でファイル同期します。デプロイの素朴な形は「rsync でコードを送り、ssh 先で docker compose up」です。

Bash 
# ファイルコピー
# scp -i ~/.ssh/id_ed25519_deploy app.tgz user@host:/tmp/

# 差分同期（削除は --delete に注意）
# rsync -avz -e "ssh -i ~/.ssh/id_ed25519_deploy" \
#   ./dist/ user@host:/var/www/app/

# ローカル 8080 → リモートの localhost:80 を見る例
# ssh -L 8080:127.0.0.1:80 user@host

💼 Tips：秘密鍵をリポジトリに入れない

id_* や .pem を Git にコミットするのは即事故です。CI ではシークレット機能に載せ、ローカルは ~/.ssh に置き、chmod 600 を徹底してください。

12

よくある事故と安全な操作 — 壊れ方を先に知る

▼

💡 権限・パス・履歴・再実行で起きる典型パターン

CLI の実力は「速いコマンドを知っている」ことより「危険な操作を避ける」ことにあります。 本番での誤削除、履歴に残した秘密、カレントディレクトリを誤ったままの rm——事前にパターンを知っていれば防げます。

やってはいけない／疑うべき操作

ルート直下への破壊的操作、未確認のコピー＆ペースト、ネットから拾ったスクリプトの直 curl | bash は特に危険です。内容を読んでから実行が原則です。

Bash 
# 危険: 中身を確認せずパイプ実行
# curl -fsSL https://example.com/install.sh | bash

# まし: 一度保存して読む
curl -fsSL https://example.com/install.sh -o /tmp/install.sh
less /tmp/install.sh
# bash /tmp/install.sh

# 現在地を確認してから消す
pwd
ls
# rm -rf ./build

履歴と再実行

history にパスワードやトークンが残ることがあります。機密は環境変数やシークレットマネージャへ。sudo !! は直前コマンドの再実行で便利ですが、内容を見てから使ってください。

Bash 
history | tail -n 20
# 履歴に残したくない（環境による）
# unset HISTFILE
# または先頭にスペース（ignorespace 設定時）

# 直前コマンドを sudo で再実行
# sudo !!

💼 Tips：本番では「読むコマンド」から入る

障害対応の初期は pwd / ls / ps / ss / tail / journalctl など状態を変えないコマンドで状況を把握します。いきなり再起動や削除に走らないのがプロの手順です。

🔴 上級編 — スクリプト化とサービス運用の入口

繰り返し作業をシェルスクリプトにし、systemd / ログで「動かし続ける」視点を得ます。Docker ホストや VPS 運用の直前段階です。

13

シェルスクリプト入門 — 繰り返しを自動化する

▼

💡 手動手順をファイルに閉じ込める

同じ 5 コマンドを毎日打つなら、スクリプトにします。 shebang、set -euo pipefail、引数、if——最低限これだけあれば、デプロイ補助やバックアップの雛形が書けます。完璧な bash 職人になる必要はなく、「再現可能な手順書」として持てることが目的です。

雛形と安全オプション

set -e はエラーで停止、-u は未定義変数をエラー、-o pipefail はパイプ途中の失敗も拾います。実務スクリプトの冒頭定番です。

Bash 
#!/usr/bin/env bash
set -euo pipefail

NAME="${1:-world}"
OUT_DIR="${2:-./out}"

mkdir -p "$OUT_DIR"
echo "Hello, $NAME" > "$OUT_DIR/greeting.txt"
echo "wrote $OUT_DIR/greeting.txt"

実行権とデバッグ

chmod +x script.sh のあと ./script.sh。bash -x script.sh で実行トレースが見えます。

Bash 
chmod +x ./greet.sh
./greet.sh "佐藤" /tmp/greet-out
cat /tmp/greet-out/greeting.txt

# 追跡実行
bash -x ./greet.sh "dev"

💼 Tips：変数は必ずクォートする

"$var" で空白や空文字の事故を防ぎます。rm -rf $dir/ で dir が空だと悲劇になる、は前述のとおりです。スペースを含むパスを扱うならクォートは必須です。

14

サービスとログ — systemctl / journalctl

▼

💡 「起動して終わり」ではなく「動かし続ける」

Linux サーバー上の常駐プロセスは、多くの場合 systemd が管理しています。 Nginx や Docker デーモン、自作 API も unit として登録されることがあります。systemctl status と journalctl -u が使えれば、VPS デプロイ後の「なぜ起動しない」の一次切り分けができます。

systemctl の基本

状態確認・起動・再起動・有効化（ブート時自動起動）が主な操作です。権限により sudo が必要です。

Bash 
# 例: ssh や docker が入っている環境で
# systemctl status docker
# sudo systemctl restart docker
# sudo systemctl enable docker   # 起動時に自動開始
# sudo systemctl disable docker

# 自分のユーザーサービスもある（環境による）
# systemctl --user status ...

journalctl でログを追う

systemd 配下のログは journal に集約されます。-u でユニット指定、-f で追従、--since で時間絞り込みが定石です。

Bash 
# sudo journalctl -u nginx -n 100 --no-pager
# sudo journalctl -u docker -f
# sudo journalctl --since "10 min ago" | less

# Docker を使っている場合の対比
# docker compose logs -f api

💼 Tips：アプリログとシステムログを両方見る

アプリが標準出力に出すログ（Docker logs）と、OS / プロキシ側のログ（journal / nginx）は別物です。502 のときは両方を見ないと原因を取り違えます。次の Docker 章・デプロイ作業でこの視点がそのまま活きます。
