🌿 Git / GitHub — チーム開発の共通言語

コードの歴史を記録し、安全に実験し、他人と協働するための道具がGitです。実務では初日から使います。「コミットとは何か」から「やらかしたときの復旧」まで、現場で必要な操作を一通り身につけましょう。

01

バージョン管理とは — Gitの3つの領域

▼

💡 Gitは「セーブポイント付きのタイムマシン」— ファイル名_final_v2_最新.zip とは今日でお別れ

なぜバージョン管理が必要なのか？

コードを書いていると「昨日までは動いていたのに」「どこを変えたか思い出せない」が必ず起きます。Gitは変更の歴史をすべて記録し、任意の時点に戻れるようにします。さらに、複数人が同じコードを同時に編集しても安全に合流（マージ）できます。

Gitを理解する鍵は3つの領域です。ファイルはこの3つの間を移動します。

概念図 — 3つの領域
┌──────────────────┐  git add   ┌──────────────────┐  git commit ┌──────────────────┐
│ ワーキングツリー │ ─────────→ │ ステージングエリア│ ──────────→ │  リポジトリ      │
│ (作業中のファイル)│            │ (次のコミットの   │             │ (確定した歴史)   │
│                  │ ←───────── │  荷造りスペース)  │             │ .git ディレクトリ │
└──────────────────┘ git restore└──────────────────┘             └──────────────────┘

・ワーキングツリー: いま編集しているファイルそのもの
・ステージング:     「次のセーブに含めるもの」を選ぶ場所
・リポジトリ:       コミット（スナップショット）の保管庫

💼 Tips：実務で遭遇する現場の勘所

なぜステージングが挟まっているのか: 「バグ修正とリファクタリングを同時にやってしまった」とき、ステージングがあれば変更の一部だけを選んでコミットできます（git add -p）。この「変更を意味の単位に分ける」機能こそがGitの設計思想の核心です。

02

基本サイクル — init / add / commit / log / diff

▼

💡 日常の90%は「編集 → status → add → commit」のループ — 手に馴染むまで繰り返す

Bash — 最初のコミットまで
# ① リポジトリを作る（プロジェクトで最初に1回だけ）
git init

# ② いま何が起きているかを確認する（迷ったら常にこれ）
git status

# ③ 変更をステージングに載せる
git add main.py          # ファイル指定
git add .                # カレント以下すべて（.gitignore 対象は除外）

# ④ コミット = スナップショットを確定する
git commit -m "Add login validation"

# ⑤ 歴史を眺める
git log --oneline --graph   # 1行表示 + ブランチの分岐図

# ⑥ 差分を見る
git diff                 # ワーキングツリー vs ステージング
git diff --staged        # ステージング vs 直前のコミット

🔍 良いコミットメッセージの型

1行目は「何をしたか」を命令形で50文字以内（例: Fix off-by-one error in pagination）。「なぜ」が自明でない場合は空行を挟んで本文に書きます。「修正」「更新」だけのメッセージは、3ヶ月後の自分にとって無情報です。

💼 Tips：実務で遭遇する現場の勘所

コミットの粒度は「1つの論理的変更 = 1コミット」: 「バグ修正とタイポ修正と機能追加」を1コミットに混ぜると、あとで片方だけ取り消す（revert）ことができなくなります。逆に細かすぎる「WIP」「fix」「fix2」の連打も歴史を汚します。レビュアーが1コミットずつ読んで理解できる単位が正解です。

03

ブランチとHEAD — 並行世界を作って安全に実験する

▼

💡 ブランチは「コミットを指す軽量なポインタ」— 作るのに1秒もかからないので、気軽に作って気軽に捨てる

ブランチの正体

ブランチと聞くと「コードのコピー」を想像しがちですが、実体は特定のコミットを指す名前付きポインタ（41バイトのテキストファイル）です。だから作成も切替も一瞬です。HEAD は「いま自分がどのブランチにいるか」を指すポインタです。

Bash — ブランチ操作
# ブランチを作って移動する（-c = create）
git switch -c feature/login-form

# いまどこにいるか確認
git branch          # * が付いているのが現在地（HEAD）

# main に戻る
git switch main

# 不要になったブランチを削除
git branch -d feature/login-form   # マージ済みなら消せる
git branch -D feature/login-form   # 強制削除（未マージでも消す）

# 歴史のイメージ:
#            ┌── C3 ── C4   ← feature/login-form
# C1 ── C2 ──┤
#            └── C5        ← main (HEAD)

💼 Tips：実務で遭遇する現場の勘所

main に直接コミットしない: 個人開発でも「main は常に動く状態を保ち、作業はブランチで行う」習慣を付けてください。チーム開発ではmainへの直接pushは保護設定（Branch Protection）で物理的に禁止されるのが普通です。ブランチ名は feature/〜、fix/〜 のように種別を接頭辞にすると一覧性が上がります。

04

mergeとコンフリクト解消 — 怖くない、慌てない

▼

💡 コンフリクトは「エラー」ではなく「Gitが判断を人間に委ねた」だけ — マーカーを読めば怖くない

マージとコンフリクトの仕組み

git merge は2つのブランチの変更を合流させます。別々のファイル・別々の行を変更していればGitが自動で統合しますが、同じファイルの同じ行を両方が変更していた場合、Gitはどちらが正しいか判断できません。これがコンフリクト（衝突）です。

Bash — コンフリクトの発生と解消
git switch main
git merge feature/pricing
# CONFLICT (content): Merge conflict in price.py

# ファイルを開くとマーカーが挿入されている:
# <<<<<<< HEAD
# TAX_RATE = 0.10          ← 自分側（main）の内容
# =======
# TAX_RATE = 0.08          ← 相手側（feature/pricing）の内容
# >>>>>>> feature/pricing

# ① 正しい内容に手で編集する（マーカーは全部消す）
# ② 解消したことをGitに伝える
git add price.py
# ③ マージを完了する
git commit               # メッセージは自動生成されるのでそのままでOK

# 途中でやめたくなったら（マージ前の状態に戻る）:
git merge --abort

💼 Tips：実務で遭遇する現場の勘所

コンフリクトを小さくするのは「こまめなマージ」: 長生きしたブランチほどmainから離れ、コンフリクトが巨大化します。作業ブランチには git merge main で定期的にmainを取り込んでください。また、解消時に「両方の変更を活かすべき」ケースが多いことに注意。機械的に片方を選ぶと相手の修正を消してしまいます（これが実務で最も多いマージ事故です）。

05

リモートリポジトリ — clone / fetch / pull / push

▼

💡 pull = fetch + merge — 「pullで何が起きるか」を分解して理解すればリモート操作は怖くない

Bash — リモートの基本操作
# GitHub上のリポジトリを手元に複製する
git clone https://github.com/user/repo.git

# リモートの最新状態を「取得だけ」する（手元のコードは変わらない）
git fetch origin

# 取得 + 現在のブランチに統合（= fetch + merge）
git pull origin main

# 手元のコミットをリモートに送る
git push origin feature/login-form

# 新しいブランチを初めてpushするとき（-u で追跡関係を記録）
git push -u origin feature/login-form
# 以降は git push / git pull だけでOKになる

# リモートの登録状況を確認
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

🔍 push が拒否されたら

! [rejected] ... non-fast-forward は「リモートに自分の知らないコミットがある」という意味です。まず git pull でリモートの変更を取り込み、必要ならコンフリクトを解消してから再度pushします。git push --force で上書きしてはいけません。チームメイトのコミットが消えます。

💼 Tips：実務で遭遇する現場の勘所

「1日の終わりにpush」を習慣に: ローカルにしかないコミットは、PCの故障で永遠に失われます。作業ブランチは未完成でもpushして構いません（それを共有するためのDraft PRという仕組みもあります）。リモートはバックアップでもあるのです。

06

GitHubフロー — Pull Requestを中心とした開発の流れ

▼

💡 Pull Request は「マージしてください」という依頼書 — コードレビューと自動テストの関所を通ってからmainに入る

GitHubフローの5ステップ

世界中のチームが使う最もシンプルな開発フローです。

① ブランチを切る — mainから feature/xxx を作成
② コミットを積む — 小さな単位でこまめにコミット
③ pushしてPRを開く — タイトルと説明で「何を・なぜ」を伝える
④ レビューとCI — 指摘に対応してコミットを追加（PRに自動反映される）
⑤ マージしてブランチ削除 — mainは常にデプロイ可能な状態を保つ

Bash + GitHub — PRを出すまで
git switch -c fix/cart-total-rounding
# ...編集...
git add -p                       # 変更を確認しながらステージング
git commit -m "Fix rounding error in cart total"
git push -u origin fix/cart-total-rounding

# → GitHubの画面に「Compare & pull request」ボタンが出る
#   PRの説明に書くべきこと:
#   - 何を変更したか（What）
#   - なぜ必要か（Why）
#   - 動作確認の方法（How to test）

# OSSに貢献する場合は fork（自分のアカウントへの複製）から:
# fork → clone → ブランチ → push → 本家リポジトリへPR

💼 Tips：実務で遭遇する現場の勘所

PRは小さいほど速くマージされる: 統計的に、レビュー時間はPRのサイズに対して非線形に増えます。400行を超えるPRはレビューの質が急落します。「機能の一部でも動く単位」で分割してPRを出すのがプロの作法です。また、自分でPRの差分を最初にセルフレビューしてからレビュアーを呼ぶと、指摘が半減します。

07

やらかし復旧大全 — restore / reset / revert / reflog

▼

💡 コミット済みの変更はほぼ絶対に復元できる — 慌てて余計なコマンドを打つ前に reflog

Bash — 状況別リカバリー早見表
# 【状況1】ファイルの変更を取り消したい（コミット前）
git restore main.py              # 最後のコミットの状態に戻す
git restore --staged main.py    # add を取り消す（変更自体は残る）

# 【状況2】直前のコミットメッセージを間違えた
git commit --amend -m "正しいメッセージ"
# ※ push済みのコミットにamendしてはいけない（歴史が食い違う）

# 【状況3】直前のコミット自体をなかったことにしたい（未push）
git reset --soft HEAD~1    # コミットだけ取り消し、変更はstagedのまま
git reset --hard HEAD~1    # 変更ごと完全に消す（要注意！）

# 【状況4】push済みのコミットを取り消したい
git revert abc1234         # 「打ち消しコミット」を新たに作る
# → 歴史を書き換えないので、共有ブランチではこれ一択

# 【状況5】reset --hard で消してしまった／ブランチを消してしまった
git reflog                 # HEADの移動履歴がすべて残っている
# abc1234 HEAD@{1}: commit: 消えたはずのコミット
git switch -c rescue abc1234   # そのコミットからブランチを復元

🔍 reset と revert の使い分け

reset は歴史そのものを巻き戻す（push前の自分専用）。revert は「取り消した」という新しい歴史を追加する（push後・共有ブランチ用）。この区別を間違えてpush済みブランチをresetすると、チーム全員のリポジトリと食い違いが起きます。

💼 Tips：実務で遭遇する現場の勘所

reflogの保持期間はデフォルト90日: つまり3ヶ月以内のやらかしはほぼ確実に救えます。本当に失われるのは「一度もaddしていないファイルの変更」だけです。だからこそ、実験前に git add か git stash（一時退避）をしておく癖が保険になります。

08

実務Tips — .gitignore / stash / rebase入門

▼

💡 「コミットしてはいけないもの」を最初に決める — .gitignore はプロジェクト開始時の必須作業

.gitignore — 典型的な除外設定
# 依存パッケージ（サイズが巨大 & 再生成可能）
node_modules/
venv/
__pycache__/

# 秘密情報（漏洩事故の最多原因！）
.env
*.pem
credentials.json

# ビルド成果物・エディタ設定
dist/
build/
.DS_Store
.vscode/
Bash — stash と rebase
# stash: 作業を一時退避（急な割り込み作業に）
git stash               # 変更を退避して作業ツリーをきれいに
git stash pop           # 退避した変更を戻す

# rebase: ブランチの根本を付け替えて歴史を一直線にする
git switch feature/xxx
git rebase main         # mainの最新の上に自分のコミットを載せ直す
# マージコミットが無い直線的な歴史になりログが読みやすい

# ⚠️ rebaseの鉄則: push済み（共有済み）のコミットをrebaseしない
#    歴史を書き換えるため、他人のリポジトリと食い違いが起きる

🔍 APIキーをコミットしてしまったら

「すぐ消せばバレない」は通用しません。歴史に残った秘密情報はpushした時点で漏洩したものとして扱い、まずキーを無効化・再発行してください。歴史からの完全削除には git filter-repo などの専用ツールが必要で、共有リポジトリでは全員の協力が要ります。予防（.gitignore + push前の git diff --staged 確認）が圧倒的に安上がりです。

💼 Tips：実務で遭遇する現場の勘所

迷ったらこの使い分け: 個人ブランチの整理は rebase、共有ブランチへの統合は merge、共有済みコミットの取り消しは revert。この3原則だけ守れば、Gitで取り返しのつかない事故はまず起きません。
