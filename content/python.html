01

変数とデータ型（基礎中の基礎）

▼

💡 データを格納する「箱」と、その種類について

なぜ変数が必要なのか？変数がなければ、プログラムはただ1回計算して終わる電卓に過ぎません。ユーザーからの入力や、APIからのレスポンスなど、「状態（データ）」を一時的に保存・更新し、次の処理へと渡していくために絶対不可欠な概念です。また、後からコードを読む人間（やAI）に対して「この値は何を意味しているか」という意図を伝える最大のドキュメントにもなります。

変数とデータ型の基礎

Pythonは「動的型付け」と呼ばれる言語で、変数を作るときに型を宣言する必要がありません。しかし、だからこそ「今この変数には何型のデータが入っているのか」を意識しないと、予期せぬエラー（TypeErrorなど）を招きます。

PYTHON 
# ❌ 悪い例（マジックナンバー：数値が直接書かれていて意図が不明）
# プロの現場でこれを書くと確実にコードレビューで弾かれます。
print(1000 * 1.1)

# ✅ 良い例（変数を使って意味を持たせる）
price = 1000        # int型（整数）
tax_rate = 1.1      # float型（浮動小数点）

# 計算結果を新たな変数に格納する
total_price = int(price * tax_rate) # 明示的にintでキャストし、小数点以下を切り捨てる

# f-string（f文字列）を使った出力
# str型とint型を直接 + で繋ぐとエラーになるため、実務ではf-stringを多用します。
print(f"商品の税込価格は {total_price} 円です。")

💼 Tips：実務で遭遇する現場の勘所

変数名には必ず「具体的で検索しやすい名前」をつけてください。a = 10 や data = [...] のような曖昧な命名は、デバッグ時に地獄を見ます。また、Pythonでは変数名にスネークケース（例: user_name）を使うのがPEP8という標準規約で決まっています。AIにコード生成を依頼する際も、「user_nameという変数に...」と具体的に指定すると精度が劇的に上がります。

02

リストと辞書（データの集まり）

▼

💡 複数のデータを一括で管理する魔法の道具

なぜリストや辞書が必要なのか？「100人のユーザーデータ」を user1, user2... user100 と変数で定義するのは不可能です。リストを使えば「順序のあるデータの列」を、辞書を使えば「ラベル（キー）で引けるデータのまとまり」を1つの変数として管理できます。特に辞書は、現代のWeb開発で標準フォーマットとなっているJSONと1対1で対応するため、API通信において最重要のデータ構造です。

リスト（配列）と辞書（dict）の使い分け

単純な一覧ならリスト、属性のまとまりなら辞書を使います。この2つをネスト（入れ子）にする構造が、実務で最も頻繁に登場します。

PYTHON 
# --- リストの基本と罠 ---
fruits = ['Apple', 'Banana', 'Orange']

# インデックスは0から始まる
print(fruits[0])  # -> Apple

# ⚠️ リストの末尾を取得するPython特有の書き方
print(fruits[-1]) # -> Orange (要素数がわからなくても最後が取れる)

# ❌ IndexError: 存在しないインデックスへのアクセス
# print(fruits[3]) # アプリケーションがクラッシュする原因第1位！


# --- 辞書の基本と実践的な使い方 ---
# APIから取得したJSONは、そのままPythonの辞書として扱われます
user_profile = {
    "id": 101,
    "username": "bob_smith",
    "is_active": True,
    "roles": ["admin", "editor"] # 辞書の中にリストを入れる（頻出パターン）
}

# キーを使ったアクセス
print(user_profile["username"])

# 💡 安全なアクセス方法 (.get)
# 存在しないキーに user_profile["age"] とアクセスするとKeyErrorでクラッシュします。
# 実務では外部データからのアクセスには必ず .get() を使い、デフォルト値を設定します。
age = user_profile.get("age", 20) # ageがなければ20を返す
print(f"年齢: {age}")

💼 Tips：実務で遭遇する現場の勘所

APIのレスポンスなど「外部から受け取ったデータ」を処理する際は、絶対に辞書の直接アクセス（data["key"]）を使わず、data.get("key") を使う癖をつけてください。データ欠損時にアプリ全体が停止（500エラー）するのを防ぐことができます。リストと辞書を組み合わせた「辞書のリスト（[{}, {}, {}]）」はDBのテーブルデータを表す基本形なので、この操作に慣れることが最初の壁になります。

03

条件分岐（If文）

▼

💡 状況に応じてプログラムの動きを変える

なぜIf文が重要なのか？If文（条件分岐）は、プログラムに「意思決定」をさせるための機能です。「ログイン済みならダッシュボードへ、未ログインならトップページへ」「残高不足ならエラーを返す」といったビジネスロジックは、すべてIf文で表現されます。If文のないプログラムは、ただの一本道のスクリプトです。

If-elif-else の基本と、Pythonならではの真偽判定

Pythonはインデント（字下げ）でブロックを表現します。また、暗黙の真偽値（Falsy/Truthy）を活用することで、非常にシンプルで読みやすいコードを書くことができます。

PYTHON 
user_age = 20
has_ticket = True

# 複数の条件を組み合わせる (and / or)
if user_age >= 18 and has_ticket:
    print("入場可能です。")
elif user_age < 18 and has_ticket:
    print("未成年は保護者の同伴が必要です。")
else:
    print("チケットがないため入場できません。")

# --- Python特有の暗黙の真偽判定 (Falsy) ---
# 以下の値はすべて「False」として判定されます。
# [], {}, "", 0, None

user_name = ""
cart_items = []

# ❌ 冗長で素人っぽい書き方
if len(cart_items) == 0:
    print("カートは空です")

# ✅ プロが書くPythonic（Pythonらしい）な書き方
# リストが空であればFalseとなるので、notで反転させます
if not cart_items:
    print("カートは空です (Pythonic)")

if not user_name:
    print("名前が入力されていません")

💼 Tips：実務で遭遇する現場の勘所

実務では「早期リターン（Early Return）」というテクニックを多用します。if文を深くネスト（入れ子）にするのではなく、「エラー条件ならさっさと return や raise で処理を抜ける」ように書くと、メインの処理がスッキリして可読性が爆上がりします（これをガード節とも呼びます）。AIはネストの深いコードを生成しがちなので、「早期リターンで書き直して」と指示するのが効果的です。

04

繰り返し処理（ForループとWhileループ）

▼

💡 面倒な単純作業を一瞬で終わらせる

なぜ繰り返し処理が必要なのか？「10件のデータを処理する」のと「100万件のデータを処理する」のは、プログラムのコード上は全く同じ手間で書くことができます。これがコンピューターの圧倒的なパワーの源泉です。Forループは「決まった回数・データ数だけ繰り返す」場合に使われ、Whileループは「特定の条件を満たすまでずっと繰り返す（例: ユーザーが終了を選ぶまで）」場合に使われます。

Forループと内包表記

PythonのForループは、他言語（C言語の for i=0; i<N など）と異なり、「データのかたまりから1つずつ要素を取り出す」という直感的な動きをします。さらに、リストを一瞬で作る「リスト内包表記」はPython最大の強みです。

PYTHON 
users = ["Alice", "Bob", "Charlie"]

# 1. 基本的なForループ
# usersリストから1つずつ要素を取り出し、userという変数に代入して実行
for user in users:
    print(f"Hello, {user}!")

# 2. 辞書のループ (.items()を使うのが鉄則)
scores = {"Math": 90, "English": 85}
for subject, score in scores.items():
    print(f"{subject}: {score}点")

# --- 🔥 Pythonicな必殺技：リスト内包表記 ---
prices = [100, 200, 300]

# ❌ 冗長な書き方
taxed_prices = []
for p in prices:
    taxed_prices.append(int(p * 1.1))

# ✅ リスト内包表記 (List Comprehension)
# ループを回しながら新しいリストを1行で生成。実行速度も通常のForより速いです。
taxed_prices = [int(p * 1.1) for p in prices]

# さらに条件でフィルタリングも可能（200円以上の商品だけ抽出）
high_prices = [p for p in prices if p >= 200]
print(high_prices) # [200, 300]

💼 Tips：実務で遭遇する現場の勘所

実務で「ループを回しながら、現在のインデックス（何番目か）も知りたい」というケースが頻出します。ここで for i in range(len(users)): のようなC言語風のコードを書くと素人感が出ます。プロは必ず for i, user in enumerate(users): を使います。Pythonでは「いかにループを簡潔に、安全に回すか（内包表記、enumerate、zipなどの活用）」がコード品質の指標になります。

関数の定義と呼び出し

▼

💡 何度も使う処理をひとまとめにして再利用する

なぜ関数に分けるのか？コードが100行、1000行と長くなってくると、「どこで何をしているか」が全く追えなくなります。関数は複雑な処理に「名前」をつけてカプセル化する機能です。また、「仕様変更があったときに、直す箇所が1箇所で済む（保守性が高い）」というのが最大のメリットです。優れた関数は、名前を見るだけで中身を読まなくても何をするか分かるようになっています。

関数の基本（def）と、型ヒント（Type Hints）

Python 3.5以降、引数と戻り値に「型」を注釈できるようになりました（型ヒント）。これは必須ではありませんが、現代の開発（特にFastAPIなど）では事実上の必須スキルです。

PYTHON 
# ❌ 型ヒントなしの古い書き方（引数に何を入れていいか、エディタが推測できない）
def calculate_tax(price, tax_rate=0.1):
    return int(price * (1 + tax_rate))

# ✅ モダンなPythonの書き方（型ヒントとDocstringを完備）
def calculate_tax_modern(price: int, tax_rate: float = 0.1) -> int:
    """
    税込み価格を計算する関数
    
    Args:
        price (int): 税抜き価格
        tax_rate (float): 税率 (デフォルトは0.1=10%)
        
    Returns:
        int: 小数点を切り捨てた税込み価格
    """
    return int(price * (1 + tax_rate))

# 呼び出し方
result1 = calculate_tax_modern(1000)             # デフォルト値 0.1 が使われる
result2 = calculate_tax_modern(1000, tax_rate=0.08) # キーワード引数で指定（可読性が高い）

print(result1) # 1100
print(result2) # 1080

💼 Tips：実務で遭遇する現場の勘所

関数の設計で最も大切なのは「単一責任の原則（1つの関数は1つのことだけをする）」です。「データをDBから取得して、計算して、メールで送信する」という関数を作ってはいけません。「取得」「計算」「送信」の3つの関数に分けるべきです。関数名に and が含まれそうになったら（例: fetch_and_calculate）、それは関数を分割すべきサインです。

文字列と f-string — テキスト処理の基礎体力

▼

💡 実務のコードの半分は「文字列いじり」でできている

なぜ文字列操作を体系的に学ぶ必要があるのか？ ログの整形、CSV の読み書き、API レスポンスの加工、ファイル名の生成——実務の Python コードでは、驚くほど多くの時間を文字列処理に費やします。Python の文字列は「イミュータブル（変更不可）なオブジェクト」であり、豊富なメソッドを持っています。ここを雑に覚えると、後々 + 連結だらけの読みにくいコードを量産することになります。

f-string：現代 Python の標準的な文字列組み立て

Python 3.6 以降の文字列組み立ては f-string 一択です。文字列の前に f を付けると、{} の中に式を直接埋め込めます。% 演算子や str.format() は古いコードを読むために知っておけば十分で、新規コードで使う理由はありません。

Python 
name = "佐藤"
score = 87.5

# f-string：{} の中に変数や式をそのまま書ける
print(f"{name}さんの得点は {score} 点です")

# 書式指定：コロンの後にフォーマットを書く
pi = 3.14159265
print(f"円周率は約 {pi:.2f}")        # 小数点以下2桁 → 3.14
print(f"|{name:>10}|")               # 右寄せ10文字 → |        佐藤|
print(f"{1234567:,}")                 # 3桁区切り → 1,234,567

# = を付けるとデバッグに便利（変数名と値を同時に表示）
print(f"{score=}")                    # score=87.5

覚えるべき頻出メソッドは 10 個だけ

文字列メソッドは 40 以上ありますが、実務で使うのはごく一部です。strip（前後の空白除去）、split（分割）、join（結合）、replace（置換）、startswith / endswith（前方・後方一致）、lower / upper（大文字小文字）、in 演算子（部分一致）。この組み合わせで日常の文字列処理の 9 割はカバーできます。

Python 
line = "  apple, banana , cherry  "

# strip → split → 各要素も strip、が定番パターン
items = [s.strip() for s in line.strip().split(",")]
print(items)                      # ['apple', 'banana', 'cherry']

# join はリストを区切り文字で連結する（区切り文字側のメソッドである点に注意）
csv_line = ",".join(items)
print(csv_line)                   # apple,banana,cherry

# 部分一致の判定は in が最速で読みやすい
if "banana" in csv_line:
    print("バナナが含まれています")

💼 Tips：ループ内の + 連結はアンチパターン

文字列はイミュータブルなので、s += "x" は毎回新しい文字列オブジェクトを生成します。ループで大量に連結する場合は、リストに append して最後に "".join(parts) でまとめるのが定石です。数千件を超える連結では速度差が体感できるレベルになります。

データ構造の使い分け、例外処理、ファイル操作、アンチパターン回避など、「動くコード」から「壊れないコード」へレベルアップします。最後に FastAPI で実際のバックエンドを構築します。

データ構造の極意（辞書とタプルの使い分け）

▼

💡 リストだけでは対応できない実務のデータ設計

なぜ複数のデータ構造が存在するのか？リストは便利ですが「どこからでも要素を変更できる」という性質があります。これが原因で「知らない間に別の関数でデータが書き換えられていた」というバグが頻発します。「絶対に書き換えられないデータ構造（タプル）」や「一意の要素を超高速に検索できるデータ構造（集合: set）」を適材適所で使い分けることで、バグを未然に防ぐ堅牢な設計が可能になります。

ミュータブル（変更可能）とイミュータブル（変更不可）

リストや辞書は「ミュータブル」ですが、タプルや文字列は「イミュータブル」です。重要な定数や、辞書のキーにはイミュータブルなオブジェクトしか使えません。

PYTHON 
# --- タプル (Tuple) の基本 ---
# () で作成。一度作ったら要素の追加や変更ができない
db_config = ("localhost", 5432, "admin", "password")

# db_config[1] = 8080 # TypeError: タプルは変更不可！
# 💡 意図せず設定が書き換わるのを防ぐため、定数的な設定値はタプルにします。

# アンパック代入（実務で多用します）
host, port, user, pwd = db_config
print(f"接続先: {host}:{port}")

# --- 集合 (Set) の基本 ---
# {} で作成。重複を許さず、順番を持たない
visited_users = {"Alice", "Bob", "Alice"}
print(visited_users) # -> {'Bob', 'Alice'} （重複は自動で消える）

# 💡 リストからの重複排除に一瞬で使える
raw_data = [1, 2, 2, 3, 3, 3]
unique_data = list(set(raw_data)) # -> [1, 2, 3]

# 💡 要素の存在確認 (in演算子) は、リストよりSetの方が圧倒的に高速(O(1))です
if "Alice" in visited_users:
    print("既に訪問済みです")

💼 Tips：実務で遭遇する現場の勘所

「数万件のデータの中から、特定のIDが含まれているか確認する」という処理を書く際、リストに対して if user_id in user_list: とすると、先頭から1つずつ探すためとんでもなく処理が遅くなります。検索対象を事前に user_set = set(user_list) のように集合（Set）に変換しておくだけで、検索速度が数千倍になることがあります。データ構造の選択は、そのままアルゴリズムのパフォーマンスに直結します。

堅牢なコードの要（try-exceptによる例外処理）

▼

💡 AIのコードによく欠けている「エラー発生時の備え」

なぜ例外処理が必要なのか？API通信の失敗、ファイルの読み込みエラー、ユーザーの不正な入力など、現実のプログラムは「予期せぬエラー（例外）」だらけです。例外処理を書かないと、1つのエラーでシステム全体が停止（クラッシュ）してしまいます。AIは正常に動く前提（Happy Path）のコードを生成しがちなため、「もしここでエラーが起きたらどうリカバリーするか」を設計するのが人間の重要な役割です。

try-except-finally ブロックの基本

危険な処理を try ブロックで囲み、エラー時の対応を except で定義します。finally はエラーの有無に関わらず最後に必ず実行したい後処理（ファイルのクローズなど）に使います。

PYTHON 
def safe_divide(a, b):
    try:
        # エラーが起きるかもしれない処理
        result = a / b
        print(f"計算結果: {result}")
        return result
        
    except ZeroDivisionError:
        # 特定のエラー（0での割り算）をピンポイントでキャッチ
        print("🚨 エラー: 0で割ることはできません。")
        return None
        
    except TypeError as e:
        # 例外オブジェクト(e)から詳細なエラー情報を取得
        print(f"🚨 型エラー: 数値を入力してください。詳細: {e}")
        return None
        
    except Exception as e:
        # ❌ 全てのエラーを握りつぶす「ベアExcept」
        # 開発中のバグまで隠してしまうので多用は厳禁ですが、
        # アプリ全体を落とさないための最後の砦として使います。
        print(f"⚠️ 予期せぬエラーが発生しました: {e}")
        return None
        
    finally:
        # 成功しても失敗しても必ず実行される（DBの切断などに使用）
        print("💡 割り算の処理が終了しました。\n")

safe_divide(10, 2)   # 正常系
safe_divide(10, 0)   # 異常系 (ZeroDivisionError)
safe_divide(10, "a") # 異常系 (TypeError)

💼 Tips：実務で遭遇する現場の勘所

絶対にやってはいけないアンチパターンが「except Exception: pass（全てのエラーを無視して何もしない）」です。これをやると、本当に致命的なバグが起きたときにログすら残らず、原因究明が不可能になります。実務では except の中で必ずロガー（logging.error など）を使ってエラー内容をファイルや監視システムに記録します。

ファイル操作と with 文 — コンテキストマネージャの流儀

▼

💡 「開いたら必ず閉じる」を言語機能で保証する

なぜ with 文が必要なのか？ ファイルを open() したまま閉じ忘れると、OS のファイルハンドルを浪費し、Windows では「別のプロセスが使用中」エラーの原因になります。厄介なのは、close() を書いていても途中で例外が発生すると実行されないことです。with 文（コンテキストマネージャ）は、ブロックを抜ける際に——正常終了でも例外発生でも——後始末を必ず実行することを言語レベルで保証します。

基本形：読み書きは with + encoding 指定

テキストファイルを扱うときは必ず encoding="utf-8" を明示します。省略すると OS 依存（Windows では CP932）になり、「手元では動くのにサーバーで文字化けする」事故の温床になります。

Python 
# 書き込み："w" は上書き、"a" は追記
with open("report.txt", "w", encoding="utf-8") as f:
    f.write("売上レポート\n")

ミュータブルなデフォルト引数の罠（超頻出）

関数のデフォルト引数に [] や {} を使うと、関数を呼び出すたびに「同じリスト」が使い回されてしまい、データがどんどん蓄積していくという恐ろしいバグを引き起こします。

PYTHON 
# ❌ 恐ろしいアンチパターン
# Pythonでは、関数のデフォルト引数は「関数が定義された時に1回だけ」評価されます。
def add_item_bad(item, target_list=[]):
    target_list.append(item)
    return target_list

print(add_item_bad("Apple"))  # ["Apple"]
print(add_item_bad("Banana")) # ["Apple", "Banana"] (空のリストを期待したのに前回のが残っている！)

# ✅ 正解パターン (Noneを使う)
def add_item_good(item, target_list=None):
    if target_list is None:
        target_list = [] # 呼び出されるたびに新しいリストを作る
    target_list.append(item)
    return target_list

print(add_item_good("Apple"))  # ["Apple"]
print(add_item_good("Banana")) # ["Banana"] (期待通り！)

# --- もう一つの罠：ループ中の破壊的変更 ---
# ❌ ループしているリスト自身の要素を削除してはいけない
nums = [1, 2, 2, 3]
for n in nums:
    if n == 2:
        nums.remove(n) 
# 結果: [1, 2, 3] (2が1つ残ってしまう。インデックスが狂うため)

# ✅ 新しいリストを作る（内包表記がベスト）
clean_nums = [n for n in nums if n != 2]
# 結果: [1, 3]

💼 Tips：実務で遭遇する現場の勘所

実務では、この「ミュータブルの罠」を避けるために、関数内に入ってきた引数（リストや辞書など）を直接書き換える「破壊的変更」を極力避けます。代わりに new_list = list(old_list) のようにコピーを作ってから操作するか、内包表記で新しいオブジェクトを返すのがモダンな設計です（イミュータブルなプログラミングパラダイム）。

デコレータの魔法

▼

💡 関数の外側から機能を「トッピング」する技術

なぜデコレータが必要なのか？「全てのAPIリクエストの実行時間を計測したい」「管理者権限があるかチェックしたい」といった要件があったとします。全ての関数の中に計測コードやチェックコードを書くと、本質的なロジックが埋もれてしまいます。デコレータ（@xxx）を使えば、元の関数のコードを1文字も変えずに、関数の外側から「処理の前後」に共通機能をガバッと被せる（ラップする）ことができます。

デコレータの仕組みと作り方

デコレータの正体は「関数を引数として受け取り、新しい関数を返す関数（高階関数）」です。

PYTHON 
import time
from functools import wraps

# --- 自作デコレータの定義 ---
def time_logger(func):
    @wraps(func) # 元の関数名(docstringなど)を保持するための必須のお作法
    def wrapper(*args, **kwargs):
        start = time.time()             # 実行「前」の処理
        
        result = func(*args, **kwargs)  # 元の関数をここで実行！
        
        end = time.time()               # 実行「後」の処理
        print(f"⏱️ [{func.__name__}] 実行時間: {end - start:.4f}秒")
        
        return result                   # 元の関数の結果をそのまま返す
    return wrapper

# --- デコレータの使い方 ---
# @をつけるだけで、関数に「時間計測」の機能がトッピングされる
@time_logger
def heavy_process():
    """1秒かかる重い処理"""
    time.sleep(1)
    return "処理完了"

# 呼び出し側は普通に呼ぶだけ
print(heavy_process())
# 出力:
# ⏱️ [heavy_process] 実行時間: 1.0012秒
# 処理完了

💼 Tips：実務で遭遇する現場の勘所

Flaskの @app.route や FastAPIの @app.get もすべてこのデコレータの応用です。実務では「ログイン必須チェック（@login_required）」や「失敗時の自動リトライ（@retry(3)）」など、横断的関心事（Cross-cutting concerns）を分離するために多用されます。自分でゼロからデコレータを書く機会は少ないですが、フレームワークを使う上で「裏で何が起きているか」を理解しておくことはシニアエンジニアへの第一歩です。

ジェネレータとメモリ節約

▼

💡 大量のデータを少しずつ処理するエコな仕組み

なぜジェネレータが必要なのか？「100GBのCSVファイルを読み込んで処理する」コードを書いたとします。リストで一度に読み込もうとすると、PCのメモリ（RAM）がパンクしてクラッシュします（OOMエラー）。ジェネレータ（yield）を使えば、「1行読み込んで処理し、終わったら次の1行を読む」という一時停止と再開が可能になり、どれだけデータが大きくても一定の少ないメモリで処理し続けることができます。

yield キーワードの魔法

return は値を返して関数を完全に終了させますが、yield は値を返したあとに「関数の状態を保持したまま一時停止」します。

PYTHON 
# --- ジェネレータ関数の定義 ---
def read_huge_file(file_path):
    # ファイル全体を読み込まず、1行ずつ返すエコな関数
    with open(file_path, "r", encoding="utf-8") as f:
        for line in f:
            yield line.strip()  # 値を返して一時停止する！

# --- 呼び出し側 ---
# 100GBのファイルでもメモリ使用量はほぼゼロ
# for line in read_huge_file("huge_data.csv"):
#     print(line)

# --- ジェネレータの動きを実感する ---
def countdown(num):
    print("🚀 カウントダウン開始！")
    while num > 0:
        yield num  # ここで一時停止し、呼び出し元に値を返す
        num -= 1

gen = countdown(3)
# next() を呼ぶたびに、yieldの次の行から再開される
print(next(gen)) # 🚀 カウントダウン開始！ -> 3
print(next(gen)) # 2
print(next(gen)) # 1
# print(next(gen)) # StopIterationエラー（終了）

💼 Tips：実務で遭遇する現場の勘所

ChatGPTなどの生成AIアプリを作るとき、回答が「1文字ずつパラパラと流れてくる（ストリーミング）」のを見たことがあると思います。あれはバックエンドでまさにこの yield を使ったジェネレータが動いています。「データが全て揃うのを待たずに、生成できた分からクライアントに返す」ことで、ユーザーの体感待ち時間をゼロにする究極のUXハックです。

型ヒントと dataclass — モダン Python の作法

▼

💡 型ヒントは「実行前にバグを見つける」ための注釈

なぜ動的型付けの Python に型を書くのか？ Python は実行するまで型の間違いに気づけません。get_user() が dict を返すのか User オブジェクトを返すのか、コードを追わないと分からない——これが大規模開発のボトルネックになります。型ヒントを書くと、エディタの補完が効き、mypy や pyright といった型チェッカーが実行せずにバグを検出してくれます。FastAPI や pydantic のように、型ヒントを実行時の機能（バリデーション）として活用するフレームワークも主流になりました。

基本の書き方：変数・関数・ジェネリクス

Python 3.9 以降は list[int] のように組み込み型をそのままジェネリクスに使えます（古い List[int] は不要）。「None かもしれない」値は int | None と書きます（3.10+）。

Python 
def find_user(user_id: int) -> dict[str, str] | None:
    """id が見つかれば dict、なければ None を返す"""
    users = {1: {"name": "田中", "role": "admin"}}
    return users.get(user_id)

# 型チェッカーは「None の可能性」を見逃さない
user = find_user(1)
# print(user["name"])        # mypy: user が None かもしれないと警告
if user is not None:
    print(user["name"])       # OK：None を除外したのでアクセス可能

scores: list[int] = [90, 75, 88]
config: dict[str, bool] = {"debug": True}

dataclass：データの入れ物クラスを 1 行で

「フィールドを持つだけのクラス」を手書きすると、__init__・__repr__・__eq__ の定型コードが大量に必要です。@dataclass デコレータはこれらを型ヒントから自動生成します。dict と違ってフィールド名のタイポが即エラーになり、エディタ補完も効くため、構造化データの受け渡しは dict より dataclass が推奨されます。

Python 
from dataclasses import dataclass, field

@dataclass
class Task:
    title: str
    done: bool = False                       # デフォルト値
    tags: list[str] = field(default_factory=list)  # 可変デフォルトは factory で

t1 = Task("牛乳を買う")
t2 = Task("牛乳を買う")
print(t1)            # Task(title='牛乳を買う', done=False, tags=[]) ← repr が自動生成
print(t1 == t2)      # True ← 値ベースの比較も自動生成

# frozen=True にするとイミュータブルになり、辞書のキーにも使える
@dataclass(frozen=True)
class Point:
    x: int
    y: int

💼 Tips：まず「関数の境界」から型を付ける

既存コードに型を後付けするときは、すべてに付けようとせず、公開関数の引数と戻り値から始めるのが定石です。関数の境界に型があれば、内部の型はほとんど推論で補完されます。CI に mypy --strict を段階的に導入すれば、型のない新規コードの混入も防げます。

🔍 一行ずつ解説

① from __future__ import annotations — Python 3.7以降で使える機能で、型ヒントの評価を遅延させます。これにより str | None のような新しい記法を古いバージョンでも使えるようにしています。OSSでは後方互換性が命なので、このインポートは非常に一般的です。

③ import typing as t — typing モジュールを t と短く別名をつけてインポートしています。型ヒントは頻繁に書くので、タイピング量を減らすためのOSSの慣習です。

④ defaultdict — 通常の dict は存在しないキーにアクセスすると KeyError が出ますが、defaultdict は自動でデフォルト値を生成します。Flaskでは「まだ登録されていないBlueprint名のキー」にアクセスされた時にも安全に動作するよう使われています。

⑤ update_wrapper — デコレータを適用すると元の関数名やdocstringが消えてしまいます。update_wrapper はそれを防ぐためのもので、Flask の @app.route の内部で使われています。

⑥ クラス変数でのデフォルト値 — _static_folder = None のようにクラス直下で定義しています。アンダースコア始まりは「外部から直接アクセスしないでね」という慣習（プライベート変数の規約）です。

⑧ 全引数に型ヒント — str | os.PathLike[str] | None のように、引数が取りうる型を全て明示しています。os.PathLike は pathlib.Path などのパスオブジェクトも受け入れることを意味しています。

⑬ ネストされた defaultdict — defaultdict(lambda: defaultdict(dict)) は「辞書の辞書の辞書」を自動生成する構造です。エラーハンドラをスコープ（全体 or Blueprint単位）× HTTPステータスコード × 例外の種類の3階層で管理する設計です。

📖 ② Requests — HTTP リクエストの中核関数

概要: requests.get() を呼んだとき、内部で何が起きているかの全貌です。わずか数行で「セッション管理」「リソースの確実な解放」「関数の委譲」という重要パターンが凝縮されています。

from . import sessions                        # ← ① 同パッケージの sessions モジュール
from .models import Response                  # ← ② レスポンスの型

def request(method, url, **kwargs):           # ← ③ 全HTTPメソッドの共通関数
    """Constructs and sends a Request.

    :param method: GET, POST, PUT, DELETE など
    :param url: リクエスト先のURL
    :param kwargs: headers, params, data, json など全てのオプション
    """
    with sessions.Session() as session:       # ← ④ with文でセッションを自動管理
        return session.request(               # ← ⑤ 実際の処理はSession に委譲
            method=method, url=url, **kwargs
        )

def get(url, params=None, **kwargs):          # ← ⑥ ユーザーが直接呼ぶ関数
    """Sends a GET request."""
    return request("GET", url,                # ← ⑦ request() に委譲するだけ
                   params=params, **kwargs)

def post(url, data=None, json=None, **kwargs):
    """Sends a POST request."""
    return request("POST", url,               # ← ⑧ POST も同じパターン
                   data=data, json=json, **kwargs)

def put(url, data=None, **kwargs):
    return request("PUT", url, data=data, **kwargs)

def delete(url, **kwargs):
    return request("DELETE", url, **kwargs)

def head(url, **kwargs):
    kwargs.setdefault("allow_redirects", False)  # ← ⑨ HEAD特有のデフォルト設定
    return request("HEAD", url, **kwargs)

def patch(url, data=None, **kwargs):
    return request("PATCH", url, data=data, **kwargs)

🔍 一行ずつ解説

③ **kwargs — 「キーワード引数の辞書」です。headers, timeout, cookies など、任意のオプションを全て受け取れるようにしています。ユーザーが requests.get(url, timeout=5) と呼ぶと、kwargs = {"timeout": 5} として渡されます。

④ with sessions.Session() as session: — これがこのファイルの最重要ポイントです。コンテキストマネージャ（with文）により、リクエスト完了後にTCP接続が確実にクローズされます。try-finally と同等ですが、遥かに読みやすいコードになっています。

⑤ session.request() への委譲 — request() 関数自身は何も処理をしていません。全ての実際の処理（DNS解決、接続確立、データ送受信）は Session クラスに委譲しています。これが単一責任の原則（SRP）の実践例です。

⑥⑦ get() は request("GET", ...) を呼ぶだけ — get(), post(), put() などは全て request() のラッパーです。「ユーザーにとって使いやすいインターフェース」を提供しつつ、内部的にはコードの重複を避けています。これがファサードパターンです。

⑨ kwargs.setdefault() — head() メソッドだけ、デフォルトでリダイレクトを追跡しないよう設定しています。setdefault は「まだそのキーが設定されていなければデフォルト値を入れる」メソッドで、ユーザーが明示的に指定した場合はそちらが優先されます。

💡 このコードから学べるプロの設計パターン

1. ファサードパターン: 複雑な内部処理をシンプルなインターフェースで覆い隠す。requests.get(url) の裏では DNS解決、TCP接続、SSL/TLSハンドシェイク、ヘッダー構築、レスポンス解析が全自動で行われています。

2. DRY原則: get(), post() 等は全て request() に委譲。HTTPメソッドごとにコードを書かないことで、バグの混入を防いでいます。

3. コンテキストマネージャ: リソース管理（接続のクローズ）を with 文で自動化。これを忘れると接続リークが起きます。

def command(name=None, cls=None, **attrs):    # ← ③ デコレータファクトリ
    """@click.command() デコレータの実装"""
    if cls is None:
        cls = Command                         # ← ④ デフォルトは Command クラス

    def decorator(func):                      # ← ⑤ 実際のデコレータ関数
        cmd = cls(                            # ← ⑥ 関数を Command オブジェクトに変換
            name=name or func.__name__.lower().replace("_", "-"),
            callback=func,                    # ← ⑦ 元の関数をコールバックとして保存
            params=_params_from_func(func),   # ← ⑧ 関数の引数から自動でパラメータ生成
            **attrs,
        )
        cmd.__doc__ = func.__doc__            # ← ⑨ docstring を引き継ぐ
        return cmd                            # ← ⑩ 関数の代わりに Command を返す
    return decorator                          # ← ⑪ デコレータファクトリ → デコレータを返す

def option(*param_decls, **attrs):            # ← ⑫ @click.option() の実装
    """オプション引数（--verbose 等）をコマンドに追加する"""
    def decorator(func):
        # ↓ ⑬ 関数に __click_params__ 属性を追加
        if not hasattr(func, "__click_params__"):
            func.__click_params__ = []
        # ↓ ⑭ Option オブジェクトをリストに積む
        func.__click_params__.append(Option(param_decls, **attrs))
        return func                           # ← ⑮ 関数自体はそのまま返す
    return decorator

③ デコレータファクトリ — @click.command() は「括弧付き」で呼ばれています。これは command() が直接デコレータではなく、デコレータを「生成して返す」ファクトリ関数だということです。この2段階構造により、@click.command(name="greet") のようにオプションを渡せるようになります。

⑥⑩ 関数を Command オブジェクトに変換 — ここが Click の核心です。@click.command() を付けた関数は、関数ではなく Command オブジェクトに差し替えられます。つまり「デコレータが関数を全く別のモノに変えてしまう」のです。これにより、ヘルプ表示や引数のバリデーションなどの機能が自動的に追加されます。

⑧ _params_from_func(func) — 関数の引数定義（名前、デフォルト値、型ヒント）を自動的に解析して、CLIのパラメータに変換します。inspect モジュールを使って関数のシグネチャを読み取る高度なテクニックです。

⑬⑭ 関数にカスタム属性を追加 — func.__click_params__ = [] という行は、Pythonの「関数もオブジェクト」という特性を活用しています。関数に独自の属性を自由に追加でき、後から別のコードがその情報を読み取れます。

⑤ 抽象基底クラス（Abstract Base Class）
AbstractMessage は直接テーブルにならず、Message（通常メッセージ）と ArchivedMessage（削除後の一時保管）の2つのテーブルが同じフィールドを持つようにしています。これがDRY原則（Don't Repeat Yourself）の実践です。チャットサービスでは「メッセージの削除」≠「本当にデータを消す」であり、一定期間アーカイブに保持してから完全削除するのが一般的です。

⑨ マルチテナント設計（realm フィールド）
Zulipは1つのサーバーで複数の組織（会社やチーム）をホストできます。realm（組織）を全てのデータに紐づけることで、「A社のデータがB社に見える」事故を構造的に防止しています。SaaSを設計する際の最重要パターンの一つです。

⑬ DM_TOPIC の工夫
これは非常に興味深い設計判断です。DMにはトピック名がありませんが、DBには空文字 "" を入れていました。しかし、空文字をトピック名として使い始めた際にPostgreSQLのクエリプランナーがDM全体をスキャンしてしまうパフォーマンス問題が発生。解決策として、印刷不可能文字 \x07（BEL文字）をDM専用のマーカーにしました。こういった「現実の問題に対する泥臭い解決策」がOSSコードを読む醍醐味です。

⑯⑰ 生テキストとレンダリング済みHTMLの二重管理
MarkdownのレンダリングはCPU負荷が高い処理です。毎回レンダリングするのではなく、送信時にHTMLに変換して保存し、表示時はHTMLをそのまま返します。ただし null=True にしているのは、レンダリングロジックを変更した際に再レンダリングが必要になるケースに備えてのことです。

㉒ Boolean フラグの設計
has_attachment, has_image, has_link という3つのフラグは、「画像付きメッセージだけを検索」「リンク付きメッセージだけを表示」のようなフィルタリング機能のために存在します。全てに db_index=True がついているのは、検索パフォーマンスのためです。

import time                                      # ← ② 高精度タイマー
from collections.abc import Callable, MutableMapping

from django.conf import settings                 # ← ③ settings.py の設定値にアクセス
from django.http import HttpRequest, HttpResponse

# ↓ ④ 専用のロガーを2つ定義（用途別にログを分離）
logger = logging.getLogger("zulip.requests")     # 通常のリクエストログ
slow_query_logger = logging.getLogger("zulip.slow_queries")  # 遅いリクエスト専用

def record_request_stop_data(log_data: MutableMapping[str, Any]) -> None:
    """リクエスト処理終了時のメトリクスを記録する"""
    log_data["time_stopped"] = time.time()       # ← ⑤ 処理終了時刻

    # ↓ ⑥ キャッシュ（Redis）のアクセス統計を記録
    log_data["remote_cache_time_stopped"] = get_remote_cache_time()
    log_data["remote_cache_requests_stopped"] = get_remote_cache_requests()

    # ↓ ⑦ Markdownレンダリングの統計を記録
    log_data["markdown_time_stopped"] = get_markdown_time()
    log_data["markdown_requests_stopped"] = get_markdown_requests()

    # ↓ ⑧ 全リクエストプロファイリングが有効なら停止
    if settings.PROFILE_ALL_REQUESTS:
        log_data["prof"].disable()

def write_log_line(log_data, path, method, remote_ip, ...) -> None:
    """リクエストの完全なログ行を出力する（簡略版）"""
    # ↓ ⑨ 処理時間の計算（ミリ秒単位）
    time_delta = log_data["time_stopped"] - log_data["time_started"]

    # ↓ ⑩ キャッシュの統計: 何回アクセスし、合計何秒かかったか
    cache_hits = (log_data["remote_cache_requests_stopped"]
                 - log_data["remote_cache_requests_started"])
    cache_time = (log_data["remote_cache_time_stopped"]
                 - log_data["remote_cache_time_started"])

    # ↓ ⑪ Markdownレンダリングの統計
    md_count = (log_data["markdown_requests_stopped"]
               - log_data["markdown_requests_started"])
    md_time = (log_data["markdown_time_stopped"]
              - log_data["markdown_time_started"])

    # ↓ ⑫ 構造化ログの出力（monitoring / alerting で活用）
    logger.info(
        "%-5s %d %s (db: %dq/%dms, cache: %d/%dms, md: %d/%dms, total: %dms)",
        method,               # GET, POST 等
        status_code,          # 200, 404, 500 等
        path,                 # /api/v1/messages 等
        db_queries, db_time,  # DBクエリ回数と時間
        cache_hits, cache_time * 1000,
        md_count, md_time * 1000,
        time_delta * 1000,    # 全体の処理時間
    )

    # ↓ ⑬ 遅いリクエストを別のログファイルに記録
    if time_delta > 5:  # 5秒以上かかったリクエスト
        slow_query_logger.info("SLOW: %s %s %dms", method, path, time_delta * 1000)

🔍 設計の重要ポイント解説

④ ロガーの分離
zulip.requests（通常ログ）と zulip.slow_queries（遅延ログ）を分けることで、ログの出力先やアラート条件を別々に設定できます。本番サービスでは「全リクエストは info レベルで記録し、5秒超えたものだけ Slack にアラートを飛ばす」といった設定が一般的です。

⑥⑦ メトリクスの「開始時」と「終了時」の差分
キャッシュやMarkdownの統計を「リクエスト処理開始時」と「終了時」の差分で計算しています。これは、Pythonプロセスがスレッドプールで複数リクエストを並列処理するため、グローバルな累計値では正確な計測ができないからです。「このリクエストだけで何回キャッシュにアクセスしたか」を知るには差分計算が必要なのです。

⑫ 構造化ログ
ログ出力に db: %dq/%dms, cache: %d/%dms のような構造化フォーマットを使っています。これにより、ログ集約ツール（Datadog, Grafana, ELK Stack 等）で「DBクエリが100回を超えたリクエスト」「キャッシュ時間が処理時間の50%以上のリクエスト」のような高度な分析・アラート設定が可能になります。

    recipient_type_name: str,
) -> UserProfile:
    sender_email = sender.strip().lower()          # ← ⑭ 入力のサニタイズ（前後空白除去+小文字化）
    referenced_users = {sender_email}              # ← ⑮ set でユニークなユーザー集合を管理

    if recipient_type_name == "private":
        referenced_users.update(                   # ← ⑯ DM の場合は受信者も追加
            email.lower() for email in recipients  # ← ⑰ ジェネレータ式で変換
        )

    # ↓ ⑱ クライアント種別に応じて検証関数を切り替え（ストラテジーパターン）
    if client.name == "irc_mirror":
        user_check = same_realm_irc_user
        fullname_function = compute_irc_user_fullname
    elif client.name in ("jabber_mirror", "JabberMirror"):
        user_check = same_realm_jabber_user
        fullname_function = compute_jabber_user_fullname
    else:
        raise InvalidMirrorInputError(             # ← ⑲ 不明なクライアントは明示的にエラー
            "Unrecognized mirroring client"
        )

    # ↓ ⑳ 全ユーザーが同一組織に属しているか検証
    for email in referenced_users:
        if not user_check(user_profile, email):
            raise InvalidMirrorInputError(
                "At least one user cannot be mirrored"
            )

🔍 設計の重要ポイント解説

⑦ @typed_endpoint デコレータ
Zulip 独自のデコレータで、HTTP リクエストのパラメータを自動的に型チェック・バリデーションします。command: str と書くだけで、パラメータが未指定なら 400 エラーを自動返却します。Django REST Framework の Serializer と似た機能を、デコレータだけで実現している簡潔な設計です。

⑭ 入力のサニタイズ
sender.strip().lower() — メールアドレスの前後の空白を除去し、小文字に統一しています。「Alice@Example.com」と「alice@example.com」を同一視するためです。ユーザー入力を信用せず必ず正規化するのは、セキュリティの基本です。

⑱ ストラテジーパターン
IRC ミラーと Jabber ミラーで検証ロジックが異なりますが、if 文で関数を変数に代入して同じインターフェースで呼び出しています。新しいミラータイプを追加する場合も、elif を1つ追加するだけで済む拡張しやすい設計です。

📖 ④ データベースインデックス設計 — 大規模サービスの速度の秘密

ファイル: zerver/models/messages.py の Message.Meta
概要: 数千万件のメッセージを高速に検索するためのインデックス設計です。「なぜこのカラムの組み合わせでインデックスを作るのか」という設計意図がコメントに明記されており、DB設計の教材として最高です。

# 出典: zulip/zulip — zerver/models/messages.py（Message クラスの Meta 定義）

class Message(AbstractMessage):
    """通常のメッセージ（削除されていない活きたメッセージ）"""

    search_tsvector = SearchVectorField(null=True)  # ← ① 全文検索用ベクトル

    # ↓ ② 関連テーブルをデフォルトで JOIN して取得する設定
    DEFAULT_SELECT_RELATED = ["sender", "realm", "recipient", "sending_client"]

    class Meta:
        indexes = [
            # ↓ ③ 全文検索用の GIN インデックス
            GinIndex(
                "search_tsvector",
                fastupdate=False,                    # ← ④ 更新速度より検索速度を優先
                name="zerver_message_search_tsvector"
            ),

            # ↓ ⑤ チャンネル内メッセージの一括操作用
            # 「メッセージの移動」「既読処理」で使う
            models.Index(
                "realm_id", "recipient_id", "id",    # ← ⑥ 3カラム複合インデックス
                name="zerver_message_realm_recipient_id",
            ),

            # ↓ ⑦ ダイジェストメール生成 & メッセージアーカイブ用
            # チャンネルごと・日時順にメッセージを取得
            models.Index(
                "realm_id", "recipient_id", "date_sent",
                name="zerver_message_realm_recipient_date_sent",
            ),

            # ↓ ⑧ エクスポート & ユーザー削除用
            # 特定ユーザーが送信したメッセージをフィルタリング
            models.Index(
                "realm_id", "sender_id", "recipient_id",
                name="zerver_message_realm_sender_recipient",
            ),
        ]

🔍 設計の重要ポイント解説

③④ 全文検索（GinIndex）
PostgreSQLの GIN（Generalized Inverted Index）インデックスを使って、メッセージの全文検索を実現しています。fastupdate=False は「新しいメッセージの挿入速度を多少犠牲にしてでも、検索速度を最優先する」設定です。チャットサービスでは検索のほうが頻繁に使われるため、この判断は合理的です。

⑥ 複合インデックスの設計意図
("realm_id", "recipient_id", "id") の3カラム複合インデックスは、「組織X のチャンネルY のメッセージを ID順に取得する」クエリを超高速にします。最後に id を入れることで、結果の順序付けがインデックスだけで完結し、ソート処理が不要になります（これを「カバリングインデックス」と呼びます）。

⑦⑧ ユースケース駆動のインデックス設計
Zulipのインデックスは全て「このインデックスはどの機能で使われるか」がコメントで明示されています。「とりあえずインデックスを張る」のではなく、「この機能のこのクエリを速くするために張る」という目的意識の明確さが、プロのDB設計です。

💼 Zulipから学べる設計原則まとめ

1. コメントで「なぜ」を書く: Zulipのコードには「何をしているか」だけでなく「なぜそうしたか」が必ずコメントされています。DM_TOPICの \x07 が良い例です。

2. 抽象化で重複を排除: AbstractMessage → Message / ArchivedMessage の継承構造は、DRY原則の教科書的実装です。

3. 計測しないものは改善できない: 全リクエストの処理時間・DB呼び出し・キャッシュヒットを自動計測するミドルウェアは、パフォーマンス問題の早期発見に不可欠です。

💼 Tips：実務で遭遇する現場の勘所

Zulipのコードから学ぶ最大の教訓は「コメントには『何をしているか』ではなく『なぜそうしたか（Why）』を書く」という点です。泥臭いハックがコメントに書かれていることで、未来の開発者が「無駄な処理だ」と勘違いして消してしまう本番障害を防ぐことができます。

▼

💡 FastAPI = Flask の簡潔さ + 自動ドキュメント + 非同期処理

なぜ重要なのか？ FastAPIは「Flaskのようにシンプルに書ける」のに「自動でSwagger UI（API仕様書）が生成される」「型チェックによりバグを未然に防ぐ」「非同期処理(async/await)対応でNode.js並みに速い」という全てを兼ね備えています。現在のAI・データサイエンス領域のバックエンド開発における事実上のグローバルスタンダードです。

FastAPI とは？

FastAPI は、Python 3.7 以降の型ヒント（Type Hints）を最大限に活用したモダンな Web API フレームワークです。Flask と同じくらいシンプルに書けて、Django REST Framework 並みの機能（バリデーション、シリアライゼーション、認証）を自動で提供します。

主な特長:

• 高速: Starlette + Uvicorn ベースで、Node.js や Go に匹敵するパフォーマンス
• 型安全: Python の型ヒントからバリデーション・ドキュメントを自動生成
• 自動ドキュメント: Swagger UI（/docs）と ReDoc（/redoc）が標準搭載
• 非同期対応: async/await をネイティブサポート
• GitHub ★80k+、Netflix、Uber、Microsoft が採用

📖 ③ 非同期処理（async/await）— 高速な並行処理

FastAPI は Python の async/await をネイティブサポートしており、I/O バウンド（DB読み書き、外部API呼び出し）の処理を効率的に並行実行できます。

PYTHON — 非同期 FastAPI 
import httpx                          # ← 非同期対応の HTTP クライアント
from fastapi import FastAPI

app = FastAPI()

# ① 同期関数 — 1リクエストずつ順番に処理
@app.get("/sync")
def sync_endpoint():
    # このリクエストの処理中、他のリクエストはブロックされる
    result = slow_database_query()    # 2秒かかる
    return {"data": result}

# ② 非同期関数 — 待ち時間中に他のリクエストを処理可能
@app.get("/async")
async def async_endpoint():          # ← async キーワードを追加
    async with httpx.AsyncClient() as client:
        # await 中に他のリクエストを処理できる
        response = await client.get(  # ← await で非同期に待機
            "https://api.example.com/data"
        )
    return response.json()

# ③ 複数の外部APIを並行呼び出し
import asyncio

@app.get("/dashboard")
async def get_dashboard():
    async with httpx.AsyncClient() as client:
        # ↓ 3つのAPIを「同時に」呼び出す（順番に呼ぶと3倍遅い）
        user_task = client.get("https://api.example.com/user")
        orders_task = client.get("https://api.example.com/orders")
        stats_task = client.get("https://api.example.com/stats")

        user, orders, stats = await asyncio.gather(
            user_task, orders_task, stats_task  # ← 全部まとめて並行実行
        )

    return {
        "user": user.json(),
        "orders": orders.json(),
        "stats": stats.json(),
    }
# 同期: 1秒 + 1秒 + 1秒 = 3秒
# 非同期: max(1秒, 1秒, 1秒) = 1秒 ← 3倍速い！

📖 ④ 依存性注入（Dependency Injection）— 認証・DB接続の共通化

FastAPI の依存性注入は、「このエンドポイントには認証が必要」「DB接続を自動で開閉する」といった共通処理を簡潔に定義できる仕組みです。

PYTHON — 依存性注入 
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

security = HTTPBearer()              # ← Bearer トークン認証スキーム

# ① 認証の依存関数 — トークンを検証してユーザーを返す
async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security)
):                                   # ← Depends() で自動注入
    token = credentials.credentials
    user = await verify_token(token)  # トークンからユーザーを特定
    if not user:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="無効なトークンです",
        )
    return user

# ② DB セッションの依存関数 — 接続の開閉を自動管理
async def get_db():
    db = SessionLocal()              # DB接続を開く
    try:
        yield db                     # ← yield でエンドポイントに渡す
    finally:
        db.close()                   # ← リクエスト完了後に必ず閉じる

# ③ エンドポイントで依存関数を使う
@app.get("/me")
async def get_my_profile(
    user = Depends(get_current_user),  # ← 認証済みユーザーを自動注入
    db = Depends(get_db),              # ← DB接続を自動注入
):
    profile = await db.get(User, user.id)
    return profile

# ④ 管理者権限のチェック（依存関数のチェーン）
async def require_admin(
    user = Depends(get_current_user)   # ← 別の依存関数を入れ子にできる
):
    if not user.is_admin:
        raise HTTPException(
            status_code=403,
            detail="管理者権限が必要です"
        )
    return user

@app.delete("/users/{user_id}")
async def delete_user(
    user_id: int,
    admin = Depends(require_admin),    # ← 管理者のみ実行可能
    db = Depends(get_db),
):
    await db.delete(User, user_id)
    return {"message": "ユーザーを削除しました"}

🔍 なぜ依存性注入が重要か

1. コードの重複排除: 認証チェックを全エンドポイントに書く代わりに、Depends(get_current_user) と1行書くだけ。

2. テスタビリティ: テスト時に get_db をモックDBに差し替えるだけで、本番コードを一切変更せずにテスト可能。

3. 安全なリソース管理: yield を使った依存関数は、エラーが起きても finally で必ずリソースを解放する。
