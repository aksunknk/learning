01

JSXの秘密（HTMLとの違い）

▼

💡 なぜJavaScriptの中にHTMLを書くのか？（JSXの魔法）

従来のWeb開発では、HTML（構造）、CSS（装飾）、JS（振る舞い）を別々のファイルに分割していました。しかし、モダンなUI開発では「機能ごとに部品（コンポーネント）を分ける」方が圧倒的に管理しやすくなります。JSXは、この「UI構造とロジックを密結合させる」ために発明されました。関心事の分離という概念を「技術ベース」から「機能ベース」へとパラダイムシフトさせた画期的な構文です。

JSXとは何か？

Reactでは、JavaScriptのコードの中に直接HTMLのようなタグ（JSX）を書きます。一見すると気持ち悪いですが、UIとロジックを1つのファイルにまとめることで、コンポーネント指向（部品化）が劇的にやりやすくなります。

REACT 
import React from 'react';

// 【Senior Engineer's Note】
// Reactコンポーネントは常に「大文字」から始めます。
// 小文字で始めると通常のHTMLタグとして解釈され、Reactはそれを描画できません。
export default function Welcome() {
  const name = 'React';
  
  // 💡 波括弧 { } の中には「式（Expression）」を書くことができます。
  // if文やfor文は「文（Statement）」なので書けません。
  return (
    

      
Hello, {name}

    

  );
}

JSXはHTMLに似ていますが、裏側では `React.createElement` などのJavaScript関数呼び出しに変換されます。そのため、JavaScriptの予約語（classやfor）は使えません。この違いを知らないと、AIの出力コードをコピペした時に謎のエラーに悩まされます。

  return (

💼 Tips：実務で遭遇する現場の勘所

JSXでの条件分岐は論理積（&&）を多用しますが、{count && <Component />} のように書くと、countが0の時に画面に「0」という数字がそのまま描画されるという非常に有名な罠があります。数値を評価する場合は {count > 0 && <Component />} または {!!count && <Component />} のように、必ず真偽値（boolean）に変換してから評価する癖をつけましょう。

02

コンポーネントとProps（UIの部品化）

▼

💡 なぜコンポーネントに分けるのか？（再利用と保守性の向上）

1つの巨大なファイル（神ファイル）に全ての画面を書くと、修正時の影響範囲が不明瞭になり、バグの温床になります。「ヘッダー」「サイドバー」「記事一覧」のようにUIを独立した部品（コンポーネント）に分割することで、コードの再利用性が高まり、チーム開発でも競合が起きにくくなります。

コンポーネントの分割

Reactの真骨頂は画面を小さな部品に分割できることです。各コンポーネントは独立して動作し、親コンポーネントがそれらを組み合わせてページ全体を構築します。

REACT 
import React from 'react';

// 【Senior Engineer's Note】
// 部品は可能な限り小さく「1つの役割だけを持つ」ように設計します（単一責任の原則）。
function Header() {
  return

}

function Sidebar() {
  return

}

// 親コンポーネントで部品をレゴブロックのように組み合わせる
export default function App() {
  return (
    

      

      
      
Main Content

    

  );
}

親の部品から子の部品へデータを渡す仕組みをProps（プロップス）と呼びます。これは関数の引数と同じ概念ですが、読み取り専用（Immutable）であるという絶対ルールがあります。子は親から貰ったデータを直接書き換えてはいけません。

💼 Tips：実務で遭遇する現場の勘所

Propsにはデータだけでなく「関数」も渡せます（コールバックProps）。親が持つStateを子が更新したい場合は、親で更新関数を定義し、それをPropsとして子に渡すパターンが頻出します。また、型安全性を担保するために実務ではTypeScriptを併用し、Propsの型（type Props = { name: string; age: number }）を必ず定義します。

03

useStateの徹底解剖（状態管理）

▼

💡 なぜ普通の変数ではなくStateを使うのか？

JavaScriptの通常の変数（let count = 0;）を変更しても、Reactは「データが変わったこと」に気づけません。画面を再描画（レンダリング）して最新のUIを反映させるには、Reactに対して明示的に「データが変わったから画面を更新してね」と伝える必要があります。そのトリガーとなるのが `useState` です。

State（状態）とは何か？

画面を書き換えたいデータ（いいね数、入力中の文字、モーダルの開閉状態など）は、必ず `useState` を使って管理します。

REACT 
import React, { useState } from 'react';

export default function Counter() {
  // count: 現在の値, setCount: 値を更新する関数
  // useState(0): 初期値を0に設定
  const [count, setCount] = useState(0);

  // 【Senior Engineer's Note】
  // stateの更新は「非同期」で行われます。
  // setCountを呼んだ直後に console.log(count) をしても古い値が出力されます。
  const handleIncrement = () => {
    // 連続で更新する場合はコールバック関数を使います
    // setCount(prev => prev + 1);
    setCount(count + 1);
  };

  return (
    

      
You clicked {count} times

      
    

  );
}

配列やオブジェクトのState更新の罠

Reactは「前のデータと今のデータが同じか」を**参照（メモリのアドレス）**で比較します（浅い比較: Shallow Compare）。配列に `push()` して中身を変えても、配列という「箱（アドレス）」自体は変わっていないため、Reactは「変化なし」と判断して画面を更新しません。

REACT 
import React, { useState } from 'react';

export default function ItemList() {
  const [items, setItems] = useState(['Apple']);

  // ❌ 画面が更新されないバグ（元の配列を直接破壊している）
  const addBad = () => {
    items.push('Banana');
    setItems(items); // React「参照が同じだから更新スキップするね」
  };

  // ✅ 画面が正しく更新される（新しい配列を作ってセットしている）
  // 【Senior Engineer's Note】
  // オブジェクトや配列を更新する際は、必ずスプレッド構文（...）を使って
  // 「新しいメモリ空間」にデータを作成するイミュータブル（不変）な操作を行います。
  const addGood = () => {
    setItems([...items, 'Banana']);
  };

  return (
    

      
      
    {items.map((item, i) => 
  • {item}
    • )}

    

  );
}

💼 Tips：実務で遭遇する現場の勘所

Stateの持ちすぎに注意しましょう。「姓（lastName）」と「名（firstName）」のStateがある時、「フルネーム（fullName）」のStateを別途作るのはアンチパターンです。const fullName = `${lastName} ${firstName}`; のように、既存のStateから算出可能な値は単なる変数としてレンダリング時に計算するのがベストプラクティスです。これを怠ると、データの不整合（姓を変えたのにフルネームが変わらない）が発生します。

04

useEffectの罠と対策（副作用）

▼

💡 なぜuseEffectが必要なのか？（純粋関数と副作用の分離）

Reactコンポーネントは本来「データを受け取ってHTML（UI）を返す」だけの純粋な関数であるべきです。しかし、現実のアプリでは「サーバーからデータを取得する」「DOMを直接操作する」「タイマーをセットする」といったReactの外の世界との通信が必要です。これらの「副作用（Side Effects）」をコンポーネントの描画サイクルから切り離し、安全に実行するための場所が `useEffect` です。

副作用（Side Effect）とは？

API通信などは、コンポーネントが画面に表示された後（マウント後）に行う必要があります。レンダリング中に直接通信処理を書くと、レンダリングの度に通信が発生してしまいます。

REACT 
import React, { useState, useEffect } from 'react';

export default function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  // 【Senior Engineer's Note】
  // useEffectの第2引数（依存配列）に [userId] を指定することで、
  // 「初回マウント時」および「userIdが変化した時」だけこの処理が走ります。
  useEffect(() => {
    let ignore = false; // レースコンディション対策フラグ

    async function fetchUser() {
      const res = await fetch(`/api/users/${userId}`);
      const data = await res.json();
      if (!ignore) {
        setUser(data);
      }
    }

    fetchUser();

    // クリーンアップ関数
    return () => { ignore = true; };
  }, [userId]);

  if (!user) return 
Loading...
;
  return 
{user.name}
;
}

クリーンアップ関数

イベントリスナーやタイマーをセットした場合、コンポーネントが画面から消えるとき（アンマウント時）にそれを解除しないと、裏でタイマーが動き続けるメモリリーク（動作が重くなる原因）になります。

REACT 
import React, { useEffect, useState } from 'react';

export default function Clock() {
  const [time, setTime] = useState(new Date());

  useEffect(() => {
    const timerId = setInterval(() => {
      setTime(new Date());
    }, 1000);

    // 【Senior Engineer's Note】
    // returnで関数を返すと、コンポーネントの破棄時や、
    // 次のuseEffectが走る直前に実行されます（お掃除）。
    return () => clearInterval(timerId);
  }, []); // 空の依存配列なので、マウント・アンマウント時のみ実行される

  return 
Current Time: {time.toLocaleTimeString()}
;
}

💼 Tips：実務で遭遇する現場の勘所

React 18のStrict Modeでは、開発環境でのみ `useEffect` が「マウント → アンマウント → マウント」と意図的に2回連続で実行されます。これはクリーンアップ関数が正しく実装されているかをReactが検証するためです。「APIが2回呼ばれてしまう！」と慌ててStrict Modeをオフにするのはアンチパターンです。クリーンアップ処理（abortControllerによるfetchキャンセルなど）を正しく実装して対処しましょう。

イベント処理とフォーム（ユーザーの入力を受け取る）

▼

💡 なぜフォーム入力をReactで「制御」するのか？

HTMLのフォーム要素（input, textarea）は、デフォルトで自身の中に「状態」を持っています。しかしReactの世界では「UIは常にStateを反映したものであるべき（Single Source of Truth）」です。そこで、入力値を常にReactのStateと同期させる「制御されたコンポーネント（Controlled Components）」という手法を用い、バリデーションや即時反映を容易にします。

イベントハンドラの基本

HTMLの `onclick` とは異なり、Reactでは `onClick` のようにキャメルケースで書き、文字列ではなく「関数そのもの」を渡します。

REACT 
import React from 'react';

export default function Button() {
  // ❌ 罠: 関数を実行した結果を渡してしまう
  // レンダリングされた瞬間にアラートが出てしまい、クリック時には何も起きない
  // return ;

  // ✅ 対策: 関数そのものを渡す（アロー関数で包むか、関数名のみを渡す）
  const handleClick = () => {
    alert('Hi!');
  };

  return (
    

      {/* アロー関数で包むパターン */}
      
      
      {/* 関数への参照を渡すパターン（推奨） */}
      
    

  );
}

制御されたコンポーネント（Controlled Components）

Reactでは、入力フォームの値をReactのStateで完全に管理するのがベストプラクティスです。

REACT 
import React, { useState } from 'react';

export default function Form() {
  // Stateで入力値を一元管理
  const [text, setText] = useState('');

  // 【Senior Engineer's Note】
  // value と onChange は必ずセットで使います。
  // onChangeがないと、ユーザーがキーボードを叩いても画面上の文字が変化しません
  // （Stateが更新されないため、UIも更新されない）。
  const handleChange = (e) => {
    setText(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault(); // フォームのデフォルトの画面遷移を阻止
    console.log('Submitted:', text);
  };

  return (
    

      
      
      
Preview: {text}

    

  );
}

💼 Tips：実務で遭遇する現場の勘所

入力フォームが多い画面で、全てのinput要素にuseStateを個別に作るとコードが肥大化しパフォーマンスも低下します。実務では React Hook Form というライブラリを導入し「非制御コンポーネント（Uncontrolled Components）」として効率的にフォームを管理するのがモダンフロントエンドの標準です。制御コンポーネントは「入力と同時に検索サジェストを出す」ような即時性が求められる場面に限定して使うことが増えています。

条件付きレンダリングとリスト表示 — key の本質

▼

💡 「UI を分岐させる」のではなく「値を分岐させる」

React には if ディレクティブがありません。 Vue の v-if や Angular の *ngIf のような専用構文は存在せず、JSX は「ただの JavaScript の式」なので、条件分岐もループも素の JavaScript（三項演算子・&&・map）で書きます。これは学習コストを下げる設計判断であり、「JavaScript ができれば React の制御構文は追加で覚えることがない」ということです。

条件付きレンダリングの 3 パターン

①「A か B か」は三項演算子、②「表示するかしないか」は &&、③ 分岐が複雑なら早期 return。この 3 つで実務のほぼ全ケースをカバーできます。null を返すと「何も描画しない」という意味になります。

JSX 
function Dashboard({ user, notifications }) {
  // ③ 分岐が大きいなら早期 return が読みやすい
  if (!user) {
    return <LoginForm />;
  }

  return (
    <div>
      {/* ① A か B か：三項演算子 */}
      {user.isPremium ? <GoldBadge /> : <FreeBadge />}

      {/* ② 表示する / しない：&& 演算子 */}
      {notifications.length > 0 && (
        <p>{notifications.length} 件の通知があります</p>
      )}
    </div>
  );
}

⚠️ && の落とし穴：0 が表示される

{count && <p>...</p>} と書くと、count が 0 のとき「何も表示しない」ではなく画面に 0 がそのまま描画されます。JavaScript の && は左辺が falsy ならその値自体を返すためです。数値を条件にするときは {count > 0 && ...} と明示的に真偽値へ変換してください。

リスト表示と key — 「並び替えても正しく追跡する」ための ID

配列の描画は map で JSX の配列を作るだけです。このとき各要素に必要な key は、React が再レンダリング時に「どの要素が追加・削除・移動したか」を特定するための識別子です。配列の index を key にすると、並び替えや先頭挿入のときに React が要素を取り違え、入力欄の中身が別の行に「引っ越す」バグが起きます。データが持つ一意な ID を使ってください。

JSX 
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo) => (
        // ✅ データ固有の ID を key にする
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  );
}

// ❌ アンチパターン：index を key にする
// {todos.map((todo, i) => <li key={i}>...</li>)}
// → 先頭に要素を追加すると全行の key がずれ、状態が取り違えられる

💼 Tips：key は「兄弟の中で一意」であればよい

key はアプリ全体でユニークである必要はなく、同じ配列（兄弟要素）の中で一意なら十分です。また、key を変えると React はその要素を「別物」とみなして作り直すため、逆にこの性質を使って「key を変えてコンポーネントの状態を意図的にリセットする」テクニックも実務で使われます。

副作用の統制（useEffectと依存配列の正しい使い方）

▼

💡 なぜ依存配列（Dependency Array）でバグが起きるのか？

Reactの useEffect は、「コンポーネントの描画結果」と「外部システム（API等）」を同期するためのツールです。依存配列を間違えると、Reactの描画データ（最新のStateやProps）と、useEffectの内部で参照しているデータがズレてしまい（Stale Closureと呼ばれる古いメモリの参照）、予測不能なバグを引き起こします。

依存配列の落とし穴

useEffectの第二引数（依存配列）に空の配列 `[]` を渡すと「初回マウント時の一度だけ」実行されます。しかし、useEffectの中で使っている変数が依存配列に含まれていないと、大きな問題になります。

REACT 
import React, { useEffect, useState } from 'react';

export default function SearchResults({ query }) {
  const [data, setData] = useState([]);

  // 【Senior Engineer's Note】
  // ESLintの `react-hooks/exhaustive-deps` ルールは絶対に無視してはいけません。
  // 警告を消すために無理やり [] にすると、古いクロージャの変数を参照し続けます。
  useEffect(() => {
    // ❌ 依存配列にqueryがないと、親から新しいqueryが渡されても再フェッチされない
    async function fetchData() {
      const res = await fetch(`/api/search?q=${query}`);
      const result = await res.json();
      setData(result);
    }
    fetchData();
  }, [query]); // ✅ 使用している外部変数(query)は全て依存配列に入れる

  return (
    
    
      {data.map(item => 
  • {item.name}
    • )}
    

  );
}

💼 Tips：実務で遭遇する現場の勘所

「コンポーネントの初回マウント時に一度だけ実行したいから」という理由で [] を指定し、ESLintの警告を // eslint-disable-next-line で握りつぶすエンジニアが後を絶ちません。これは技術的負債の第一歩です。useEffect内で使う関数やオブジェクトが毎回新しく生成されて不要な再発火を招く場合は、安易に依存を消すのではなく、その関数を useCallback でキャッシュするか、useEffectの外側（コンポーネントの外）に移動させるのが正しい解決策です。

状態のバケツリレー回避（PropsとContext）

▼

💡 なぜContextを使うのか？（Props Drilling問題の解決）

コンポーネントが深くネスト（階層化）されると、親のデータを孫やひ孫に渡すためだけに、中間のコンポーネントが不要なPropsを受け渡す「Props Drilling（バケツリレー）」が発生します。これはコードの可読性を下げ、中間層の再利用性を損ないます。Context APIはこのバケツリレーをスキップし、必要なコンポーネントに直接データを「ワープ」させる仕組みです。

Contextによるグローバルなデータ共有

💼 Tips：実務で遭遇する現場の勘所

Contextは便利ですが、万能ではありません。Providerの `value` が更新されると、そのContextを購読している（useContextしている）全ての子孫コンポーネントが無条件で再レンダリングされます。頻繁に更新されるデータ（入力フォームの文字や、ドラッグ＆ドロップの座標など）をContextに入れるとアプリ全体が激しく重くなります。頻繁な更新があるグローバル状態の管理には、Contextではなく `Zustand` や `Jotai` といった専用の状態管理ライブラリを採用するのが実務の定石です。

useRef — DOM 参照と「再レンダリングを起こさない値」

▼

💡 useRef は「React に監視されない箱」

useState との決定的な違いは「変更しても再レンダリングが起きない」ことです。 useRef が返すオブジェクトの .current プロパティは自由に読み書きでき、コンポーネントが再レンダリングされても値が保持されます。つまり「画面表示には使わないが、レンダリングを跨いで覚えておきたい値」の置き場所です。用途は大きく 2 つ——① DOM 要素への直接アクセス、② タイマー ID や前回値などのインスタンス変数の代替です。

用途①：DOM 要素を直接操作する

「マウント時に入力欄へフォーカスする」「動画を再生する」など、宣言的な JSX では表現できない命令的操作には DOM への参照が必要です。JSX の ref 属性に渡すと、マウント後に .current へ DOM 要素が入ります。

JSX 
import { useRef, useEffect } from "react";

function SearchBox() {
  const inputRef = useRef(null);

  useEffect(() => {
    // マウント直後に検索欄へ自動フォーカス
    inputRef.current.focus();
  }, []);

  return <input ref={inputRef} placeholder="検索..." />;
}

用途②：レンダリングを跨ぐ「ただの変数」

タイマー ID を state に入れると、セットするたびに無意味な再レンダリングが走ります。「値が変わっても画面は変わらない」ものは ref に入れるのが正解です。

JSX 
function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  const timerRef = useRef(null);   // タイマー ID は表示に関係ない → ref

  const start = () => {
    if (timerRef.current !== null) return;  // 二重起動を防止
    timerRef.current = setInterval(() => {
      setSeconds((s) => s + 1);
    }, 1000);
  };

  const stop = () => {
    clearInterval(timerRef.current);
    timerRef.current = null;
  };

  return (
    <div>
      <p>{seconds} 秒</p>
      <button onClick={start}>開始</button>
      <button onClick={stop}>停止</button>
    </div>
  );
}

⚠️ レンダリング中に ref を読み書きしない

ref の読み書きはイベントハンドラか useEffect の中で行うのが原則です。レンダリング中（JSX を返す本体部分）で .current を書き換えると、React の並行レンダリング機能と相性が悪く、予測不能な挙動の原因になります。「表示に使う値は state、使わない値は ref」という線引きを崩さないでください。

💼 Tips：state か ref か迷ったら

判断基準は 1 つだけ——「その値が変わったとき、画面を描き直す必要があるか？」 Yes なら useState、No なら useRef。フォームの入力値は表示に反映するので state、スクロール位置の記録や WebSocket インスタンスの保持は ref、と機械的に判定できます。

カスタムフック（Custom Hooks）

▼

💡 なぜカスタムフックを作るのか？（ロジックの分離と再利用）

UIのコード（JSX）と、データの取得や計算のロジックが混ざり合うとコンポーネントはあっという間に肥大化します（ファットコンポーネント）。カスタムフックは、Reactの機能（useState等）を含んだ自分専用の関数を作る機能です。これにより、「見た目」と「機能」を完全に分離し、同じ機能を別のコンポーネントでも簡単に使い回せるようになります。

API取得ロジックの抽象化

データフェッチの処理はどの画面でも似たような構成（ローディング、エラー、データ保持）になります。これをカスタムフックに切り出します。

REACT 
import { useState, useEffect } from 'react';

// 【Senior Engineer's Note】
// カスタムフックの関数名は必ず `use` から始める規約があります。
// これにより、ReactのLinterがフックのルール（条件分岐の中で呼んではいけない等）を検査してくれます。
export function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const abortController = new AbortController();

    const fetchData = async () => {
      try {
        setLoading(true);
        const res = await fetch(url, { signal: abortController.signal });
        if (!res.ok) throw new Error('API Error');
        const json = await res.json();
        setData(json);
      } catch (err) {
        if (err.name !== 'AbortError') setError(err);
      } finally {
        setLoading(false);
      }
    };

    fetchData();

    // マウント解除時にフェッチをキャンセル
    return () => abortController.abort();
  }, [url]);

  return { data, loading, error };
}

// 使う側（UIコンポーネント）は驚くほどシンプルになります！
export default function UserProfile() {
  const { data, loading, error } = useFetch('/api/user');
  
  if (loading) return 
Loading...
;
  if (error) return 
Error!
;
  return 
{data.name}
;
}

💼 Tips：実務で遭遇する現場の勘所

実務でデータフェッチを行う際、自分で `useFetch` のようなカスタムフックを一から書くことは現在では稀です。キャッシュ管理、重複リクエストの排除、リトライ処理など考慮すべきエッジケースが多すぎるためです。通常は `SWR` や `TanStack Query (React Query)` という強力なデータフェッチ専用ライブラリが提供するカスタムフックを利用します。自作カスタムフックは「ローカルストレージとの同期」や「複雑なフォーム制御」などに用いるのが一般的です。

パフォーマンス最適化（メモ化）

▼

💡 なぜメモ化が必要なのか？（不要な再レンダリングの抑止）

Reactは「親コンポーネントが再描画されると、子コンポーネントも無条件で全て再描画される」という仕様があります。大抵のUI更新は非常に高速なので問題になりませんが、巨大なテーブルや複雑なグラフなどを描画する際、無関係なState（例えばタイピング中の検索バー）が更新されるたびに重いコンポーネントまで再描画されると、画面がカクつきます。「メモ化（Memoization）」は、結果をキャッシュして不要な再計算・再描画を防ぐテクニックです。

useMemo と useCallback

計算結果を保存するのが `useMemo`、関数のインスタンスを保存するのが `useCallback`、コンポーネント自体の再描画を防ぐのが `React.memo` です。

REACT 
import React, { useState, useMemo, useCallback, memo } from 'react';

// 【Senior Engineer's Note】
// memo() で囲むと、渡されるPropsが「前回と完全に一致」する場合のみ再描画をスキップします。
const ExpensiveChild = memo(({ onClick }) => {
  console.log('Heavy Child Renders!');
  return ;
});

export default function App({ largeList }) {
  const [count, setCount] = useState(0);

  // useMemo: 重い計算結果をキャッシュする
  // largeListが変わらない限り、countが変わって再レンダリングされてもここはスキップされる
  const sortedList = useMemo(() => {
    console.log('Heavy Sorting...');
    return [...largeList].sort((a, b) => b.score - a.score);
  }, [largeList]);

  // useCallback: 関数のインスタンス（参照）をキャッシュする
  // これがないとAppが再描画される度に関数が新しく生成され、
  // ExpensiveChildは「新しいonClickが来た！」と勘違いして再描画されてしまう。
  const handleClick = useCallback(() => {
    console.log('Clicked!');
  }, []); // 依存配列が空なので、この関数は最初に作られたものを使い回す

  return (
    

      {/* このボタンを押すとAppは再描画されるが、ExpensiveChildは再描画されない */}
      
      
    

  );
}

💼 Tips：実務で遭遇する現場の勘所

「とりあえず全部にmemoとuseCallbackをつければ速くなる」と勘違いしているエンジニアが多いですが、これはアンチパターンです。メモ化自体にも「依存配列を比較する」というコストがかかるため、軽量なコンポーネントに乱用するとかえってパフォーマンスが悪化します。実務ではまず「状態（State）をコンポーネントのより下位に移動できないか」という設計の見直しを行い、それでも遅い場合にのみプロファイラ（React DevTools）で計測しながらピンポイントでメモ化を適用します。

React 19 の新機能 — Server Components と use / Actions

▼

💡 React は「クライアントだけのライブラリ」をやめた

React 19 最大の変化は、サーバーとクライアントの役割分担が言語レベルで表現できるようになったことです。 従来の React はすべての JS をブラウザに送り、ブラウザで描画していました。React Server Components（RSC）では、データ取得だけが目的のコンポーネントはサーバーで実行して HTML 相当の結果だけを送り、インタラクションが必要な部分だけをクライアント JS として送ります。バンドルサイズと初期表示速度が構造的に改善される仕組みです。Next.js（App Router）はこのモデルを最初に採用した代表的フレームワークです。

Server Components と "use client" の境界

デフォルトはサーバーコンポーネントで、async/await で直接 DB や API を呼べます（useEffect でのフェッチが不要になる）。useState などブラウザ側の機能を使うファイルの先頭には "use client" を宣言します。

JSX 
// app/posts/page.jsx — Server Component（デフォルト）
// コンポーネント自体を async にでき、await で直接データ取得できる
export default async function PostsPage() {
  const posts = await db.query("SELECT * FROM posts");  // サーバーで実行
  return (
    <ul>
      {posts.map((p) => <PostItem key={p.id} post={p} />)}
    </ul>
  );
}

// LikeButton.jsx — Client Component（インタラクションが必要）
"use client";
import { useState } from "react";

export function LikeButton({ initial }) {
  const [likes, setLikes] = useState(initial);
  return <button onClick={() => setLikes(likes + 1)}>👍 {likes}</button>;

use フックと Actions — 非同期処理の新しい書き方

React 19 では use(promise) で Promise の結果を Suspense と連携しながら読み取れます。またフォーム送信は「Action」として関数を <form action={fn}> に渡す方式になり、送信中状態は useActionState が自動管理します。「loading フラグを useState で手動管理する」定型コードが不要になります。

JSX 
"use client";
import { useActionState } from "react";

function SignupForm() {
  // action 関数の戻り値が state に、実行中は pending が true になる
  const [result, submitAction, pending] = useActionState(
    async (prev, formData) => {
      const res = await fetch("/api/signup", {
        method: "POST",
        body: formData,
      });
      return res.ok ? { ok: true } : { error: "登録に失敗しました" };
    },
    null
  );

  return (
    <form action={submitAction}>
      <input name="email" type="email" required />
      <button disabled={pending}>{pending ? "送信中..." : "登録"}</button>
      {result?.error && <p>{result.error}</p>}
    </form>
  );
}

💼 Tips：forwardRef の廃止と学習の優先順位

React 19 では ref がただの props として渡せるようになり、forwardRef は不要になりました。新機能は多いですが、RSC はフレームワーク（Next.js 等）とセットで初めて使えるものです。素の React（Vite 構成）で開発する場合は従来どおりの書き方が現役なので、「概念を理解した上で、必要になったら使う」姿勢で十分です。

📖 ① Zustand — たった40行で作れる状態管理ライブラリ

Zustand は「Reactの状態管理を最もシンプルに」をコンセプトにしたライブラリです。Reduxのボイラープレートの多さを排除し、数十行のコードで状態管理の核心を実現しています。そのコア（createStore）を読み解きます。

TYPESCRIPT 
// 出典: pmndrs/zustand — src/vanilla.ts (MIT)
// 【Senior Engineer's Note】
// Zustandの状態はグローバル変数ではなく、「クロージャ（関数スコープに閉じ込められた変数）」
// として隠蔽（カプセル化）されています。外部からはgetState()でしか読めません。
const createStoreImpl = (createState) => {
  let state;                            // 状態の隠蔽（カプセル化）
  const listeners = new Set();          // 購読者リスト（重複防止のためSetを利用）

  const setState = (partial, replace) => {
    // 関数が渡された場合は現在のstateを引数として実行
    const nextState = typeof partial === 'function' ? partial(state) : partial;

    // Reactの最適化: 状態が同一(参照が同じ)なら通知をスキップする
    if (!Object.is(nextState, state)) {
      const previousState = state;
      // replaceフラグによる完全置換 or スプレッド構文によるマージ
      state = replace ? nextState : Object.assign({}, state, nextState);
      
      // 状態変更を全てのコンポーネント（リスナー）にブロードキャスト
      listeners.forEach((listener) => listener(state, previousState));
    }
  };

  const getState = () => state;

  const subscribe = (listener) => {
    listeners.add(listener);
    // useEffectのクリーンアップで呼ばれる「解除関数」を返す美しい設計
    return () => listeners.delete(listener); 
  };

  // ストアの初期化とAPIの公開
  const api = { setState, getState, subscribe };
  state = createState(setState, getState, api);
  return api;
};

📖 ② React Router — 宣言的ルーティングの内部

React Router は、URLが変わっても画面全体をリロードせず、JavaScriptの力でコンポーネントだけを入れ替える SPA（Single Page Application）の根幹を支えるライブラリです。その魔法の裏側は Context API とブラウザの History API の組み合わせです。

REACT 
// 出典: remix-run/react-router (MIT) の概念実装
import React, { createContext, useContext, useState, useEffect } from 'react';

const RouterContext = createContext(null);

export function BrowserRouter({ children }) {
  // ブラウザの現在のURLをReactのStateとして管理する
  const [location, setLocation] = useState(window.location.pathname);

  useEffect(() => {
    // ブラウザの「戻る/進む」ボタンを検知してStateを同期
    const handlePopState = () => setLocation(window.location.pathname);
    window.addEventListener('popstate', handlePopState);
    return () => window.removeEventListener('popstate', handlePopState);
  }, []);

  const navigate = (to) => {
    // History APIを使ってリロードせずにURLだけを書き換える
    window.history.pushState({}, '', to);
    setLocation(to); // URL変更をReactに教え、再レンダリングを発火させる
  };

  return (
    
      {children}
    
  );
}

// 【Senior Engineer's Note】
// aタグのデフォルト挙動（ページ全体リロード）を preventDefault() で殺し、
// 代わりに navigate 関数を実行することで爆速の画面遷移を実現しています。
export function Link({ to, children }) {
  const { navigate } = useContext(RouterContext);
  return (
     { e.preventDefault(); navigate(to); }}>
      {children}
    
  );
}

💼 Tips：実務で遭遇する現場の勘所

OSSのコードを読む時は、全てを理解しようとせず「ディレクトリ構造」と「テストコード」から入るのが鉄則です。特にテストファイル（*.test.ts）は「そのライブラリが想定しているユースケースとエッジケース（異常系）」を開発者自身が解説している究極のドキュメントです。機能追加で行き詰まった時は、似た機能を持つOSSのテストコードを漁るのがシニアエンジニアの裏技です。

実サービスのコードを読む① — Excalidraw（ホワイトボードツール）

▼

💡 なぜ大作アプリの設計を学ぶのか？（スケーラビリティの確保）

小さなアプリでは適当な設計でも動きますが、Excalidrawのような数万行のコードベース、かつ「元に戻す/やり直す（Undo/Redo）」や「リアルタイム共同編集」が飛び交う複雑なアプリでは、データ構造とアクションの設計が命です。「全ての操作を統一化する設計」や「物理削除しないデータ管理（論理削除）」は、大規模アプリケーションを崩壊させないための防波堤となります。

アクションシステムと論理削除（Soft Delete）

キャンバスの全消去機能の実装から、Excalidrawの「削除 ≠ データの破壊」という設計思想が明確に分かります。Undoを実現するためには絶対に物理削除してはいけません。

TYPESCRIPT 
// 出典: excalidraw/excalidraw (MIT)
// packages/excalidraw/actions/actionCanvas.tsx

export const actionClearCanvas = register({
  name: "clearCanvas",
  label: "labels.clearCanvas",
  trackEvent: { category: "canvas" }, // アナリティクス用のトラッキング

  // predicate: このアクションが現在「実行可能か」を判定する述語関数
  predicate: (elements, appState, props, app) => {
    return (
      !!app.props.UIOptions.canvasActions.clearCanvas
      && !appState.viewModeEnabled // 閲覧モード中は消去ボタンを無効化
    );
  },

  // perform: 実際の処理ロジック（純粋関数として新しい状態を返す）
  perform: (elements, appState, _, app) => {
    app.imageCache.clear();

    return {
      // 【Senior Engineer's Note】
      // 要素を配列から消し去る（物理削除）のではなく、
      // `isDeleted: true` というフラグを立てて隠す（論理削除）だけ。
      // これにより「Ctrl+Z」を押した時に isDeleted: false にするだけで復活できる。
      elements: elements.map((element) =>
        newElementWith(element, { isDeleted: true })
      ),
      
      // アプリの状態はリセットするが、ダークモードやペンの設定は「保持」する
      appState: {
        ...getDefaultAppState(),
        theme: appState.theme,
        penMode: appState.penMode,
      },
      captureUpdate: CaptureUpdateAction.IMMEDIATELY, // Undo履歴に即時追加
    };
  },
});

ブランド型（Branded Types）による型安全の極致

TypeScriptの限界を超えるテクニックとして、Excalidrawでは「エクスポート処理済みの要素」と「未処理の要素」を型レベルで厳密に区別しています。

TYPESCRIPT 
// 【Senior Engineer's Note】
// 通常の配列に `_brand` という架空のプロパティを交差型（&）で付与します。
// これにより、通常の要素配列を間違ってエクスポート関数に渡すと
// 「_brand プロパティがありません」とコンパイルエラーになり、バグを未然に防ぎます。
export type ExportedElements = readonly NonDeletedExcalidrawElement[] & {
  _brand: "exportedElements";
};

export const prepareElementsForExport = (elements): ExportedElements => {
  // 削除済みフラグが立っている要素を除外する
  const cleanElements = getNonDeletedElements(elements);
  
  // 処理が終わった配列を「ブランド型」としてアサーション（キャスト）して返す
  return cleanElements as unknown as ExportedElements;
};

💼 Tips：実務で遭遇する現場の勘所

実務のデータベース設計でも「論理削除」は頻出します（ユーザーの退会処理など）。しかし、論理削除フラグ（`deleted_at` 等）を導入すると、全ての取得クエリで `WHERE deleted_at IS NULL` を付け忘れないよう常に気を配る必要があり、バグの温床になりがちです。Excalidrawが `getNonDeletedElements` のようなヘルパー関数を噛ませているように、論理削除されたデータを自動で除外するリポジトリ層の設計や、Prismaの中間ミドルウェアでの自動除外設定が実務では必須となります。

実サービスのコードを読む② — Cal.com（スケジュール予約サービス）

▼

💡 なぜCal.comのスタックが「モダン」なのか？（型安全性の一気通貫）

Cal.comは Next.js, tRPC, Prisma, Zod などを採用したフルスタックTypeScriptアプリの最高峰です。従来の開発では、フロントエンドとバックエンドのAPIスキーマがズレて「Cannot read property of undefined」でアプリが落ちるのが日常茶飯事でした。Cal.comのアーキテクチャでは、DBのスキーマからフロントエンドのUIコンポーネントまで、全てが「型（Type）」でシームレスに繋がっており、コンパイルが通れば動くという堅牢なDX（開発者体験）を実現しています。

tRPCとZodによる End-to-End の型安全

バックエンドのAPIハンドラの実装です。入力データの検証（バリデーション）からDBクエリまでが驚くほど綺麗に整理されています。

TYPESCRIPT 
// 出典: calcom/cal.com (AGPLv3 + Commercial)
// packages/trpc/server/routers/viewer/bookings/get.handler.ts

// 【Senior Engineer's Note】
// 入力の型（TGetInputSchema）は Zod によってランタイム検証済みの安全なデータです。
// もしリクエストに不正な文字列が混ざっていたら、この関数に到達する前に弾かれます。
type GetOptions = {
  ctx: {
    user: NonNullable; // 認証済みであることが型レベルで保証されている
    prisma: PrismaClient;               // DBアクセスクライアント
  };
  input: TGetInputSchema;
};

export const getHandler = async ({ ctx, input }: GetOptions) => {
  const { prisma, user } = ctx;
  const take = input.limit;
  let skip = input.offset;

  // オフセット（skip=20）とカーソル（id=xyz）の両方のページネーションに対応する柔軟な設計
  if (input.cursor) {
    const parsedCursor = parseInt(input.cursor, 10);
    if (!Number.isNaN(parsedCursor) && parsedCursor >= 0) skip = parsedCursor;
  }

  // ステータスフィルター（未指定の場合は 'upcoming' をデフォルトとする）
  const statuses = input.filters.statuses?.length
    ? input.filters.statuses
    : [input.filters.status || "upcoming"];

  // ビジネスロジックを別ファイルに分離（Fat Controllerを防ぐ）
  const { bookings, totalCount } = await getAllUserBookings({
    ctx: { user: { id: user.id }, prisma },
    statuses, take, skip
  });

  const nextOffset = skip + take;
  const hasMore = nextOffset < totalCount;

  // 【Senior Engineer's Note】
  // ここでreturnしたオブジェクトの型が、自動的にフロントエンドの
  // useQuery().data の型として推論されます。型定義の二重管理が不要です。
  return { 
    bookings, 
    totalCount, 
    nextCursor: hasMore ? nextOffset.toString() : undefined 
  };
};

モノレポ（Monorepo）によるコードの組織化

Cal.comは1つのリポジトリの中に複数のパッケージを持っています（Turborepoを使用）。UIコンポーネント（@calcom/ui）やデータベーススキーマ（@calcom/prisma）を独立したパッケージに切り出し、Webアプリ本体やAPIサーバーから再利用しています。大規模開発における「依存関係の整理」の最適解です。

💼 Tips：実務で遭遇する現場の勘所

tRPCは小〜中規模なTypeScriptフルスタックプロジェクトでは最強の生産性を誇りますが、「マイクロサービス化」や「モバイルアプリ（Swift/Kotlin）へのAPI提供」が必要になった瞬間に破綻しやすいという弱点があります（TypeScript以外の言語から型を参照しづらいため）。そのため、事業がスケールして多言語展開が見込まれるプロジェクトでは、tRPCではなく `GraphQL` や `OpenAPI + Swagger` によるスキーマ駆動開発を採用するのが技術選定のセオリーです。

📱 モバイルへの橋 — React Native / Expo

ここまでの React は Web（DOM）向けです。同じ「コンポーネントと状態」の思考をネイティブへ写す最短の地図だけ置きます。通しプロジェクト（縦糸）は Web のまま進め、本格的な RN カリキュラムは別章（elective）候補です。

Web React と React Native の対応表

▼

💡 同じ React、違う「描画先」

React Native（RN）も useState / props / フックのルールは同じです。違うのは DOM が無いこと。HTML タグの代わりにネイティブ部品（View / Text / Pressable）を使い、CSS もブラウザのカスケードではなく StyleSheet のオブジェクトです。

よく使う対応

頭の中の辞書を一度入れ替えると、既存のコンポーネント設計がそのまま移植しやすくなります。

対応表 
Web (DOM)              React Native
──────────────         ─────────────────
div / section          View
p / span / h1          Text
button / a             Pressable / TouchableOpacity
img                    Image
input                  TextInput
className + CSS        style={styles.xxx}（StyleSheet）
react-router           @react-navigation/*
REACT NATIVE 
import { View, Text, Pressable, StyleSheet } from "react-native";

export function TaskRow({ title, onPress }) {
  return (
    <View style={styles.row}>
      <Text>{title}</Text>
      <Pressable onPress={onPress}>
        <Text>開く</Text>
      </Pressable>
    </View>
  );
}

const styles = StyleSheet.create({
  row: { padding: 12, flexDirection: "row", justifyContent: "space-between" },
});

💼 Tips：現場の勘所

「CSS をそのまま持っていく」は失敗しやすいです。レイアウトは Flexbox が中心で、Web のセレクタ・メディアクエリ・一部単位は使えません。状態とデータ取得のフックは再利用し、見た目レイヤだけ RN 用に切り替えるのが定石です。

いつ Web / いつ RN か — このサイトでの進め方

▼

💡 選択は「同じ状態モデルをどこに載せるか」

モバイルが必要でも、まず Web でデータ形・API・認可を固めると総コストが下がります。このサイトの通しプロジェクト（TaskBoard）も Web 固定です。RN は「同じ思考の移植」として後続オプションに置きます。

ざっくり判断

優先 Web（ブラウザ）
  - 社内ツール、管理画面、SEO が要る公開サイト
  - まず API とドメインを検証したい

優先 RN / Expo
  - カメラ・プッシュ・オフラインが中核
  - ストア配布がプロダクト要件

両方
  - 共有は「ドメインロジックと API クライアント」
  - UI は Web / Native で分ける（無理な完全共有は避ける）

このカリキュラムでの位置づけ

  1. React（Web）章を完了する ← 今ここまでの本体
  2. Pathway / Capstone は Web で通す
  3. 必要なら RN/Expo を elective 章として深掘り（将来）

橋渡し 3 レッスンのゴールは「対応表が頭にあり、Expo で Hello を想像できる」までです。ストア審査・ネイティブモジュール・性能チューニングは本編の外です。

💼 Tips：現場の勘所

「最初からクロスプラットフォームで全部書く」は、状態と API が固まっていない段階では逆に遅いです。Web で契約を固定 → モバイルは薄いクライアント、の順がフルスタック学習と相性が良いです。
