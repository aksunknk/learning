🦀 Rust 入門

所有権による安全性と C 並みの性能を両立する言語を、初級（文法）・中級（所有権）・上級（トレイト / ライフタイム / テスト）の 3 段階で学びます。

🟢 初級編 — Rust の文法に慣れる

Cargo によるプロジェクト管理、変数と可変性、データ型、式指向の制御フローまで。所有権に入る前の「読み書きの基礎体力」を作ります。

01

はじめに：Rust の位置づけと Cargo

▼

💡 なぜ今 Rust を学ぶ価値があるのか

「速さ」と「安全」は普通トレードオフの関係にあります。
C / C++ は高速ですが、メモリ管理を人間が担うため、解放済み領域への参照（use-after-free）や二重解放といった深刻なバグを生みます。一方 Python や Java はガベージコレクタ（GC）で安全性を確保しますが、実行時にコストを払います。Rust は「GC を持たない」のに「コンパイル時にメモリ安全を保証する」という第三の道を選びました。所有権という仕組みで、危険なコードをそもそもコンパイルエラーとして弾くのです。だから実行速度は C++ 級のまま、安全性は言語仕様で守られます。

最初のプログラムと println!

Rust のエントリポイントは main 関数です。画面出力には println! を使います。末尾の ! は「これは関数ではなくマクロ」という印で、Rust では出力系の多くがマクロとして提供されます。文字列中の {} はプレースホルダで、変数名を直接埋め込むこともできます。C の printf と違い、型ごとに書式指定子（%d など）を変える必要はありません。

Rust 
fn main() {
    let name = "世界";
    let count = 3;
    // {name} のように変数名を直接埋め込める
    println!("{name} に {count} 回こんにちは");
}

Cargo：ビルドとパッケージ管理の中心

Rust の開発は Cargo を中心に回ります。Python の pip + venv、Node の npm をまとめたような存在です。cargo new で雛形を作り、cargo run でビルドと実行を同時に行えます。依存ライブラリ（Rust では「クレート crate」と呼ぶ）は Cargo.toml に書くだけで自動取得・コンパイルされます。初学者はまず cargo run と cargo build の 2 つを覚えれば十分です。

Bash 
# 新規プロジェクトを作成
cargo new hello
cd hello

# ビルドして実行（デバッグビルド）
cargo run

# 最適化した本番ビルド
cargo build --release

💼 Tips：エラーメッセージは「先生」である

Rust のコンパイラ（rustc）のエラーメッセージは、他言語と比べて群を抜いて親切です。「どこが」「なぜ」ダメで、「どう直せばよいか」まで提案してくれます。初学者は借用や所有権のエラーで何度もつまずきますが、その都度メッセージを丁寧に読むのが最短の上達法です。エラーを敵ではなく先生として扱いましょう。

02

変数・可変性・シャドーイング

▼

💡 「変わらないのが普通」という思想

なぜ Rust の変数はデフォルトで書き換え不可なのか？
多くの言語では変数は自由に再代入できますが、それは「いつどこで値が変わるか分からない」という読みにくさとバグの温床でもあります。Rust は逆に「不変（immutable）が既定、可変は例外」という立場を取ります。変更したいときだけ mut を明示することで、コードを読む人は「mut が無い変数は最後まで値が変わらない」と信頼できます。これは大規模開発で効いてくる設計思想です。

let と mut と const

let で束縛した変数は不変で、再代入するとコンパイルエラーになります。変更するには let mut とします。const は定数で、mut を付けられず、型注釈が必須、かつコンパイル時に値が確定している必要があります。定数名は慣習的に大文字スネークケースで書きます。

Rust 
const MAX_POINTS: u32 = 100_000; // 定数は型必須・不変

fn main() {
    let x = 5;
    // x = 6;      // ❌ 不変なのでコンパイルエラー
    let mut y = 5;
    y = 6;         // ✅ mut があるので再代入できる
    println!("x={x}, y={y}, max={MAX_POINTS}");
}

シャドーイング：同じ名前で作り直す

Rust では同じ名前を let で再宣言でき、後の宣言が前を覆い隠します。これをシャドーイングと呼びます。mut の再代入と似て非なるもので、シャドーイングは新しい変数を作り直すため型を変えられます。文字列で受け取った入力を数値へ変換し、同じ名前で束縛し直す、といった用途が典型です。

Rust 
fn main() {
    let spaces = "   ";        // &str 型（文字列）
    let spaces = spaces.len(); // usize 型（数値）に変わる
    // mut では型を変えられないが、シャドーイングなら可能
    println!("空白は {spaces} 文字");
}

💼 Tips：mut を付けるのは「必要になってから」

最初はすべて let（不変）で書き、コンパイラに「ここは可変にしないと駄目」と言われてから mut を足すのが良い習慣です。使わない mut があると警告が出るのも Rust の親切な点で、「本当に変更が必要な箇所」だけが可視化されます。

03

データ型：スカラ型と複合型

▼

💡 型を意識することがバグを減らす

Rust は静的型付け言語で、すべての値の型がコンパイル時に決まります。
多くの場面では型推論が効くため注釈を省けますが、「どんな型が流れているか」を意識することが安全なコードの前提です。特に整数は「符号の有無」と「ビット幅」で複数の型に分かれており、範囲を超えるとどうなるかを知っておく必要があります。ここを曖昧にすると、他言語では気づきにくい桁あふれ（オーバーフロー）バグを生みます。

スカラ型：整数・浮動小数点・論理・文字

基本の型は 4 種類です。整数型（i32, u32, usize など。i は符号付き、u は符号なし、数字はビット幅）、浮動小数点型（f32, f64）、論理型（bool）、文字型（char。Unicode 1 文字でシングルクォート）。既定は整数が i32、浮動小数点が f64 です。

Rust 
fn main() {
    let a: i32 = -42;      // 符号付き整数
    let b: f64 = 3.14;     // 浮動小数点
    let flag: bool = true; // 論理値
    let letter: char = 'あ'; // Unicode 1 文字
    println!("{a} {b} {flag} {letter}");
}

複合型：タプルと配列

複数の値をまとめる基本型が 2 つあります。タプルは異なる型を固定個数まとめ、分解（destructuring）で取り出せます。配列は同じ型を固定個数並べたもので、[i32; 3] のように「型と長さ」が型の一部になります。長さが実行時に変わるデータには、後の章で学ぶ可変長の Vec を使います。

Rust 
fn main() {
    let tup: (i32, f64, char) = (500, 6.4, 'r');
    let (x, y, z) = tup;    // 分解して取り出す
    let arr = [10, 20, 30]; // [i32; 3]
    println!("{x} {y} {z} 先頭={}", arr[0]);
}

💼 Tips：オーバーフローは黙って起きない

整数には表現範囲があり（u8 は 0〜255）、超える演算をオーバーフローと呼びます。デバッグビルドでは実行時に panic してバグを気づかせ、リリースビルドでは性能優先でラップアラウンド（255+1 が 0）します。この挙動に頼らず意図を明示したいときは checked_add（あふれたら None）や wrapping_add（明示的にラップ）を使いましょう。

04

関数と制御フロー：式指向の言語

▼

💡 「文」と「式」の区別が Rust 理解の鍵

Rust は式指向（expression-oriented）言語です。
文（statement）は値を返さず、式（expression）は値を返します。関数の最後に書いた式を「セミコロンなし」にすると、それが戻り値になります。ここで初学者が最も引っかかるのが、うっかりセミコロンを付けて式を文に変えてしまい「値が返らない」エラーを出すケースです。逆にこの性質を活かすと、if やブロックが値を返せるため、簡潔で安全なコードが書けます。

関数定義と末尾式による戻り値

関数は fn で定義し、引数には型注釈が必須、戻り値の型は -> の後ろに書きます。return も使えますが、慣習では末尾式（セミコロンなし）で返します。セミコロンを付けると文になり値を返さなくなる点に注意してください。

Rust 
fn add(a: i32, b: i32) -> i32 {
    a + b // セミコロンなし = これが戻り値
}

fn main() {
    println!("{}", add(2, 3)); // 5
}

if 式とループ（loop / while / for）

if は式なので値を返して代入できます（三項演算子が無いのはこのため）。条件は必ず bool 型で、整数を真偽扱いする暗黙変換はありません。繰り返しは loop（無限。break に値を添えて返せる）、while、for の 3 種。範囲外アクセスの危険がない for 要素 in コレクション が最も推奨されます。

Rust 
fn main() {
    let cond = true;
    let number = if cond { 5 } else { 6 }; // if が値を返す
    println!("number = {number}");

    for i in 1..4 { // 1, 2, 3（終端は含まない）
        println!("i = {i}");
    }
}

💼 Tips：早期 return より式で表現する

Rust では「分岐の結果を変数に束縛する」書き方が自然です。match 式（後の章）と組み合わせると、状態ごとの値を漏れなく 1 つの式で表現でき、可読性と安全性が上がります。まずは「ブロックや if が値を返す」感覚に慣れましょう。

🔶 中級編 — 所有権システムを制する

Rust 最大の関門である所有権・借用・スライスと、構造体によるデータ設計、コレクションの実践的な使い方を学びます。ここを越えれば Rust の景色が変わります。

05

所有権：Rust の核心概念

▼

💡 ここが Rust 最大の山場（そして価値）

所有権（ownership）は Rust を Rust たらしめる中心概念です。
他言語出身者が最初に戸惑うのがここですが、乗り越えれば「なぜ Rust が GC なしで安全なのか」が腑に落ちます。ルールはたった 3 つ。(1) すべての値は所有者と呼ばれる 1 つの変数を持つ。(2) 所有者は同時に 1 つだけ。(3) 所有者がスコープを抜けると値は自動破棄（drop）される。この規則により、メモリの解放タイミングがコンパイル時に一意に決まり、解放漏れも二重解放も原理的に起きません。

スコープと自動解放（drop）

ヒープに確保される String のような値は、所有者がスコープを抜ける瞬間に自動的に drop され、メモリが解放されます。C の free を手で呼ぶ必要も、GC の回収を待つ必要もありません。「解放を書き忘れる」というバグのクラス全体が消滅します。

Rust 
fn main() {
    {
        let s = String::from("hello"); // s が所有者
        println!("{s}");
    } // ここで s がスコープを抜け、自動で drop（解放）される
}

ムーブ：所有権の移動

String のようなヒープデータを別の変数へ代入すると、コピーではなく所有権の移動（move）が起きます。移動後、元の変数は無効になり、使うとコンパイルエラーです。これは C++ の浅いコピーで二重解放が起きる問題を言語仕様で封じたものです。整数など単純でスタックに載る型は Copy なので、代入しても元が有効なままです。関数に値を渡すときも同様に所有権が移動します。

Rust 
fn main() {
    let s1 = String::from("hi");
    let s2 = s1;        // 所有権が s2 へ移動
    // println!("{s1}"); // ❌ s1 はもう無効

    let x = 5;
    let y = x;          // i32 は Copy。x も有効なまま
    println!("{s2} / {x} {y}");
}

💼 Tips：借用チェッカと戦わず、対話する

所有権エラーが出たら、「コンパイラが将来のクラッシュを先回りで防いでいる」と捉えましょう。多くの場合、次章で学ぶ「参照（借用）」を使えば所有権を移さずに解決できます。ムーブか借用か——この判断が Rust コーディングの日常です。

06

参照と借用：所有権を移さず値を使う

▼

💡 データ競合を「コンパイル時に」根絶する

所有権を毎回やり取りするのは煩雑です。そこで「借用（borrow）」があります。
参照（&）を使うと、所有権を移さずに値を貸し出せます。しかも Rust は借用に厳格なルールを課します——「複数の読み取り専用参照はOK、書き換え可能な参照はただ 1 つだけ」。これは並行処理でデータ競合が起きる条件そのものを排除する設計で、マルチスレッドのバグをコンパイル時に潰せる理由です。

不変参照 & による借用

& を付けると値を借用でき、所有権は元の変数に残ります。関数に &String を渡せば、関数は値を読めますが所有権は奪わないため、呼び出し元は引き続き使えます。参照は借り物なので、参照がスコープを抜けても元の値は drop されません。

Rust 
fn length(s: &String) -> usize {
    s.len()
} // s は参照なので、ここで元の値は drop されない

fn main() {
    let s = String::from("hello");
    let n = length(&s);   // 借用して渡す
    println!("{s} の長さは {n}"); // s はまだ有効
}

可変参照 &mut と借用規則

借用先を変更するには可変参照 &mut を使い、元の変数も mut である必要があります。中心的な規則はこうです：ある時点で「複数の不変参照」か「ただ 1 つの可変参照」のどちらか一方だけが許されます。読むだけなら何人でも、書く人がいるなら他は誰もアクセスできない、という直感に対応します。

Rust 
fn push_world(s: &mut String) {
    s.push_str(", world");
}

fn main() {
    let mut s = String::from("hello");
    push_world(&mut s); // 可変参照で貸し出す
    println!("{s}");    // hello, world
}

💼 Tips：ライフタイムは「参照が生きる期間」

参照は元データより長生きできません。これを保証するのがライフタイムで、多くは推論され明示不要です。初学段階では「借用した参照を、元データが消えた後に使わない」とだけ意識すれば十分です。複雑なライフタイム注釈が必要になるのは、もっと後の話です。

07

スライスと文字列：String と &str

▼

💡 2 種類の文字列に混乱しないために

Rust の文字列には String と &str の 2 つがあり、初学者を最初に悩ませます。
違いは「所有するか、借用するか」です。String はヒープに確保される所有権付きの可変文字列。&str（文字列スライス）は既存の文字列の一部を所有せずに参照するビューです。ソースコード中のリテラル "abc" の型は &str です。この区別は所有権・借用の応用そのものなので、前章までの理解がそのまま活きます。

スライス：所有せずに一部を参照する

スライスはコレクションの一部を「所有せずに参照する」ビューで、開始位置と長さを持ちコピーを伴いません。文字列スライスは範囲構文 [開始..終了] で切り出します（開始は含み終了は含まない、バイト単位）。スライスは借用なので、生きている間は元データを可変で書き換えられません。

Rust 
fn main() {
    let s = String::from("hello world");
    let hello = &s[0..5];  // "hello"
    let world = &s[6..11]; // "world"
    println!("{hello} / {world}");
}

関数引数は &str で受けるのが定石

実用上の指針は明快です。関数の文字列引数は原則 &str で受け取ると、String からもリテラルからも借用して渡せて柔軟です。所有して保持・変更したいときだけ String を使います。「借りるだけなら &str、持つなら String」と覚えましょう。

Rust 
// &str で受けると String もリテラルも渡せる
fn greet(name: &str) -> String {
    format!("こんにちは、{name}")
}

fn main() {
    let owned = String::from("太郎");
    println!("{}", greet(&owned)); // String を借用
    println!("{}", greet("花子"));  // リテラル(&str)
}

💼 Tips：UTF-8 とバイト境界

Rust の文字列は UTF-8 です。日本語のようなマルチバイト文字を含む文字列を [0..1] のように文字の途中で切ると、実行時に panic します。バイト単位で切っている点を忘れないでください。文字単位で扱うなら chars() を使います。

08

構造体とメソッド

▼

💡 データと振る舞いをまとめる

構造体（struct）は、関連する複数の値に名前を付けて 1 つの型にまとめる仕組みです。
Rust にはクラスがありませんが、構造体と impl ブロックの組み合わせが、他言語のクラスに近い役割を果たします。データ（フィールド）と振る舞い（メソッド）を分けて定義するのが特徴で、これにより「データ構造」と「操作」の関心が自然に分離されます。

構造体の定義とインスタンス生成

各フィールドに名前と型を与えます。生成時はフィールド名を指定して値を入れ、アクセスはドット記法で行います。インスタンス全体を可変にするには let mut で束縛します（Rust では特定フィールドだけを可変にはできず、インスタンス単位です）。

Rust 
struct User {
    name: String,
    age: u32,
    active: bool,
}

fn main() {
    let user = User {
        name: String::from("Alice"),
        age: 30,
        active: true,
    };
    println!("{} ({}) active={}", user.name, user.age, user.active);
}

メソッドと関連関数

impl ブロックに振る舞いを定義します。第一引数が &self（自身への参照）のものをメソッドと呼び、インスタンス.メソッド() で呼びます。self を取らないものは関連関数と呼ばれ、型名::関数() で呼びます。コンストラクタ相当の new を関連関数として定義するのが慣習です。デバッグ表示は #[derive(Debug)] を付けると {:?} で中身を確認できます。

Rust 
#[derive(Debug)]
struct Rectangle { width: u32, height: u32 }

impl Rectangle {
    fn new(w: u32, h: u32) -> Self { // 関連関数
        Self { width: w, height: h }
    }
    fn area(&self) -> u32 {           // メソッド
        self.width * self.height
    }
}

fn main() {
    let rect = Rectangle::new(3, 4);
    println!("{:?} 面積={}", rect, rect.area());
}

💼 Tips：まず #[derive(Debug)] を付ける

構造体を作ったら反射的に #[derive(Debug)] を付けておくと、println!("{:?}", val) や {:#?}（整形表示）で中身を即確認でき、デバッグが劇的に楽になります。比較や複製が必要なら Clone, PartialEq も derive で自動生成できます。

09

コレクションとイテレータ — Vec・HashMap を実務レベルで使う

▼

💡 for ループより「イテレータチェーン」が Rust の母国語

Rust のコレクション処理は、イテレータのメソッドチェーンで書くのが慣用です。 map や filter は「怠惰（lazy）」で、collect などの消費メソッドが呼ばれるまで一切実行されません。しかも驚くべきことに、チェーンはコンパイル時に単純なループへ展開（ゼロコスト抽象化）されるため、手書きの for ループと同速です。「高級な書き方をしても遅くならない」のが Rust の設計哲学です。

Vec：伸縮する配列

Vec<T> は最も使うコレクションです。vec! マクロで生成し、push で追加します。所有権のルールはそのまま適用される——たとえば要素を借用している間は push できません（拡張時のメモリ再配置で参照が無効になるのを防ぐため）。

Rust 
fn main() {
    let scores = vec![55, 82, 91, 68, 77];

    // filter → map → collect のチェーン：合格者の得点を10点加算して収集
    let passed: Vec<i32> = scores
        .iter()                      // 借用のイテレータ（&i32 を返す）
        .filter(|&&s| s >= 70)       // 条件を満たすものだけ通す
        .map(|&s| s + 10)            // 変換する
        .collect();                  // Vec に集める（ここで初めて実行）

    println!("{:?}", passed);        // [92, 101, 87]

    // sum / max / count などの集約も一級市民
    let total: i32 = scores.iter().sum();
    let best = scores.iter().max(); // Option<&i32>（空かもしれないので）
    println!("合計 {total}, 最高 {best:?}");
}

HashMap：キーと値の対応表

HashMap<K, V> は use std::collections::HashMap; で導入します。頻出パターンは entry API——「キーがあれば更新、なければ初期値を挿入」を 1 行で表現でき、単語カウントのようなロジックが簡潔に書けます。

Rust 
use std::collections::HashMap;

fn main() {
    let text = "apple banana apple cherry banana apple";

    let mut counts: HashMap<&str, i32> = HashMap::new();
    for word in text.split_whitespace() {
        // entry: キーが無ければ 0 を挿入し、可変参照を返す
        *counts.entry(word).or_insert(0) += 1;
    }

    // get は Option を返す：存在しないキーでもパニックしない
    println!("apple = {:?}", counts.get("apple"));   // Some(3)
    println!("grape = {:?}", counts.get("grape"));   // None
}

iter / iter_mut / into_iter の使い分け

イテレータの作り方は 3 種類あり、所有権の扱いが異なります。iter() は不変借用（読むだけ）、iter_mut() は可変借用（その場で書き換え）、into_iter() は所有権ごと消費（元のコレクションは使えなくなる）。「読むだけなら iter、変換して新しい Vec を作るなら into_iter」が基本の選び方です。

💼 Tips：インデックスアクセスは避ける

v[10] は範囲外で即パニックします。安全に取り出すなら v.get(10)（Option が返る）を使い、そもそも添字が不要になるようイテレータで書くのが Rust 流です。「添字ループを書きたくなったら、イテレータのメソッドで表現できないか考える」を習慣にしてください。

🔴 上級編 — 型システムで設計する

列挙型とパターンマッチング、Result によるエラー処理、トレイトとジェネリクス、ライフタイム、そしてテストまで。Rust の型システムを「設計の道具」として使いこなします。

10

列挙型・Option・パターンマッチング

▼

💡 「値が無いかもしれない」を型で表す

Rust には null がありません。
null は「10 億ドルの過ち」と呼ばれ、多くの言語で null 参照エラーを生み続けてきました。Rust は代わりに列挙型 Option<T> を使い、「値がないかもしれない」ことを型システムに強制的に明示させます。中の値を使うには必ず場合分けが必要なため、「無い値をうっかり触る」ことがコンパイル時に不可能になります。

列挙型 enum と Option

列挙型（enum）は「取りうる値がいくつかの種類のいずれか」を表し、各バリアントが異なるデータを保持できます。標準の Option<T> は Some(値) か None のどちらかで、値の有無を表現します。

Rust 
enum Shape {
    Circle(f64),          // 半径
    Rectangle(f64, f64),  // 幅・高さ
}

fn main() {
    let some_number: Option<i32> = Some(5);
    let no_number: Option<i32> = None;
    println!("{:?} {:?}", some_number, no_number);
    let s = Shape::Circle(2.0);
    if let Shape::Circle(r) = s {
        println!("半径 {r} の円");
    }
}

match による網羅的な分岐

match は値をパターンに照合して分岐します。最大の特徴は網羅性チェックで、enum の全バリアントを扱わないとコンパイルエラーになり、case の書き漏らしを原理的に防ぎます。1 つのパターンだけ扱うなら if let、残りをまとめるなら _（ワイルドカード）を使います。

Rust 
fn describe(n: Option<i32>) -> String {
    match n {
        Some(x) if x > 0 => format!("正の数 {x}"),
        Some(x) => format!("0 以下の数 {x}"),
        None => String::from("値なし"),
    }
}

fn main() {
    println!("{}", describe(Some(7)));
    println!("{}", describe(None));
}

💼 Tips：unwrap の乱用に注意

Option の中身は unwrap() で取り出せますが、None のとき panic します。学習中は便利でも、実務では match や if let、unwrap_or(既定値) で安全に扱うのが基本です。「とりあえず unwrap」は本番クラッシュの温床です。

11

エラー処理：panic と Result

▼

💡 エラーを「握りつぶせない」設計

Rust はエラーを 2 種類に分け、どちらも無視できないようにしています。
1 つは回復不能なエラー（panic!）でプログラムを止めます。もう 1 つは回復可能なエラーで、戻り値 Result<T, E> として表現します。例外を投げっぱなしにできる言語と違い、Rust では失敗しうる関数の戻り値が型に現れるため、呼び出し側は必ずエラーの扱いを書くことになります。これが堅牢さの源です。

回復可能なエラー：Result<T, E>

Result は成功なら Ok(値)、失敗なら Err(エラー) を返す列挙型です。ファイルが開けない、数値変換に失敗した、といった「起こりうる失敗」を表現します。match で場合分けして扱うため、エラー処理が呼び出し側に強制されます。

Rust 
fn parse(s: &str) -> Result<i32, std::num::ParseIntError> {
    s.parse::<i32>()
}

fn main() {
    match parse("42") {
        Ok(n) => println!("成功: {n}"),
        Err(e) => println!("失敗: {e}"),
    }
}

? 演算子でエラーを伝播する

毎回 match を書くのは冗長です。? 演算子は、式が Ok なら中の値を取り出して続行し、Err ならその場で関数から Err を早期リターンします。「成功なら値を取り出し、失敗なら呼び出し元へ委譲」を 1 文字で表せます。? を使う関数自身の戻り値も Result である必要があります。

Rust 
fn sum_two(a: &str, b: &str) -> Result<i32, std::num::ParseIntError> {
    let x = a.parse::<i32>()?; // 失敗なら即 Err を返す
    let y = b.parse::<i32>()?;
    Ok(x + y)
}

fn main() {
    println!("{:?}", sum_two("10", "20")); // Ok(30)
    println!("{:?}", sum_two("10", "x"));  // Err(...)
}

💼 Tips：panic と Result の使い分け

panic! は「バグや続行不能な想定外」に限定し、ユーザー入力エラーやファイル不在のような「起こりうる失敗」は Result で扱うのが原則です。ライブラリでは基本 Result を返し、呼び出し側に判断を委ねます。実務では anyhow や thiserror といったクレートでエラー処理をさらに簡潔にできます。

12

トレイトとジェネリクス — 共有される振る舞いを定義する

▼

💡 トレイトは「継承なきインターフェース」

Rust にはクラス継承がありません。 代わりに「この型は◯◯できる」という能力（振る舞い）をトレイトとして定義し、任意の型に実装します。Java の interface や TypeScript の interface に似ていますが、決定的な違いは既存の型（他人が作った型でも）に後からトレイトを実装できること。「データ（struct）と振る舞い（trait）を分離し、必要な組み合わせを合成する」——これが Rust の設計スタイルです。

トレイトの定義と実装

trait ブロックでメソッドのシグネチャ（と必要ならデフォルト実装）を定義し、impl トレイト名 for 型名 で実装します。デフォルト実装があるメソッドは実装を省略できます。

Rust 
trait Summary {
    fn title(&self) -> String;

    // デフォルト実装：実装側で上書きしなければこれが使われる
    fn summarize(&self) -> String {
        format!("【{}】を読む", self.title())
    }
}

struct Article { headline: String }
struct Tweet { user: String, text: String }

impl Summary for Article {
    fn title(&self) -> String { self.headline.clone() }
}

impl Summary for Tweet {
    fn title(&self) -> String { format!("@{} のツイート", self.user) }
    fn summarize(&self) -> String { self.text.clone() }  // 上書き
}

ジェネリクスとトレイト境界：「Summary を実装した型なら何でも」

ジェネリック関数の型パラメータに「このトレイトを実装していること」という条件（トレイト境界）を付けられます。C++ のテンプレートと違い、境界に書いた能力しか使えないため、エラーは呼び出し側ではなく定義側で明快に出ます。しかもコンパイル時に型ごとの専用コードが生成される（単相化）ため、実行時コストはゼロです。

Rust 
// impl Trait 構文：Summary を実装した型なら何でも受け取れる
fn notify(item: &impl Summary) {
    println!("新着: {}", item.summarize());
}

// ジェネリクス + トレイト境界の明示形（複数条件は + で連結）
fn largest<T: PartialOrd + Copy>(list: &[T]) -> T {
    let mut max = list[0];
    for &item in list {
        if item > max { max = item; }
    }
    max
}

fn main() {
    let tweet = Tweet { user: "rustlang".into(), text: "1.80 リリース".into() };
    notify(&tweet);
    println!("{}", largest(&[3, 7, 2]));      // i32 でも
    println!("{}", largest(&[1.5, 0.2]));     // f64 でも動く
}

💼 Tips：derive はトレイト自動実装の糖衣

構造体に付けてきた #[derive(Debug, Clone, PartialEq)] は、実は Debug・Clone・PartialEq という標準トレイトの実装をコンパイラに自動生成させる指示です。「Rust の機能の多くはトレイトでできている」——println!("{}") は Display トレイト、+ 演算子は Add トレイト、for ループは Iterator トレイト。この見方ができると標準ライブラリのドキュメントが一気に読めるようになります。

13

ライフタイム入門 — 参照の「寿命」を明示する

▼

💡 ライフタイムは新概念ではなく「既に守ってきたルールの名前」

参照は、指している値より長生きしてはならない——借用のレッスンで学んだこのルールを、コンパイラは全ての参照の「寿命（ライフタイム）」を追跡して検査しています。ほとんどの場合は自動推論されますが、「複数の参照を受け取って参照を返す関数」では、戻り値の寿命がどの引数由来なのか判断できず、人間がヒントを書く必要があります。それが 'a という注釈です。注釈は寿命を変えるのではなく、引数と戻り値の寿命の「関係」をコンパイラに伝えるだけです。

なぜ注釈が必要になるのか

次の関数はコンパイルできません。戻り値の &str が x 由来か y 由来か、コンパイラには分からないためです。<'a> を付けて「戻り値は両引数のうち短い方の寿命に縛られる」と宣言すれば解決します。

Rust 
// ❌ コンパイルエラー：戻り値の寿命が特定できない
// fn longest(x: &str, y: &str) -> &str { ... }

// ✅ 'a：「x と y の寿命が重なる範囲でだけ、戻り値も有効」という契約
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("こんにちは、世界");
    let result;
    {
        let s2 = String::from("やあ");
        result = longest(s1.as_str(), s2.as_str());
        println!("長い方: {result}");   // ✅ s2 が生きている間は使える
    }
    // println!("{result}");  // ❌ s2 が破棄された後は使えない
    //                            —— コンパイラが契約違反を検出する
}

省略規則：だから今まで書かずに済んだ

「引数の参照が 1 つだけなら、戻り値はその寿命を引き継ぐ」「&self があればそれを引き継ぐ」という省略規則（lifetime elision）があるため、単純な関数では注釈が不要でした。つまり fn first_word(s: &str) -> &str は暗黙に fn first_word<'a>(s: &'a str) -> &'a str と解釈されています。構造体が参照を保持する場合も struct Excerpt<'a> { part: &'a str } のように注釈が必要です。

💼 Tips：初学者は「注釈を書かない設計」に逃げてよい

ライフタイム注釈が複雑になってきたら、参照ではなく所有権ごと渡す（String を返す・clone() する）設計に切り替えるのは、恥ではなく実践的な判断です。クローンのコストが問題になることは稀で、まず動くコードを書き、プロファイリングで必要になってから参照に最適化すれば十分です。

14

テストを書く — cargo test で品質を守る

▼

💡 テストが言語に標準装備されている

Rust では、テストフレームワークの選定も導入も不要です。 #[test] 属性を付けた関数を書いて cargo test を叩くだけ。テストは製品コードと同じファイルに書ける（モジュールの非公開関数も検証できる）ため、「実装とテストを往復するリズム」が非常に軽い。コンパイラが型とメモリ安全を保証し、テストがロジックの正しさを保証する——2 段構えで品質を守るのが Rust の開発スタイルです。

基本形：#[cfg(test)] モジュールと assert 系マクロ

慣例として、ファイル末尾に #[cfg(test)] を付けた tests モジュールを置きます。この属性により、テストコードは通常ビルドから完全に除外されます。検証には assert!（真偽）、assert_eq!（等値）、assert_ne!（非等値）を使います。

Rust 
fn calc_bmi(weight_kg: f64, height_m: f64) -> Result<f64, String> {
    if height_m <= 0.0 {
        return Err("身長は正の値である必要があります".to_string());
    }
    Ok(weight_kg / (height_m * height_m))
}

#[cfg(test)]
mod tests {
    use super::*;   // 親モジュールの関数を取り込む

    #[test]
    fn 正常な入力でbmiを計算できる() {
        let bmi = calc_bmi(60.0, 1.7).unwrap();
        assert!((bmi - 20.76).abs() < 0.01);  // 浮動小数は誤差込みで比較
    }

    #[test]
    fn 身長ゼロはエラーになる() {
        let result = calc_bmi(60.0, 0.0);
        assert!(result.is_err());
    }

    #[test]
    #[should_panic(expected = "index out of bounds")]
    fn 範囲外アクセスはパニックする() {
        let v = vec![1, 2, 3];
        let _ = v[99];
    }
}

cargo test の使い方と結合テスト

cargo test は全テストを並列実行します。名前の部分一致で絞り込みもできます（cargo test bmi）。また、プロジェクト直下の tests/ ディレクトリに置いたファイルは「結合テスト」として扱われ、外部ユーザーと同じ視点で公開 API だけを検証します。さらにドキュメンテーションコメント内のコード例も cargo test が自動実行する（doctest）ため、「ドキュメントのサンプルが古くて動かない」問題が構造的に起きません。

Bash 
# 全テストを実行
cargo test

# 名前に "bmi" を含むテストだけ実行
cargo test bmi

# println! の出力を表示しながら実行（デバッグに便利）
cargo test -- --nocapture

💼 Tips：Result を返すテストで ? を使う

テスト関数は fn test() -> Result<(), String> のように Result を返すこともできます。すると unwrap() の連打ではなく ? 演算子でエラーを伝播でき、本編で学んだエラー処理の作法がテストでもそのまま活きます。「テストも普通の Rust コード」——この一貫性が Rust のテスト文化の強さです。
