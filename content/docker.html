01

コンテナとは何か — 仮想マシンとの違いを根本から理解する

▼

💡 コンテナは「軽量な隔離環境」— OSカーネルを共有するからこそ速い

なぜコンテナが必要なのか？

開発者なら誰もが経験する 「自分の環境では動くのに、本番サーバーで動かない」 問題。原因はPythonのバージョン差異、ライブラリの依存関係、OS固有の設定など多岐にわたります。コンテナはこの問題を 「アプリケーション + その実行に必要な全て」をパッケージ化 することで解決します。

従来の仮想マシン（VM）も同じ問題を解決できますが、VMはゲストOS全体を起動するため 数分かかり、数GBのリソースを消費 します。コンテナはホストOSのカーネルを共有するため、数秒で起動し、数十MBで済む のです。

アーキテクチャ比較
┌──── 仮想マシン（VM） ────┐    ┌──── コンテナ ────────────┐
│ ┌─App─┐ ┌─App─┐ ┌─App─┐│    │ ┌─App─┐ ┌─App─┐ ┌─App─┐│
│ ├─Lib─┤ ├─Lib─┤ ├─Lib─┤│    │ ├─Lib─┤ ├─Lib─┤ ├─Lib─┤│
│ ├Guest┤ ├Guest┤ ├Guest┤│    │ └─────┘ └─────┘ └─────┘│
│ │ OS  │ │ OS  │ │ OS  ││    │   Container Runtime     │
│ └─────┘ └─────┘ └─────┘│    │     (Docker Engine)      │
│    Hypervisor           │    ├─────────────────────────┤
├─────────────────────────┤    │     ホスト OS カーネル    │  ← 共有！
│     ホスト OS            │    ├─────────────────────────┤
├─────────────────────────┤    │      ハードウェア         │
│      ハードウェア         │    └─────────────────────────┘
└─────────────────────────┘
起動: 数分 / サイズ: 数GB        起動: 数秒 / サイズ: 数十MB

🔍 カーネル共有の仕組み — namespaces と cgroups

コンテナの「隔離」は Linux カーネルの2つの機能で実現されます：

namespaces（名前空間）: プロセス、ネットワーク、ファイルシステム等を隔離。コンテナAからコンテナBのプロセスは見えない。

cgroups（コントロールグループ）: CPU、メモリ等のリソース使用量を制限。1つのコンテナが全リソースを占有しないように制御。

💼 Tips：実務で遭遇する現場の勘所

Docker を使わない方が良い場面: GUI アプリケーション、低レイテンシが求められるリアルタイムシステム、Windows ネイティブの .NET Framework アプリ。また、チームに Docker の知見がない場合、学習コストが開発速度を上回ることもあります。「全てを Docker 化すべき」は誤りです。

02

Docker の基本概念 — イメージ・コンテナ・レジストリ

▼

💡 イメージ = レシピ（設計図）、コンテナ = 料理（実体）

Docker の3大要素

イメージ（Image）: コンテナの「設計図」。OS + ライブラリ + アプリが層（レイヤー）構造で積み重なったもの。読み取り専用。

コンテナ（Container）: イメージから作られた「実行中のインスタンス」。書き込み可能なレイヤーが1層追加される。停止すると変更は消える（揮発性）。

レジストリ（Registry）: イメージの保管・配布場所。Docker Hub（公開）、Amazon ECR / Google GCR（プライベート）。

SHELL — Docker 基本コマンド
# ── イメージの取得と管理 ──
# Docker Hub から Python 3.12 の公式イメージを取得
# "python" がイメージ名、"3.12-slim" がタグ（バージョン）
# slim は不要なパッケージを省いた軽量版（推奨）
docker pull python:3.12-slim

# ローカルに保存されているイメージ一覧を表示
docker images

# ── コンテナの起動と操作 ──
# イメージからコンテナを起動して中に入る（対話モード）
# -it = 標準入力を開いてターミナルに接続
# --rm = 終了時にコンテナを自動削除（ゴミが残らない）
docker run -it --rm python:3.12-slim bash

# バックグラウンドで Web サーバーを起動
# -d = デタッチモード（バックグラウンド実行）
# -p 8080:80 = ホストの8080番ポートをコンテナの80番に転送
# --name = コンテナに名前を付ける（管理しやすい）
docker run -d -p 8080:80 --name my-nginx nginx:alpine

# ── コンテナの状態確認と停止 ──
docker ps              # 実行中のコンテナ一覧
docker ps -a           # 停止中も含めた全コンテナ一覧
docker logs my-nginx   # コンテナのログを表示
docker stop my-nginx   # コンテナを停止
docker rm my-nginx     # コンテナを削除

💼 Tips：実務で遭遇する現場の勘所

Docker Hub の「公式イメージ」の見分け方: 公式イメージは python, nginx, postgres のようにユーザー名なしで表示されます。someone/python のように / が入っているものは非公式です。セキュリティリスクを避けるため、本番では必ず公式イメージ（Official Image）のバッジを確認してください。

03

Dockerfile — イメージ設計の技術

▼

💡 Dockerfile は「再現可能な環境構築の手順書」

Dockerfile — Python FastAPI アプリの例
# ── ステージ1: ビルド環境 ──
# マルチステージビルド: 最終イメージにビルドツールを含めない
FROM python:3.12-slim AS builder

# 作業ディレクトリを設定（以降のコマンドはここで実行される）
WORKDIR /app

# まず依存関係ファイルだけをコピー → pip install
# なぜ先にこれだけ？ → ソースコード変更時にpip installの
# キャッシュが効くため。ビルド時間が大幅に短縮される。
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# ── ステージ2: 本番用の軽量イメージ ──
FROM python:3.12-slim

# セキュリティ: root ではなく一般ユーザーで実行
RUN useradd --create-home appuser
WORKDIR /app

# ビルドステージからpipパッケージだけをコピー
# ビルドツール(gcc等)は持ち込まない → イメージサイズ削減
COPY --from=builder /root/.local /home/appuser/.local
COPY . .

# 一般ユーザーに切り替え（rootで実行しない = セキュリティの基本）
USER appuser
ENV PATH=/home/appuser/.local/bin:$PATH

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

🔍 レイヤーキャッシュの仕組み

Dockerfile の各命令（FROM, COPY, RUN 等）は1つのレイヤーを生成します。Docker は変更がないレイヤーをキャッシュして再利用します。そのため、変更頻度の低い命令（pip install 等）を上に、変更頻度の高い命令（COPY . .）を下に書くのが最適化の鉄則です。

💼 Tips：実務で遭遇する現場の勘所

.dockerignore の作成を忘れない: .git, node_modules, __pycache__, .env 等をビルドコンテキストから除外しないと、機密情報がイメージに含まれたり、ビルドが遅くなったりします。.gitignore と同じ感覚で必ず作成してください。

🔶 中級編 — 複数容器・永続化・現場の切り分け

Compose、ボリュームとネットワーク、開発〜本番の流れに加え、HEALTHCHECK・イメージ肥大化対策・ログ／exec デバッグを固めます。

04

docker compose — 複数コンテナの連携

▼

💡 現実のサービスは「Web + DB + キャッシュ」の組み合わせで動く

YAML — docker-compose.yml 実践構成
# docker compose up -d で全サービスが一括起動する
# docker compose down で全サービスを停止・削除

services:
  # ── Web アプリケーション ──
  web:
    build: .                        # カレントディレクトリの Dockerfile を使用
    ports:
      - "8000:8000"                 # ホスト:コンテナ のポートマッピング
    volumes:
      - .:/app                      # ソースコードをバインドマウント（ホットリロード用）
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/myapp
      - REDIS_URL=redis://cache:6379
    depends_on:                     # 起動順序の制御（注意点あり↓）
      db:
        condition: service_healthy  # ヘルスチェックが通るまで待つ
    restart: unless-stopped         # クラッシュ時に自動再起動

  # ── データベース ──
  db:
    image: postgres:16-alpine       # 公式の軽量版イメージ
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp
    volumes:
      - db_data:/var/lib/postgresql/data  # データを永続化
    healthcheck:                    # DB が「本当に準備完了」かチェック
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 5s
      timeout: 3s
      retries: 5

  # ── キャッシュサーバー ──
  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  db_data:                          # Named Volume（Docker が管理）

💼 Tips：実務で遭遇する現場の勘所

depends_on の罠: デフォルトの depends_on は「コンテナが起動した」だけを保証し、「アプリが準備完了した」ことは保証しません。PostgreSQL のコンテナは起動しても、接続を受け付けるまでに数秒かかります。condition: service_healthy + healthcheck の組み合わせで解決してください。

05

ボリュームとネットワーク — データ永続化と通信

▼

💡 コンテナは「使い捨て」— だからこそデータの置き場所が重要

なぜコンテナのデータは消えるのか？

コンテナはイミュータブル（不変）なイメージから起動し、変更は書き込み可能な一時レイヤーに記録されます。コンテナを削除するとこのレイヤーも消えます。DBのデータや、アップロードされたファイルを永続化するには ボリューム が必要です。

SHELL — ボリュームの種類と使い分け
# ── Named Volume（Docker が管理。本番DB向き） ──
docker volume create my_data
docker run -v my_data:/var/lib/postgresql/data postgres:16

# ── Bind Mount（ホストのディレクトリを直接マウント。開発向き） ──
# ソースコードの変更がリアルタイムにコンテナに反映される
docker run -v $(pwd):/app python:3.12

# ── ネットワーク ──
# 同じ docker compose 内のサービスは、サービス名でアクセスできる
# 例: web コンテナから db コンテナへは "db:5432" でアクセス
# これは Docker が自動的に DNS を設定しているため

💼 Tips：実務で遭遇する現場の勘所

最もよくある事故: docker compose down -v の -v フラグは Named Volume も削除 します。開発中にデータベースを含むボリュームを消してしまい、データが全て消失する事故が頻発します。-v は「何が削除されるか」を理解してから使ってください。

06

実務の Docker ワークフロー — 開発→テスト→本番

▼

💡 「開発で docker compose、本番は別」が現実解

SHELL — CI/CD パイプラインでの Docker 活用
# ── GitHub Actions での典型的なワークフロー ──
# 1. コードがプッシュされる
# 2. CI でテスト実行（Docker コンテナ内で）
# 3. Docker イメージをビルド
# 4. プライベートレジストリ（ECR/GCR）にプッシュ
# 5. 本番環境（ECS/Cloud Run）がイメージを取得して起動

# イメージのビルドとタグ付け
docker build -t myapp:latest -t myapp:v1.2.3 .

# プライベートレジストリにプッシュ
docker tag myapp:latest 123456789.dkr.ecr.ap-northeast-1.amazonaws.com/myapp:latest
docker push 123456789.dkr.ecr.ap-northeast-1.amazonaws.com/myapp:latest

💼 Tips：実務で遭遇する現場の勘所

docker-compose.yml を本番と開発で分ける: docker-compose.yml（共通）+ docker-compose.override.yml（開発用: バインドマウント、デバッグポート）+ docker-compose.prod.yml（本番用: ヘルスチェック、リソース制限）のように分離するのが定番パターンです。

.dockerignore とイメージ肥大化対策 — 速く・小さく・安全に

▼

💡 コピーした瞬間にイメージは太る

マルチステージビルドだけでは不十分です。 node_modules や .git、.env をビルド文脈に入れるとサイズ・秘密漏洩・キャッシュ無効化の三重苦になります。.dockerignore とレイヤ順の最適化は、CI 時間を直接削ります。

最低限 ignore する項目

VCS、依存のローカルコピー、秘密、ビルド成果、エディタ設定を外します。

Text 
.git
.gitignore
.env
.env.*
node_modules
dist
__pycache__
*.pyc
.venv
.pytest_cache
README.md
docs/

レイヤキャッシュを活かす順序

変わりにくいファイル（依存定義）を先に COPY → install、ソースは後。これだけで再ビルドが桁違いに速くなります。

Dockerfile 
FROM python:3.12-slim AS base
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

💼 Tips：docker history で太ったレイヤを見る

docker history IMAGE でどの命令が重いが一目瞭然です。感覚ではなく数値でダイエットしてください。

08

HEALTHCHECK — 「起動した」と「準備できた」を分ける

▼

💡 depends_on だけではアプリは待ってくれない

コンテナプロセスの起動と、ポートで応答できる状態は別物です。 DB や API に対し HEALTHCHECK（または Compose の healthcheck）を定義し、depends_on: condition: service_healthy で依存を待つと、起動レースのフレークが消えます。

Dockerfile 
HEALTHCHECK --interval=10s --timeout=3s --retries=5 --start-period=10s \
  CMD curl -fsS http://127.0.0.1:8000/health || exit 1
YAML 
services:
  api:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-fsS", "http://127.0.0.1:8000/health"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 10s
  web:
    depends_on:
      api:
        condition: service_healthy

💼 Tips：/health は軽く保つ

ヘルスチェックが重い外部 API を叩くと、負荷試験中に自分で自分を落とします。プロセス生存か、ローカル DB ping 程度に留めます。

09

コンテナのデバッグ — logs / exec / inspect

▼

💡 壊れた箱の中を、壊さずに覗く

本番相当の障害切り分けは、ほぼログと一時シェルです。 docker compose logs -f で時系列を追い、exec で中に入り、inspect でネットワークとマウントを確認します。Linux 章の ps / ss / curl がここで活きます。

Bash 
# ログ追従
docker compose logs -f api

# 稼働中コンテナでシェル（イメージに sh がある前提）
docker compose exec api sh
# curl localhost:8000/health
# ls -la /data

# 設定・IP・マウントの確認
docker inspect $(docker compose ps -q api) | head

# 終了済みの理由
docker compose ps -a
docker inspect --format '{{.State.ExitCode}} {{.State.Error}}' NAME

💼 Tips：exec できないときは run で同イメージを調査

起動即死するコンテナには docker compose run --rm --entrypoint sh api で入り、コマンドを手動実行して原因を特定します。

🔴 上級編 — 環境差を消し、本番耐性を上げる

セキュリティとオーケストレーション概論に加え、dev/prod の Compose 分離、リソース制限、BuildKit キャッシュまで。

10

Docker セキュリティ — 本番運用の鉄則

▼

💡 コンテナは「デフォルトで安全」ではない

Dockerfile — セキュリティのベストプラクティス
# ❌ 悪い例: root で実行、フルOSイメージ
FROM python:3.12
COPY . /app
CMD ["python", "app.py"]

# ✅ 良い例: 非root、最小イメージ、脆弱性スキャン対応
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

FROM gcr.io/distroless/python3-debian12
COPY --from=builder /root/.local /root/.local
COPY . /app
WORKDIR /app
# distroless: シェルすら含まない最小イメージ
# → 攻撃者がコンテナに侵入しても操作手段がない
CMD ["app.py"]

🔍 Secrets 管理の鉄則

環境変数にパスワードを直接書かない。 docker-compose.yml に POSTGRES_PASSWORD=mysecret と書くと、docker inspect で誰でも見られます。本番では Docker Secrets、HashiCorp Vault、クラウドの Secrets Manager（AWS/GCP）を使ってください。

💼 Tips：実務で遭遇する現場の勘所

Alpine vs Distroless: Alpine は軽量（5MB）でシェルが使えるためデバッグしやすい一方、musl libc を使うため一部のPythonパッケージが動かないことがあります。Distroless はシェルもパッケージマネージャもない究極の最小イメージで、攻撃面が最小ですがデバッグが困難です。開発は Alpine、本番は Distroless が現実的な使い分けです。

コンテナオーケストレーション概論 — Kubernetes への橋渡し

▼

💡 「コンテナ1個」から「コンテナ1000個」への道

なぜオーケストレーションが必要か

docker compose は1台のサーバーで複数コンテナを管理できますが、トラフィック増加で複数サーバーが必要になると対応できません。Kubernetes（K8s）は数百〜数千のコンテナを複数サーバーに分散配置し、自動スケーリング、自動復旧、ローリングアップデートを実現するプラットフォームです。

YAML — Kubernetes の基本リソース（概念理解用）
# Pod: コンテナの最小単位（通常1Pod = 1コンテナ）
# Deployment: Pod のレプリカ数やアップデート戦略を管理
# Service: Pod へのネットワークアクセスを提供

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 3            # 同じコンテナを3つ起動
  selector:
    matchLabels:
      app: my-api
  template:
    spec:
      containers:
      - name: api
        image: myapp:v1.2.3
        resources:
          limits:          # リソース制限（重要！）
            memory: "256Mi"
            cpu: "500m"

💼 Tips：実務で遭遇する現場の勘所

「Kubernetes は本当に必要か？」: K8s の学習・運用コストは非常に高いです。月間100万リクエスト以下、チーム5人以下なら、AWS ECS / Google Cloud Run / Fly.io のようなマネージドサービスの方が費用対効果が高いことが多いです。「K8s を使いこなすこと」を目的にしないでください。

compose override — 開発と本番でファイルを分ける

▼

💡 1 ファイルに全部書くと必ず腐る

共通定義を compose.yml に置き、開発用は compose.override.yml（自動マージ）または -f で重ねます。 ソースのバインドマウントやホットリロードは開発だけ、本番はイメージ内の成果物だけ、が基本です。

YAML 
# compose.yml（共通）
services:
  api:
    build: ./api
    environment:
      DATABASE_URL: sqlite:////data/app.db
    volumes:
      - api-data:/data
volumes:
  api-data:

# compose.override.yml（ローカル開発・自動で読まれる）
services:
  api:
    ports: ["8000:8000"]
    volumes:
      - ./api:/app
    command: uvicorn main:app --reload --host 0.0.0.0 --port 8000
Bash 
# 本番寄りの起動例（override を使わない）
docker compose -f compose.yml -f compose.prod.yml up -d --build

💼 Tips：秘密は override にも直書きしない

env_file: .env か実行環境のシークレットに寄せます。CI/CD 章の Environment secrets と同じ思想です。

リソース制限と restart — 近所迷惑を防ぐ

▼

💡 1 コンテナがホストを食い潰さないようにする

メモリリークしたアプリは、制限が無いとホストごと落とします。 CPU/メモリ上限と restart ポリシーを Compose で明示し、落ちたときの振る舞い（常に再起動するか、失敗時だけか）を決めます。

YAML 
services:
  api:
    image: myapi:1.2.3
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: "0.50"
          memory: 256M
        reservations:
          memory: 128M
    # Swarm 以外でも、近年の Compose は上記を解釈する実装が多い。
    # エンジンによっては mem_limit / cpus の旧キーを使う。
Bash 
# 単体実行での制限例
docker run --rm -m 256m --cpus 0.5 myapi:1.2.3

💼 Tips：OOM で死んだかログを見る

突然死のときは docker inspect の OOMKilled やホストの dmesg を疑います。「アプリ例外」と「カーネルに殺された」は対処が別です。

14

BuildKit キャッシュ — CI の待ち時間を削る

▼

💡 毎回 pip/npm し直すのはパイプラインの敵

GitHub Actions 上の docker build は、キャッシュ戦略で何倍も差が出ます。 BuildKit の cache mount や registry cache を使うと、依存インストール層を再利用できます。CI/CD 章の「速いフィードバック」と直結します。

Dockerfile 
# syntax=docker/dockerfile:1
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
COPY . .
Bash 
# ローカルで BuildKit 有効化（多くの環境で既定）
export DOCKER_BUILDKIT=1
docker build -t myapi:dev .

# CI では docker/build-push-action の cache-from / cache-to を使う

💼 Tips：キャッシュは正しさの敵にもなる

依存を上げたのに古い wheel が残る、などの事故に注意。requirements.txt の変更でレイヤが無効化される設計を維持し、怪しいときだけ --no-cache します。
