🚀 CI/CD・デプロイ

「公開して運用する」までを、初級（GitHub Actions）・中級（デプロイと環境分離）・上級（高速化と観測）の 3 段階で学びます。

🟢 初級編 — 手動デプロイからパイプラインへ

「動いたら本番にコピー」を卒業し、GitHub Actions でテストとビルドを自動実行する土台を作ります。

01

なぜ CI/CD が必要か — 公開して運用する視点

▼

💡 デプロイはイベントではなくプロセスである

フルスタックのゴールは「動くコード」ではなく「ユーザーに届き続けるサービス」です。 CI（Continuous Integration）は変更のたびに検証を自動実行し、CD（Continuous Delivery/Deployment）はリリースを再現可能にします。手動 FTP や「本番で直接編集」は、再現性と監査可能性を失うため避けます。

CI と CD の違い

CI は「統合のたびにビルド・テスト」。CD は「リリース可能な状態を常に保つ／自動で本番へ出す」。まずは CI（PR でテスト）から始め、慣れたらデプロイ自動化へ進むのが挫折しにくい順序です。

Text 
手動の問題:
  - 手順が人の頭の中にしかない
  - 「先週の本番」と差が分からない
  - テストし忘れがそのまま公開される

パイプライン後:
  - main への変更 = 検証済み
  - 同じ手順で staging / production へ
  - 失敗したら自動で止まる

💼 Tips：最初の成功体験は「PR でテストが走る」

いきなり本番自動デプロイを目指すと秘密鍵や権限で詰みます。Week 1 の目標は GitHub Actions で npm test / pytest が緑になることです。

02

GitHub Actions 入門 — リポジトリに付くワーカー

▼

💡 YAML 一つでクラウド上の Linux が借りられる

GitHub Actions は、リポジトリのイベント（push / pull_request）をきっかけに仮想マシンでコマンドを走らせる仕組みです。 無料枠でも学習と個人プロジェクトには十分です。ランナーは実質 Linux/CLI 章の延長戦です。

最小のワークフロー

.github/workflows/*.yml に置くと GitHub が認識します。on がきっかけ、jobs が実行単位、steps が手順です。

YAML 
name: ci
on:
  pull_request:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
          cache: npm
      - run: npm ci
      - run: npm test

画面での確認

GitHub の Actions タブでログを見ます。失敗したら赤くなった step を開き、ローカルと同じコマンドで再現するのが定石です。

💼 Tips：actions/checkout を忘れない

ランナーは空の VM です。リポジトリを取ってこないとファイルがありません。最初の step はほぼ必ず checkout です。

03

ワークフロー構文 — job・step・needs

▼

💡 パイプラインをグラフとして設計する

複数 job は並列実行が既定で、needs で依存を書けます。 「まずテスト、成功したらビルド、成功したらデプロイ」という段階を YAML で表現します。

job を分ける利点

失敗箇所が明確になり、権限（後述の environment）をデプロイ job だけに絞れます。

YAML 
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

条件実行

if: でブランチやイベント種別を制限できます。本番デプロイは main の push だけ、などに使います。

YAML 
deploy:
  needs: build
  if: github.ref == 'refs/heads/main' && github.event_name == 'push'
  runs-on: ubuntu-latest
  steps:
    - run: echo "deploy here"

💼 Tips：ローカル再現を優先する

CI にしか通じない魔法を増やさないでください。npm test がローカルで通ることが、デバッグ速度を決めます。

04

PR チェックを必須にする — 品質ゲート

▼

💡 レビュー前に機械が一次審査する

ブランチ保護と Required Checks を使うと、テスト失敗の PR はマージできません。 人のレビューは設計と意図に集中させ、整形・型・テストは CI に任せます。

入れると効くチェック

テスト、lint、型チェック、npm audit（または同等）の順で費用対効果が良いです。本リポジトリの npm test（スモーク）も同じ思想です。

YAML 
- run: npm ci
- run: npm run lint
- run: npm test
# Python なら:
# - run: pip install -r requirements.txt
# - run: pytest
# - run: pip-audit

ブランチ保護（概念）

GitHub の Settings → Branches で main を保護し、指定 workflow の成功を必須にします。管理者でもバイパスできない設定にするとチームの規律が保たれます。

💼 Tips：flaky テストは敵

たまに落ちるテストは Required Checks を形骸化させます。揺れるなら直すか隔離し、「赤い = 本当に悪い」状態を守ってください。

05

CI のシークレット — 鍵をログに出さず渡す

▼

💡 GitHub Secrets と環境変数の接続

デプロイ用トークンや SSH 鍵は、ワークフローに直書きしません。 Repository secrets / Environment secrets に登録し、${{ secrets.NAME }} で渡します。セキュリティ章の「シークレット管理」の実装編です。

使い方

GitHub → Settings → Secrets and variables → Actions。ログに echo しない、PR から fork 経由で秘密が読まれない設計に注意します。

YAML 
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh
        env:
          FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

environment で保護を厚くする

GitHub Environments に production を作り、必須レビュアーや待機時間を設定できます。本番デプロイだけ追加承認、が少人数チームでも効きます。

💼 Tips：マスクされても安心しすぎない

Actions は secret をログでマスクしますが、加工して出力すると漏れることがあります。デバッグ時の env 全吐き出しは禁止です。

🔶 中級編 — 実際に公開し、止め方・戻し方まで持つ

静的サイトとコンテナアプリのデプロイ、staging/production、ヘルスチェックとロールバックまで。

06

静的サイトのデプロイ — Cloudflare Pages / GitHub Pages

▼

💡 フロントを「世界に出す」最短経路

ビルド成果物（HTML/CSS/JS）を CDN 付きホスティングへ載せるのが静的デプロイです。 React / Vite アプリの学習成果を公開する最短ルートで、カスタムドメインと HTTPS もプラットフォーム側が面倒を見ます。

一般的な流れ

push → CI で npm run build → dist/ を Pages へ。GitHub 連携なら YAML なしの GUI 設定でも始められますが、学習目的では Actions 経由が再現性を理解しやすいです。

YAML 
# 概念例: ビルド成果を artifact にする
- run: npm ci && npm run build
- uses: actions/upload-artifact@v4
  with:
    name: site
    path: dist/

# 実際の Pages への publish は公式 action か
# ホスティング側の Git 連携に任せる

環境変数と公開サイト

フロントに埋め込まれる変数（VITE_* 等）はユーザーから見える前提です。秘密は入れず、API キーはサーバー側へ。

💼 Tips：プレビューデプロイを使う

PR ごとにプレビュー URL が出るサービスを選ぶと、本番前に関係者と画面確認できます。CD の本質は「早く安全に見せる」ことです。

07

コンテナのデプロイ — イメージをレジストリへ、そして起動

▼

💡 Docker 章の成果を本番相当へ運ぶ

アプリをイメージに固め、レジストリ（GHCR 等）へ push し、サーバーや PaaS で pull して起動します。 Fly.io / Railway / 自前 VPS + Docker Compose はどれもこの型です。Linux の SSH と Docker 章の知識がここで合流します。

CI でのビルド＆プッシュ

タグは Git SHA か semver を使い、latest だけに依存しないとロールバックが楽です。

YAML 
- uses: docker/setup-buildx-action@v3
- uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}
- uses: docker/build-push-action@v6
  with:
    push: true
    tags: ghcr.io/${{ github.repository }}/api:${{ github.sha }}

ホスト側の更新（概念）

VPS なら SSH して docker compose pull && docker compose up -d。PaaS なら CLI（fly deploy）を CI から呼びます。

Bash 
# VPS 上での更新イメージ
docker compose pull
docker compose up -d
docker compose ps
curl -fsS http://127.0.0.1:8000/health

💼 Tips：イメージに秘密を焼かない

ENV JWT_SECRET=... を Dockerfile に書かず、実行時に環境変数で渡します。レイヤキャッシュに秘密が残る事故を防げます。

08

staging と production — 環境を分ける

▼

💡 本番で初めて動かすのを禁止する

設定・データ・秘密鍵を環境ごとに分離します。 staging でマイグレーションやスモークを試し、production へ進む。同じコンテナイメージを使い、違うのは環境変数だけ、が理想です。

分けるもの / 分けないもの

分ける: DB、API キー、ドメイン、ログレベル。分けない（揃える）: アプリのビルド成果、依存バージョン。

Text 
image: ghcr.io/org/api:abc123f  # 同じ成果物

staging:
  DATABASE_URL=postgres://staging...
  APP_URL=https://staging.example.com

production:
  DATABASE_URL=postgres://prod...
  APP_URL=https://example.com

GitHub Environment と紐付け

job の environment: staging / production で、参照できる secrets を切り替えます。

💼 Tips：本番データを staging に安易にコピーしない

個人情報が入る場合、マスキング無しのダンプは事故です。シードデータか匿名化データで十分揃うことが多いです。

09

ヘルスチェックとロールバック — 壊れたら戻す

▼

💡 デプロイ成功 ≠ サービス正常

プロセスが起動しても、依存 DB が落ちていればユーザーには障害です。 /health で生存確認し、失敗時は前イメージへ戻す手順を事前に書きます。「戻し方を知らない自動デプロイ」は危険です。

ヘルスエンドポイント

軽量な GET を用意し、CI やオーケストレータが叩きます。内部依存（DB）まで見るかはトレードオフですが、少なくともプロセス生存は確認します。

Python 
@app.get("/health")
def health():
    return {"status": "ok"}
Bash 
# デプロイ後スモーク
curl -fsS https://api.example.com/health

# イメージタグが分かっていればロールバックは再デプロイ
# docker compose up -d で前のタグを指定する、など

失敗時の自動停止

ヘルスチェック失敗で job を fail させ、後続の DNS 切替や告知を止める——これが CD の安全弁です。

💼 Tips：データベース移行は特に慎重に

後方互換のマイグレーション（expand → migrate → contract）を心がけると、ロールバック可能な時間帯が作れます。破壊的変更を一発でやらないのがコツです。

🔴 上級編 — 速く・観察できるパイプラインへ

キャッシュと行列で CI を速くし、ログ・アラートで運用し、一連の流れをケーススタディで固定します。

10

高速化 — cache と matrix

▼

💡 待ち時間はフィードバックを殺す

CI が 15 分かかると、人はローカルだけで押し切り始めます。 依存キャッシュと、バージョン行列（matrix）の使い分けで、必要な検証だけを並列に回します。

依存キャッシュ

actions/setup-node の cache: npm や actions/cache で ~/.npm 等を再利用します。ロックファイルをキーに含めます。

YAML 
- uses: actions/setup-node@v4
  with:
    node-version: "22"
    cache: npm
- run: npm ci

matrix で行列テスト

Node 20/22 や Python 3.11/3.12 を並べて互換性を確認できます。増やしすぎると課金と待ち時間が増えるので、方針を決めてから使います。

YAML 
strategy:
  matrix:
    node: [20, 22]
steps:
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node }}

💼 Tips：キャッシュは正しさより再現性を優先

キャッシュが原因で「自分の環境だけ通る」なら、キー設計を見直すか npm ci を徹底します。速さのために汚染された状態を残さないでください。

11

観測の入口 — ログ・メトリクス・アラート

▼

💡 デプロイ後に「見えない」をなくす

CD のゴールはリリースボタンではなく、異常に気づけることです。 まずはプラットフォームのログ（Actions, docker compose logs, ホスティングのダッシュボード）を見る習慣を付け、次に uptime 監視（無料枠の外形監視）を足します。

最小構成

① アプリログを集約 ② /health の外形監視 ③ エラー率や 5xx の通知——この 3 点で個人サービスの運用はかなり安定します。

Text 
デプロイ直後チェックリスト:
  [ ] Actions が緑
  [ ] /health が 200
  [ ] ログインなど代表導線を手動で1回
  [ ] エラーログに新規スタックが無いか確認
  [ ] 戻し手順（前タグ）をメモした

💼 Tips：アラートは「起きる頻度 × 対応可否」で設計

夜中に鳴るが無視するアラートは無いのと同じです。最初は「本番 5xx が N 分継続」だけでも十分価値があります。

12

総合ケース — PR から本番までの一本道

▼

💡 学習した部品を一つの物語にする

変更がユーザーに届くまでの経路を、自分の言葉で説明できることがゴールです。 以下の流れを、自分のポートフォリオプロジェクトに適用してください。

推奨フロー

機能ブランチ → PR → Actions（lint/test/audit）→ レビュー → main マージ → イメージ build/push → staging デプロイ＆スモーク → production デプロイ → 監視。

Text 
1. git switch -c feature/xxx
2. 実装 + テスト
3. PR 作成 → CI 必須チェック
4. main マージ
5. CI が image:git-sha を GHCR へ
6. staging に自動デプロイ → /health
7. production は environment 承認後
8. 問題時は前 sha のイメージで再デプロイ

セキュリティ章との接続

シークレットは Environments に、依存監査は CI に、認可テストは test job に載せます。公開はゴールではなく、運用のスタートです。

💼 Tips：ポートフォリオに Actions のバッジを貼る

README に CI バッジがあると、採用側に「テストと公開プロセスを持っている」ことが一目で伝わります。技術力アピールとして費用対効果が非常に高いです。
